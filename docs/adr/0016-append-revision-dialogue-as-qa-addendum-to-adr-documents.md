# 16. Append revision dialogue as Q&A addendum to ADR documents

Date: 2026-04-02

## Status

Proposed

## Context

ADR-0015 introduced the interactive revise task to the author-adr workflow. It
explicitly left the persistence format and location of the revision summary as
out of scope: "The persistence format and location of the summary is out of
scope for this ADR and will be addressed separately."

Today, the revise task produces a revision summary table showing which review
findings were addressed or rejected. However, this output lives only in the
agent conversation — it is ephemeral. Once the session ends, the reasoning
behind each revision decision is lost.

In a team workflow, this dialogue would naturally be captured in a pull request:
reviewers leave comments, the author responds, and the PR preserves the full
thread for posterity. When operating locally — drafting and revising ADRs
without a PR — there is no equivalent artifact to capture this exchange.

The current revision process takes the review output and produces a summary of
items that need to be addressed, combined with a first-pass analysis of the
conclusions stated in the ADR. This analysis represents valuable reasoning that
should be preserved alongside the decision record itself, so that future readers
(or the author returning after a break) can understand not just what was decided,
but what was questioned and how those questions were resolved.

### Decision Drivers

The primary motivation is operational: dogfooding the author-adr skill has shown
that the review and revision dialogue — currently captured only in the agent
session — contains reasoning that should survive the session. This ADR formalizes
that informal habit into a protocol by deciding where the dialogue is persisted.

The key drivers are:

- **Formalization over informality** — revision dialogue is currently an
  accidental artifact of agent sessions. It should be a deliberate part of the
  ADR lifecycle.
- **Colocation** — the dialogue should be discoverable alongside the decision it
  shaped, not in a separate file or directory.
- **Minimal friction** — the persistence mechanism should not introduce file
  management overhead or new naming conventions.

### Considered Alternatives

1. **Store revision dialogue in a separate companion file** (e.g.,
   `0016-title.revisions.md` alongside the ADR). This keeps the ADR clean but
   scatters related content across multiple files, making it easy to miss the
   revision context when reading the decision. It also introduces a naming
   convention and file management overhead.

2. **Store revision dialogue in a centralized log** (e.g.,
   `docs/adr/revisions/`). This collects all revision history in one place but
   further disconnects it from the ADR it belongs to. Readers must navigate
   between directories to understand the full story of a decision.

3. **Append a Q&A addendum directly to the ADR document.** The revision
   dialogue becomes part of the ADR itself, separated by a horizontal rule
   (`---`). The addendum is organized as a list of questions and responses,
   making it self-contained and immediately visible to anyone reading the ADR.

## Decision

Append the revision dialogue as a Q&A addendum to the end of the ADR document
itself. The addendum is separated from the main ADR body by a horizontal rule
(`---`) and structured as a list of questions and responses.

### Format

The revise task appends the addendum as part of its analysis of the review
output. Each review finding becomes a question, and the resolution (whether
addressed or rejected) becomes the response:

```markdown
---

## Revision Addendum

<!-- Generated by the revise task. Do not edit above the horizontal rule. -->

### Q: [Review finding, phrased as a question]

**[Addressed|Rejected]** — [Response explaining what was changed or why the
finding was declined]

### Q: [Next review finding]

**[Addressed|Rejected]** — [Response]
```

The `---` horizontal rule serves as a **semantic boundary**: content above is
the immutable decision record; content below is a mutable worksheet. The
`## Revision Addendum` heading is the **canonical detection marker** that the
revise task uses to locate an existing addendum when appending new entries.

### Workflow Integration

The revise task appends this addendum as part of Step 5 (Produce a Revision
Summary) in `references/revise.md`. The addendum is written to the ADR file
after all review comments have been processed and revisions applied.

This supports two usage patterns:

1. **Agent-assisted revision** — the user invokes the revise task, which walks
   through each finding interactively and appends the Q&A addendum
   automatically at the end of the session.

2. **Manual revision** — the user breaks from the agent after reviewing the
   findings, edits the ADR and addendum directly, then re-invokes the
   author-adr skill to continue revisions on the same ADR. The existing
   addendum provides continuity — the agent can read it to understand what has
   already been addressed.

### Accumulation Across Rounds

If multiple revision rounds occur, each round appends its entries to the same
addendum section. A second revision pass does not create a second `---`
separator — it adds new Q&A entries below the existing ones, optionally
prefixed with a round marker (e.g., `<!-- Round 2 -->`).

## Consequences

- **Revision context is preserved in-place** — future readers see both the
  decision and the dialogue that shaped it, without navigating to separate files.
  This mirrors how PR comments provide context alongside the code they discuss.

- **Enables asynchronous, multi-session revision** — because the addendum is
  written to the ADR file, a user can stop mid-revision, return later, and the
  agent (or the user themselves) can pick up where they left off by reading the
  existing Q&A entries.

- **ADR documents grow longer** — the addendum adds content to the ADR file.
  For ADRs with many revision rounds, this could make the file unwieldy. Based
  on dogfooding, most ADRs go through 1–2 revision rounds with 4–7 findings
  each, keeping the addendum manageable.

- **The `---` separator is a convention, not enforcement** — there is no
  tooling to prevent edits above or below the separator. Authors who manually
  edit the addendum may inadvertently break the structure. The `<!-- Generated
  by the revise task -->` comment serves as a soft indicator of the boundary.

- **Couples the revise task to ADR file format** — the revise task must know
  how to detect an existing addendum (by scanning for `## Revision Addendum`)
  and append to it rather than creating a duplicate section. This is a modest
  parsing requirement.

- **Does not replace PR-based review** — the addendum is a local collaboration
  artifact between the developer and the agent — a "review before the review."
  It captures the developer's pre-commit quality pass over agent-drafted
  content. When the ADR is later submitted via pull request, the PR becomes the
  canonical review artifact. The developer may choose to remove the addendum
  before merging if the PR discussion supersedes it.

- **Requires updating `references/revise.md`** — Step 5 (Produce a Revision
  Summary) must be updated to include addendum-appending behavior and to scan
  for an existing `## Revision Addendum` section before appending new entries.
