# 9. Implement-adr uses incremental progress tracking during plan execution

Date: 2026-04-02

## Status

Accepted

## Context

The `implement-adr` skill generates implementation plans with staged tasks, each
containing **Test & Acceptance Criteria** checkboxes (`- [ ]`). When an agent
executes a plan, there is currently no instruction in the skill specification
about how to report progress. The plan file is static — criteria checkboxes are
not updated during execution, so there is no visibility into which criteria have
been satisfied, which tasks are partially complete, or where execution was
interrupted.

The plan revision `0006-0008.1.plan.md` introduced an ad-hoc **Task Execution
Protocol** to address this gap. The protocol required agents to update checkboxes
incrementally as each criterion was satisfied, rather than batching updates at
the end. In a single trial with a 7-task, 3-stage plan, the protocol turned the
plan file into a live progress tracker, made partial completion visible on
interruption, and gave reviewers a clear picture of which criteria had been
verified.

However, the protocol lives only in that specific plan revision. It is not part
of the skill specification, so agents executing other plans do not follow it.
The protocol needs to be codified in the skill itself to apply universally.

## Decision

### 1. Embed the Task Execution Protocol in the plan template

The ad-hoc protocol from `0006-0008.1.plan.md` will be added to the plan
template (`assets/templates/plan-template.md`) so that every generated plan
includes the protocol as a self-contained section. This ensures the executing
agent sees the protocol in the plan file itself — the artifact it reads during
execution — rather than relying on the agent having loaded SKILL.md.

**Protocol (added to the plan template):**

When executing any task from this plan:

1. **Before starting a task** — read the task's Test & Acceptance Criteria
   section in full.
2. **As each criterion is satisfied** — immediately edit the plan file to mark
   the checkbox from `- [ ]` to `- [x]`.
3. **After completing all implementation work** for a task — review the criteria
   list one final time. Any criterion still unchecked is either incomplete work
   (go back and finish it) or an oversight (check it off with a brief inline
   note if it was already satisfied by prior work).
4. **Do not batch checkbox updates** to the end of a stage — update them
   incrementally so that progress is visible in the file at all times.

This protocol enables:
- The plan file serves as a live progress tracker, not just a static document.
- Partial completion is visible if a task is interrupted.
- Reviewers can see exactly which criteria have been verified.

### 2. Alternatives considered

Four approaches to progress tracking were evaluated:

| # | Approach | Mechanism | Pros | Cons |
|---|----------|-----------|------|------|
| A | **In-file checkbox updates** (chosen) | Edit `- [ ]` → `- [x]` in the plan file as criteria are met | Zero new files; reuses existing plan structure; human-readable in any markdown viewer; `git diff` shows exactly what changed | Single-writer assumption (concurrent agents editing the same plan could conflict); ties progress to the plan file's format |
| B | **Separate status file** | Write a `<plan>.status.json` alongside the plan, recording task/criterion completion state | Avoids modifying the plan file; structured data for tooling; no format coupling | Adds a new file convention; status can drift from plan content; not human-readable at a glance; requires tooling to correlate with the plan |
| C | **Checkpoint comments in the plan** | Append completion notes as markdown comments (`<!-- Task 1.1 done at ... -->`) at the end of each task section | Plan file unchanged visually; comments are invisible in rendered markdown | Invisible progress — defeats the goal of live visibility; comments can be stripped by editors; no structured data for tooling |
| D | **No codification** | Keep the protocol ad-hoc; include it per plan when desired | No template change; flexible per plan | Agents only follow the protocol when a specific plan happens to include it; inconsistent progress tracking across plans |

**Why Option A:** The primary goal is *visible, incremental progress* — a human
or agent glancing at the plan should immediately see what is done and what
remains. In-file checkboxes achieve this with zero additional tooling or
conventions. The single-writer limitation is acceptable because plan execution
is inherently sequential (one agent works through tasks in order), and the
existing plan revision workflow (ADR-0005) already assumes a single writer per
revision.

### 3. Scope and compatibility

The protocol requires tasks to have `- [ ]` checkboxes in their acceptance
criteria. All plans generated by the current `implement-adr` skill use this
format (established by the plan template in `assets/templates/plan-template.md`).
Plans generated by other tools or written manually may use different formats
(numbered lists, prose); the protocol does not apply to those and the agent
should not attempt to retrofit checkboxes into non-checkbox criteria.

### 4. Revisit trigger

Revisit this decision after **5 plan executions** using the protocol to validate:
- Whether incremental checkbox edits cause excessive file I/O or merge conflicts
  in practice.
- Whether the single-writer assumption holds across real usage patterns.
- Whether plans with many small tasks (20+ criteria) produce noisy diffs.

## Consequences

**Positive:**

- The Task Execution Protocol becomes a permanent part of every generated plan,
  enabling all plan executions to benefit from live progress tracking — not just
  plans that happen to include the protocol ad-hoc.
- Incremental checkbox updates make the plan file an intended reliable indicator
  of current state, useful for interrupted sessions, handoffs between agents,
  and human review.
- No new files, formats, or tooling are introduced — the protocol reuses the
  plan structure already produced by the skill.

**Negative / Risks:**

- The protocol assumes a single agent writes to the plan file at a time. If two
  sessions resume work on the same plan concurrently, checkbox edits could
  conflict. Mitigated by the sequential nature of plan execution and by the
  plan revision workflow (a second agent would create a new revision rather than
  editing the same file).
- Plans with many small tasks may produce frequent, small diffs that are noisy
  in version control. Mitigated by the fact that checkbox changes are
  single-character edits (`[ ]` → `[x]`) that are easy to scan and collapse.
- The evidence base is a single trial with a 7-task plan. The protocol's
  effectiveness at larger scales (20+ tasks, multi-session execution) is
  plausible but unvalidated. Mitigated by the revisit trigger above.

**Neutral:**

- The protocol is a behavioral instruction, not a structural change. It does not
  alter the plan format, task structure, or any other skill output.
- Plans generated before this ADR is implemented will benefit from the protocol
  if executed by an agent running the updated skill, provided they use the
  checkbox format for acceptance criteria.
- This ADR is independent of [ADR-0010](0010-implement-adr-optionally-commits-on-task-completion.md),
  which proposes git commits at task boundaries. The protocol applies regardless
  of whether auto-commit is enabled.
