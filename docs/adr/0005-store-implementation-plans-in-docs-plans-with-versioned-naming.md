# 5. Store implementation plans in docs/plans with versioned naming

Date: 2026-04-01

## Status

Accepted

## Context

The `implement-adr` skill generates `plan.md` files as its primary output, but
the current workflow writes plans to the repository root (or a user-specified
path) with no standard location or naming convention. This creates several
problems:

- **No canonical location** — plans end up in arbitrary locations, making them
  hard to discover. A developer joining the project has no reliable place to
  look for implementation plans.
- **No traceability from filename** — a plan named `plan.md` gives no
  indication of which ADRs it implements. When multiple plans exist, it is
  impossible to tell them apart by filename alone.
- **No revision history** — implementation plans evolve as users request
  changes, but the current approach overwrites the plan in-place. Previous
  iterations are lost, making it impossible to understand how a plan evolved
  or to revert to an earlier version.
- **Gitignore conflicts** — `plan.md` at the repository root is commonly
  gitignored (as in this very repository), since session-specific plan files
  should not be committed. Implementation plans, however, are project artifacts
  that _should_ be committed alongside ADRs.

The ADR decision log lives in `docs/adr/`. A parallel directory for plans would
maintain the `docs/` convention and make the relationship between decisions and
plans discoverable.

## Decision

We will establish `docs/plans/` as the canonical directory for implementation
plans generated by the `implement-adr` skill, and adopt a versioned naming
convention that encodes ADR scope and revision number.

### 1. Plan directory

All implementation plans are stored in `docs/plans/`. The skill will create
this directory if it does not exist.

### 2. Naming convention

Plan files follow this pattern:

```
<adr-range>.<revision>.plan.md
```

Where:
- **`<adr-range>`** is the zero-padded ADR number(s) the plan covers:
  - Single ADR: `0003`
  - Range of ADRs: `0003-0004`
  - Non-contiguous ADRs: `0003-0004-0007`
- **`<revision>`** is a zero-based integer incremented on each plan revision:
  - First plan: `0`
  - First revision: `1`
  - Second revision: `2`

Examples:
```
docs/plans/0003-0004.0.plan.md    # initial plan for ADRs 3–4
docs/plans/0003-0004.1.plan.md    # first revision after user feedback
docs/plans/0007.0.plan.md         # initial plan for ADR 7 alone
docs/plans/0003-0004-0007.0.plan.md  # plan spanning non-contiguous ADRs
```

### 3. Revision workflow

When the user requests changes to a plan:
1. The revision number is incremented.
2. A new plan file is created at the new revision (the previous revision is
   preserved unchanged).
3. The new revision documents the requested changes in its header, linking
   back to the previous revision.

```markdown
**Revision:** 1 (previous: [0003-0004.0.plan.md](docs/plans/0003-0004.0.plan.md))
**Changes:** Reorganized Stage 2 per user feedback; added caching task.
```

### 4. SKILL.md and template updates

The `implement-adr` skill's Step 5 (Review and Iterate) and the plan template
will be updated to use `docs/plans/` and the versioned naming convention
instead of writing to `plan.md` in the repository root.

## Consequences

**Positive:**

- Plans are discoverable in a canonical location alongside ADRs (`docs/plans/`
  next to `docs/adr/`).
- The filename encodes which ADRs a plan implements, enabling traceability
  without opening the file.
- Revision history is preserved as separate files, enabling comparison between
  iterations and audit of how plans evolved.
- Avoids `.gitignore` conflicts with session-specific `plan.md` files at the
  repository root.

**Negative / Risks:**

- Multiple revision files may accumulate for frequently revised plans. Mitigated
  by the fact that revisions are typically few (1–3) and the files serve as
  valuable history.
- The naming convention is specific to this skill and not a widely recognized
  standard. Mitigated by the convention being self-explanatory and documented.

**Neutral:**

- Existing plans written to the repository root by earlier versions of the
  skill are unaffected. Migration is optional.
