## Stage 1: Library Migration

### Task 1.1: Create format module in adr-db-lib    [medium]

**Description:** Create `src/crates/adr-db-lib/src/format/` module directory. Move `schema.rs` and `template.rs` from `adr-format/src/` into it. Create `mod.rs` that re-exports the public API. Update `adr-db-lib/src/lib.rs` to declare `pub mod format` and re-export via `pub use format::*`.

Note: `adr_db_lib::format::schema` (TOML document types like `Adr`, `Meta`, `Decision`) is distinct from `adr_db_lib::schema` (Diesel-generated table definitions for `task_summaries`). These must not be merged.

**Implementation Notes:**

New file structure:
```
src/crates/adr-db-lib/src/
├── format/
│   ├── mod.rs          ← pub mod schema; pub mod template; pub use schema::*; pub use template::*;
│   ├── schema.rs       ← moved from adr-format/src/schema.rs
│   └── template.rs     ← moved from adr-format/src/template.rs
├── db.rs               (unchanged)
├── models.rs           (unchanged)
├── remote/             (unchanged)
├── schema.rs           (Diesel — unchanged)
└── lib.rs              ← add: pub mod format; pub use format::*;
```

**Test & Acceptance Criteria:**
- [x] `src/crates/adr-db-lib/src/format/mod.rs` exists and declares `pub mod schema` and `pub mod template`
- [x] `src/crates/adr-db-lib/src/format/schema.rs` contains all types from `adr-format/src/schema.rs` (Adr, Meta, Decision, Consequence, etc.)
- [x] `src/crates/adr-db-lib/src/format/template.rs` contains `generate_template` from `adr-format/src/template.rs`
- [x] `adr-db-lib/src/lib.rs` declares `pub mod format` and re-exports with `pub use format::*`
- [x] `cargo build -p adr-db-lib` compiles without errors

**Dependencies:** None

## Stage 2: CLI Migration

### Task 2.1: Add format subcommand group to adr-db    [medium]

**Description:** Move the CLI command definitions from `adr-format/src/main.rs` into `adr-db/src/main.rs` as a `format` subcommand group. The existing adr-db commands (`init`, `ingest`, `view`, `fetch`) remain unchanged. The new `format` group adds: `new`, `init`, `list`, `rename`, `status`, `lifecycle`, `export`.

Since both adr-db and adr-format have an `init` subcommand, the format version becomes `adr-db format init` (no collision with top-level `adr-db init`).

**Implementation Notes:**

Add to adr-db's `Commands` enum:
```rust
/// ADR format operations (TOML document management)
Format {
    #[command(subcommand)]
    command: FormatCommands,
},
```

Define `FormatCommands` enum with all 7 subcommands from adr-format's `Commands`. Move the handler logic from `adr-format/src/main.rs` `match` arms into the `Format { command }` arm.

Update `adr-db/Cargo.toml`:
- The `adr-db-lib` dependency already provides the format types.
- Add `serde_json = "1"` if not already present (already present).

**Test & Acceptance Criteria:**
- [x] `adr-db format new <remote> <id> <title> <dir>` works (same behavior as `adr-format new`)
- [x] `adr-db format init <dir>` works
- [x] `adr-db format list <dir>` works
- [x] `adr-db format rename <old> <new>` works
- [x] `adr-db format status <file> [--set <status>]` works
- [x] `adr-db format lifecycle <file>` works
- [x] `adr-db format export <file>` works
- [x] Existing commands (`adr-db init`, `adr-db ingest`, `adr-db view`, `adr-db fetch`) still work
- [x] `cargo build -p adr-db` compiles without errors

**Dependencies:** Stage 1

**ADR Reference:** ADR-0061, Decision §Migration step 3
