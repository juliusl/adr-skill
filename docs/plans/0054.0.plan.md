## Stage 2: author-adr — Add Policies Section and Elevate Inline Policies

### Task 2.1: Identify and extract cross-cutting policies from author-adr    [medium]

**Description:** Add a `## Policies` section to `src/skills/author-adr/SKILL.md` between the skill header and `## Procedure`. Elevate the following inline enforcement statements that are cross-cutting:

1. **Mandatory dispatch compliance** (line 118): "When `[author.dispatch]` keys are configured, always use the configured agent — do not substitute `general-purpose` or skip dispatch." This applies across A-3 (Review) and A-4 (Revise) — cross-cutting.

2. **Cross-ADR modification guardrail** (line 200): "When modifying ADRs, never modify other existing ADRs without explicit instruction." This applies across A-2 (Create), A-4 (Revise), and A-6 (Manage) — cross-cutting.

Each elevated policy keeps a brief inline reference at its original location pointing to the policy ID.

**Implementation Notes:**
Insert after the skill description (line 10) and before `## Procedure` (line 11):
```markdown
## Policies

**Ignoring any of the below policies is a runtime violation ESPECIALLY when the user is away and operating autonomously**

| ID | Policy |
|----|--------|
| P-1 | Always use configured dispatch agents — do not substitute or skip |
| P-2 | Never modify other existing ADRs without explicit instruction |

### P-1: Mandatory Dispatch Compliance

When `[author.dispatch]` keys are configured, always use the configured agent — do not substitute `general-purpose` or skip dispatch. The user configured these agents for a reason. This applies in all modes, including autonomous workflows triggered by other skills (e.g., implement-adr invoking author-adr for ADR creation).

### P-2: Cross-ADR Modification Guardrail

When modifying ADRs, never modify other existing ADRs without explicit instruction. Cross-references and status updates to other ADRs (e.g., marking one as superseded) are the user's responsibility — suggest the change but do not apply it unilaterally.
```

Replace the original inline statements with back-references:
- Line 118: Replace the full paragraph with "See P-1 (Mandatory Dispatch Compliance)."
- Line 200: Replace the full paragraph with "**Guardrail (P-2):** See P-2 (Cross-ADR Modification Guardrail)."

## Stage 3: implement-adr — Evaluate Inline Enforcement for Elevation

### Task 3.1: Evaluate and document inline enforcement in implement-adr    [small]

**Description:** Review implement-adr's inline enforcement statements (I-4b sequencing constraint, I-7b QA mandatory, auto-commit enforcement logging) against ADR-0054 Requirement 5 (elevation criteria). Determine which, if any, should be elevated to the Policies section.

Analysis: The existing P-1 ("Never skip QA Validation") already covers the I-7b enforcement. The I-4b sequencing constraint ("Parallel dispatch of I-4 and I-4b is a workflow violation") and the auto-commit enforcement logging ("This logging is mandatory") are step-specific — they only apply within their respective steps and don't constrain behavior across multiple steps. These should remain inline.

Document this analysis as a comment in the plan for traceability.

**Test & Acceptance Criteria:**
- [ ] Each inline enforcement statement in implement-adr is enumerated with disposition (step-specific or cross-cutting)
- [ ] Step-specific statements remain inline with no changes
- [ ] No new policies need to be added to implement-adr's Policies section (confirm existing P-1–P-3 are sufficient)

**Dependencies:** None

**ADR Reference:** ADR-0054, Decision §Requirements 5, §File-Specific Actions (implement-adr)
