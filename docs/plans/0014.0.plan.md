## Stage 2: Author Skill Config Integration

Update `author-adr` to read `[author].template` from the config file,
replacing the current format-detection-from-project-files behavior.

### Task 2.1: Rewrite Format Detection to use config file    [medium]

**Description:** Rewrite the `### Format Detection` section in
`src/skills/author-adr/SKILL.md` to read the template format from the config
file instead of detecting it from the first ADR in `docs/adr/`.

Per ADR-0014 key design choice #3: "Config is authoritative —
`[author].template` is the single source of truth for template format. The
skill does not re-detect format from project files."

The new logic should be:

1. Resolve the config path (per the Configuration section added in Task 1.2).
2. Read `[author].template` from `preferences.toml`.
3. **If set to `"nygard"` or `"madr"`:** use it directly. Cache the format for
   the session.
4. **If not set (key absent):** default to `"nygard"` (ADR-0014 §key design
   choice #4).
5. When bootstrapping a new project (`docs/adr/` does not exist):
   - Prompt the user for format choice (Nygard or MADR) — same as current.
   - Bootstrap via `make init`.
   - **Offer to save** the chosen format to `[author].template` in
     `preferences.toml` so future projects use the same default.

The "read first ADR to infer format" logic is removed entirely.

**Implementation Notes:**

Replace the current Format Detection section (lines 36–59 of SKILL.md) with:

```markdown
### Format Detection

Before any ADR operation, determine which ADR format to use:

1. **Read the config file** — resolve the config path (see [Configuration](#configuration))
   and read `[author].template` from `preferences.toml`.
   - If set to `"nygard"` or `"madr"`, use it directly.
   - If absent, default to `"nygard"`.
2. **If `docs/adr/` does not exist** — prompt the user with a discrete choice:

   > Which ADR format would you like to use?
   > 1. **Nygard** — lightweight, widely adopted (Status / Context / Decision / Consequences)
   > 2. **MADR** — structured tradeoff analysis (adds Options, Pros/Cons, Decision Outcome)

3. **Bootstrap the decision log** — initialize `docs/adr` and create the first
   ADR recording the format choice. Use the Makefile target:

   ```bash
   make -f <skill-root>/Makefile init DIR=docs/adr
   ```

   This first ADR always uses Nygard format (the bootstrap default). If the user
   chose MADR, set `ADR_AGENT_SKILL_FORMAT=madr` for all subsequent ADRs.

4. **Offer to persist** — if the user chose a format during bootstrap, offer to
   save it to `[author].template` in `preferences.toml`:

   > Save this format as your default for all projects?

   If yes, write `[author]\ntemplate = "<format>"` to `preferences.toml`
   (creating the file and directory if needed with `mkdir -p`).

5. **Cache the format** — for the rest of the session, pass
   `ADR_AGENT_SKILL_FORMAT=nygard` or `ADR_AGENT_SKILL_FORMAT=madr` to all
   Makefile targets.
```

**Test & Acceptance Criteria:**
- [x] Format Detection no longer mentions reading the first ADR to detect format
- [x] Format Detection reads `[author].template` from config file as step 1
- [x] Default when key is absent is `"nygard"`
- [x] Bootstrap prompt for new projects is preserved
- [x] Offer to persist format choice to config file is present
- [x] The `ADR_AGENT_SKILL_FORMAT` caching for Makefile targets is preserved
- [x] The workflow diagram (lines 61–73) is consistent with the new detection logic
- [x] SKILL.md remains under 500 lines

## Behavioral Policies

The skill supports persistent behavioral preferences stored in the user-scoped
config file at `~/.config/adr-skills/preferences.toml` (see
[Configuration](#configuration)).

### Supported Policies

| Policy | Config Key | Default | Effect |
|--------|-----------|---------|--------|
| Participation mode | `[implement].participation` | `"guided"` | Sets the default participation level (full-control, guided, autonomous, weighted) |

   `[implement]` table, use it.
