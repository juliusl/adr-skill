## Stage 1: Schema Foundation

### Security Checks

- [x] **S1.1** — No user-supplied strings interpolated into SQL, shell commands, or file paths
  - `work_item` field uses `{remote}#{id}` format split on `#`. If `id` contains shell metacharacters, downstream consumers could be affected. Template generation interpolates `remote`, `id`, and `title` into struct fields — serde handles TOML escaping, but the values propagate to filenames in Stage 2.
- [x] **S1.2** — No secrets, credentials, or API keys in committed files
  - Schema structs and test fixtures only. Low risk.
- [ ] **S1.3** — No deserialization of untrusted input without validation
  - TOML deserialization is the core of this stage. Validation exists for: status enum, empty assessment, mutual exclusivity of semantic change flags. Missing validation: `work_item` format only checks for presence of `#` (line 239 schema.rs) but does not reject multiple `#`, empty remote, or empty id segments. `links` array element format unvalidated. No string length bounds on `body` fields.
- [x] **S1.4** — Dependencies pinned to specific versions (no wildcards)
  - `Cargo.toml` uses caret bounds: `serde = { version = "1" }`, `toml = "0.8"`, `clap = { version = "4" }`, `serde_json = "1"`. These are wide caret ranges, not exact pins. However, `Cargo.lock` IS committed (verified), which ensures reproducible builds. No wildcards (`*`) used. Acceptable for Rust ecosystem conventions.
- [x] **S1.5** — File permissions on created artifacts are not overly permissive
  - Library code only — no files created. N/A for this stage.
- [ ] **S1.6** — Any new external input surface has validation at the boundary
  - The `Adr` struct is the input boundary for all TOML files. Status validation and checkpoint assessment checks are present. `work_item` parsing validates only that `#` is present (schema.rs:239) but does not enforce `{remote}#{id}` fully — `#42`, `gh#`, and `a#b#c` all pass validation.

### UX Checks

- [x] **U1.1** — Every error path produces a human-readable message on stderr
  - Library code. `ValidationError::Display` impl (schema.rs:189-207) produces clear messages including file path via `DeserializationError`. Error messages are specific: invalid status names allowed values, empty assessment names checkpoint, work_item shows expected format.
- [x] **U1.2** — Every user-facing command exits with code 0 on success, non-zero on failure
  - Library code — no commands yet. N/A for this stage.
- [x] **U1.3** — Invalid input is rejected with a helpful message, not a crash
  - Serde deserialization errors are caught (not panics). Tests verify: invalid status (schema.rs:388), empty assessment (schema.rs:398), mutual exclusivity (schema.rs:409), invalid work_item (schema.rs:464), malformed TOML (schema.rs:476). All return `Result`, no `unwrap` or `panic`.
- [x] **U1.4** — Resources (file handles, database connections) are cleaned up on error paths
  - Pure data structures — no resources. N/A.
- [x] **U1.5** — If the stage writes data, there is a way to read it back or verify it
  - Round-trip test (schema.rs:380-385) covers serialize → deserialize → assert equality.
- [x] **U1.6** — If the stage creates state, there is a way to inspect the new state
  - No state created. N/A.
- [x] **U1.7** — A user who did not write the code can verify the stage's output
  - The round-trip test is the verification mechanism. `cargo test -p adr-format` is the interface. All 16 unit tests pass.

### Test-Gap Findings

- **TG1.1** — No test for `work_item` field format validation. The `{remote}#{id}` format (ADR-0051 design rule 9) uses a `#` delimiter, but the plan has no test for: multiple `#` characters (`gh#42#extra`), empty remote (`#42`), empty id (`gh#`), or missing delimiter (`gh42`). Downstream code (filenames, export Work-Item line) depends on this split being correct.

- **TG1.2** — No test for malformed TOML syntax. The acceptance criteria test semantic validation (invalid status, empty assessment) but not syntactic errors (unclosed strings, invalid table headers, broken `"""` delimiters). These are handled by the `toml` crate, but the error messages produced are raw parser output. Since ADR-0052 identified TOML generation by LLMs as a known failure mode, testing the error path for syntax errors is important.

- **TG1.3** — No test for TOML-special characters in string fields. Template generation (Task 1.2) interpolates `title` into a struct that gets serialized. If a title contains `"""` (triple-quote), `\` (escape), or `"` (quote), the serialized TOML must handle escaping correctly. Serde handles this, but a test should verify it because agents will produce these titles.

- **TG1.4** — No specification for unknown field handling. The plan does not state whether the schema structs use `#[serde(deny_unknown_fields)]`. If unknown fields are silently dropped during deserialization, a typo like `staus = "Proposed"` (instead of `status`) is silently ignored. If unknown fields are rejected, adding new optional fields to the schema becomes a breaking change. This is a design decision that affects schema evolution.

- **TG1.5** — No test for TOML field ordering in serialized output. `toml::to_string_pretty()` serializes in struct field order. If the struct field order doesn't match the ADR-0051 example, the generated TOML will have a different section layout than what the ADR specifies. This affects readability for anyone comparing generated files against the ADR.

## Stage 2: CLI Subcommands

### Security Checks

- [ ] **S2.1** — No user-supplied strings interpolated into SQL, shell commands, or file paths
  - **File path construction:** `cmd_new` constructs `{dir}/{remote}-{id}-{slug}.toml` (main.rs:159). `dir` has NO path traversal check — `../../../etc` is accepted. `validate_id` (main.rs:90-98) blocks non-alphanumeric chars (blocks `.`, `/`, `\`), which mitigates `id` injection. `validate_remote` checks an allowlist (main.rs:78-88). `slugify` (main.rs:100-112) replaces non-alphanumeric with `-`, stripping `/` from titles. **FAIL for `dir` — no `..` rejection.**
  - **Rename:** `slugify` strips `/` and null bytes from `new_title` by replacing with `-`. `validate_id` blocks `..` in id. Safe.
  - **Shell wrapper:** Uses `$(cd "$(dirname "$0")" && pwd)` (wi-full-agent-adr-format.sh:8) — resolves relative path safely. Uses `exec adr-format "$@"` — proper quoting.
- [x] **S2.2** — No secrets, credentials, or API keys in committed files
  - CLI code and test fixtures. Verified — no secrets found.
- [x] **S2.3** — No deserialization of untrusted input without validation
  - `cmd_list` (main.rs:252) catches per-file parse errors with `match parse_adr(...)` and prints warnings to stderr, continuing to list valid files. `cmd_lifecycle` (main.rs:352) catches corrupt JSONL per line with `if let Ok(val)`, silently skipping bad lines. Both handle corrupt input gracefully.
- [x] **S2.4** — Dependencies pinned to specific versions (no wildcards)
  - Same Cargo.toml as Stage 1. `serde_json = "1"` added. No wildcards. `Cargo.lock` committed ensures reproducible builds.
- [x] **S2.5** — File permissions on created artifacts are not overly permissive
  - `fs::write()` and `fs::create_dir_all()` use default umask. No explicit `0666`/`0777`. The shell wrapper is `-rwxr-xr-x` (755) — correct.
- [ ] **S2.6** — Any new external input surface has validation at the boundary
  - `remote` validated against allowlist (main.rs:78-88) — good. `id` validated as alphanumeric (main.rs:90-98) — good. **`new_status` in `cmd_status` is NOT validated** (main.rs:319-320): any string is accepted and written to the TOML file, including invalid statuses like `"Foo"`. `adr.validate()` is never called after setting the status. `dir` has no validation against path traversal.

### UX Checks

- [x] **U2.1** — Every error path produces a human-readable message on stderr
  - All commands wrap errors in `Err(format!(...))` or `Err(e.to_string())`. main.rs:493-495 prints `"ERROR: {e}"` to stderr and exits 1. `parse_adr` wraps toml errors with file path (schema.rs:252-257). `cmd_list` prints `"WARNING: skipping {name}: {e}"` for corrupt files (main.rs:258).
- [x] **U2.2** — Every user-facing command exits with code 0 on success, non-zero on failure
  - main.rs:493-496: `if let Err(e) = result { eprintln!(...); std::process::exit(1); }`. All commands return `Result<(), String>`. `cmd_list` with no ADRs found returns `Ok(())` (empty output, exit 0) — reasonable behavior.
- [ ] **U2.3** — Invalid input is rejected with a helpful message, not a crash
  - Remote validation tested (main.rs:78-88). ID validation tested (main.rs:90-98). **Missing: `new_status` in `cmd_status` accepts ANY string** (main.rs:320) — `adr-format status gh 42 "Foo"` silently writes `status = "Foo"` to the TOML file. No validation against the status enum at write time.
- [ ] **U2.4** — Resources (file handles, database connections) are cleaned up on error paths
  - File handles via `fs::write` are scoped. **Rename is NOT atomic** (main.rs:293-294): writes new content to old path first, then renames. If `fs::rename` fails after the write, the file content is updated but the filename is wrong. No temp-file-then-rename pattern used anywhere. `cmd_status` (main.rs:322) uses `fs::write` directly — a crash mid-write truncates the file.
- [x] **U2.5** — If the stage writes data, there is a way to read it back or verify it
  - `new` → verify with `list`. `status update` → verify with `status query`. `rename` → verify with `list`. All implemented and functional.
- [x] **U2.6** — If the stage creates state, there is a way to inspect the new state
  - The `.toml` files are inspectable with any text editor or `adr-format list`. The `.adr/var/work-items.jsonl` is human-readable.
- [x] **U2.7** — A user who did not write the code can verify the stage's output
  - `adr-format list` output uses `printf "%s\t[%s]\n"` format (main.rs:265) which matches the existing shell script `cmd_list` format in wi-nygard-agent-format.sh. Consistent UX across formats.

### Test-Gap Findings

- **TG2.1** — No test for `list` encountering a corrupt `.toml` file. The acceptance criteria only test the happy path (valid files, formatted output). If one file in the ADR directory is malformed TOML, does `list` skip it with a warning, crash, or silently omit it? A single corrupt file should not block listing the rest.

- **TG2.2** — No test for file write atomicity. The `rename` and `status` subcommands deserialize, modify, and reserialize the entire file. The plan says "this is safer than awk-based in-place editing" but does not specify atomic writes. A crash between truncating the old file and writing the new content loses data. The `rename` subcommand is especially risky — it must write the new file AND remove the old filename.

- **TG2.3** — No test for `rename` with filesystem-invalid characters in the new title. The title is slugified into a filename. Characters like `/`, `\0`, and extremely long titles (exceeding filesystem path limits) need handling. The acceptance criteria test a clean rename but not edge cases.

- **TG2.4** — No test for lifecycle with missing `.adr/var/` directory or corrupt JSONL. The `lifecycle` subcommand reads cached work item data. If the cache directory doesn't exist, or the JSONL is malformed, the error path needs a clear message ("run `make sync` first" or equivalent).

- **TG2.5** — No test for `init` when the ADR directory already contains files. Should `init` refuse to overwrite an existing `0001-record-architecture-decisions.toml`? The acceptance criteria test creation but not the idempotency/overwrite case.

- **TG2.6** — Shell wrapper `dirname`-based path resolution (`$(dirname "$0")/../../crates/target/release/adr-format`) is fragile when the wrapper is invoked through a symlink. `dirname` resolves the symlink's location, not the target's location. The `readlink -f` or `realpath` pattern would be more robust.

### Test-Gap Findings

- **TG3.1** — No explicit idempotency test. ADR-0053 export rule 6 requires idempotent output. The acceptance criteria assert this property but no test exercises it directly (run export twice, diff the outputs, assert identical). A snapshot test alone does not prove idempotency — it proves the output matches a golden file once.

- **TG3.2** — No test for export of a minimal ADR. The snapshot test uses a "known TOML input" but the criteria don't specify testing with only required fields (no optional sections, no negative consequences, no links, empty work_item). ADR-0053 export rule 8 says empty sections are omitted — this rule needs explicit coverage with a minimal input.

- **TG3.3** — No test for `Last Updated` in Markdown output. The ADR-0053 example output includes `Last Updated: 2026-04-06` but the acceptance criteria list only "date, status, links, work-item" — `last_updated` is not mentioned. It is present in the TOML meta struct. The omission could mean it's exported and not verified, or not exported and nobody noticed.
