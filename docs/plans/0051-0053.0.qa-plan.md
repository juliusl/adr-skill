## Stage 1: Schema Foundation

### Security Checks

- [ ] **S1.1** â€” No user-supplied strings interpolated into SQL, shell commands, or file paths
  - `work_item` field uses `{remote}#{id}` format split on `#`. If `id` contains shell metacharacters, downstream consumers could be affected. Template generation interpolates `remote`, `id`, and `title` into struct fields â€” serde handles TOML escaping, but the values propagate to filenames in Stage 2.
- [ ] **S1.2** â€” No secrets, credentials, or API keys in committed files
  - Schema structs and test fixtures only. Low risk.
- [ ] **S1.3** â€” No deserialization of untrusted input without validation
  - TOML deserialization is the core of this stage. Validation exists for: status enum, empty assessment, mutual exclusivity of semantic change flags. Missing validation: `work_item` format (multiple `#` characters, empty segments), `links` array element format, string length bounds on `body` fields.
- [ ] **S1.4** â€” Dependencies pinned to specific versions (no wildcards)
  - Verify `crates/adr-format/Cargo.toml` pins `serde`, `toml`, `clap`, `chrono` to exact or caret-bounded versions. `Cargo.lock` must be committed for a binary crate.
- [ ] **S1.5** â€” File permissions on created artifacts are not overly permissive
  - Library code only â€” no files created. N/A for this stage.
- [ ] **S1.6** â€” Any new external input surface has validation at the boundary
  - The `Adr` struct is the input boundary for all TOML files. Status validation and checkpoint assessment checks are present. `work_item` parsing is an input surface without boundary validation in this stage.

### UX Checks

- [ ] **U1.1** â€” Every error path produces a human-readable message on stderr
  - Library code. ADR-0052 acknowledges serde default errors are not user-friendly and calls for custom error formatting. Task 1.1 acceptance criteria test rejection of invalid status and empty assessment but do not specify error message format. The error types defined here flow to Stage 2 CLI output.
- [ ] **U1.2** â€” Every user-facing command exits with code 0 on success, non-zero on failure
  - Library code â€” no commands yet. N/A for this stage.
- [ ] **U1.3** â€” Invalid input is rejected with a helpful message, not a crash
  - Serde deserialization errors are caught (not panics). The plan tests invalid status, empty assessment, and mutual exclusivity. Missing: malformed TOML syntax errors, which serde+toml crate handle but produce raw error strings.
- [ ] **U1.4** â€” Resources (file handles, database connections) are cleaned up on error paths
  - Pure data structures â€” no resources. N/A.
- [ ] **U1.5** â€” If the stage writes data, there is a way to read it back or verify it
  - Round-trip test (serialize â†’ deserialize â†’ assert equality) covers this.
- [ ] **U1.6** â€” If the stage creates state, there is a way to inspect the new state
  - No state created. N/A.
- [ ] **U1.7** â€” A user who did not write the code can verify the stage's output
  - The round-trip test is the verification mechanism. `cargo test` is the interface.

### Test-Gap Findings

- **TG1.1** â€” No test for `work_item` field format validation. The `{remote}#{id}` format (ADR-0051 design rule 9) uses a `#` delimiter, but the plan has no test for: multiple `#` characters (`gh#42#extra`), empty remote (`#42`), empty id (`gh#`), or missing delimiter (`gh42`). Downstream code (filenames, export Work-Item line) depends on this split being correct.

- **TG1.2** â€” No test for malformed TOML syntax. The acceptance criteria test semantic validation (invalid status, empty assessment) but not syntactic errors (unclosed strings, invalid table headers, broken `"""` delimiters). These are handled by the `toml` crate, but the error messages produced are raw parser output. Since ADR-0052 identified TOML generation by LLMs as a known failure mode, testing the error path for syntax errors is important.

- **TG1.3** â€” No test for TOML-special characters in string fields. Template generation (Task 1.2) interpolates `title` into a struct that gets serialized. If a title contains `"""` (triple-quote), `\` (escape), or `"` (quote), the serialized TOML must handle escaping correctly. Serde handles this, but a test should verify it because agents will produce these titles.

- **TG1.4** â€” No specification for unknown field handling. The plan does not state whether the schema structs use `#[serde(deny_unknown_fields)]`. If unknown fields are silently dropped during deserialization, a typo like `staus = "Proposed"` (instead of `status`) is silently ignored. If unknown fields are rejected, adding new optional fields to the schema becomes a breaking change. This is a design decision that affects schema evolution.

- **TG1.5** â€” No test for TOML field ordering in serialized output. `toml::to_string_pretty()` serializes in struct field order. If the struct field order doesn't match the ADR-0051 example, the generated TOML will have a different section layout than what the ADR specifies. This affects readability for anyone comparing generated files against the ADR.

## Stage 2: CLI Subcommands

### Security Checks

- [ ] **S2.1** â€” No user-supplied strings interpolated into SQL, shell commands, or file paths
  - **File path construction:** `new` takes a `dir` argument and constructs a file path as `{dir}/{remote}-{id}-{slug}.toml`. The `dir`, `remote`, `id`, and `title` (slugified) are all user-supplied. Path traversal (`../../../etc/`) in `dir` could write outside the ADR directory. Shell metacharacters in `remote` or `id` flow into filenames.
  - **Rename:** `new_title` is slugified into a new filename. Filesystem-invalid characters (`/`, `\0`, null bytes) must be stripped.
  - **Shell wrapper:** `exec adr-format "$@"` uses proper quoting. The `dirname`-based fallback path is constructed from the script location, not user input.
- [ ] **S2.2** â€” No secrets, credentials, or API keys in committed files
  - CLI code and test fixtures. Low risk.
- [ ] **S2.3** â€” No deserialization of untrusted input without validation
  - `list` deserializes every `.toml` file in the ADR directory â€” any of these could be malformed or adversarial. `lifecycle` reads `.adr/var/work-items.jsonl` (JSONL from external source). Both are untrusted input.
- [ ] **S2.4** â€” Dependencies pinned to specific versions (no wildcards)
  - Same Cargo.toml as Stage 1. Adding `serde_json` for lifecycle JSONL parsing.
- [ ] **S2.5** â€” File permissions on created artifacts are not overly permissive
  - `new` creates `.toml` files. `init` creates a directory and seed file. `lifecycle --auto` writes to `.adr/var/lifecycle.jsonl`. Default umask should apply, but verify files are not created with 0666/0777.
- [ ] **S2.6** â€” Any new external input surface has validation at the boundary
  - CLI arguments: `remote` validated against allowlist (gh|ado|gitea|local) â€” good. `id` validated as alphanumeric â€” good. `new_status` should be validated against the status enum. `dir` has no validation against path traversal.

### UX Checks

- [ ] **U2.1** â€” Every error path produces a human-readable message on stderr
  - The plan calls for "deserialization errors include the source file path and a human-readable description." Verify this covers: file not found, permission denied, TOML syntax error, schema validation error, duplicate ADR detection, and invalid arguments.
- [ ] **U2.2** â€” Every user-facing command exits with code 0 on success, non-zero on failure
  - Each subcommand must follow this. `list` with no ADRs found â€” is that success (empty output, exit 0) or failure?
- [ ] **U2.3** â€” Invalid input is rejected with a helpful message, not a crash
  - The plan tests remote validation and ID validation. Missing: invalid `new_status` value in `status` subcommand, non-existent remote+id in `rename`/`status`/`lifecycle`.
- [ ] **U2.4** â€” Resources (file handles, database connections) are cleaned up on error paths
  - File handles during write operations. If `rename` fails after creating the new file but before deleting the old one, both files exist. If `status` update fails mid-write, the file could be truncated. **File write atomicity** â€” write to a temporary file then rename â€” prevents data loss on crash.
- [ ] **U2.5** â€” If the stage writes data, there is a way to read it back or verify it
  - `new` â†’ verify with `list`. `status update` â†’ verify with `status query`. `rename` â†’ verify with `list`. `lifecycle --auto` â†’ verify with `lifecycle` (no --auto).
- [ ] **U2.6** â€” If the stage creates state, there is a way to inspect the new state
  - The `.toml` files are inspectable with any text editor or `adr-format list`. The `.adr/var/lifecycle.jsonl` is human-readable. Good.
- [ ] **U2.7** â€” A user who did not write the code can verify the stage's output
  - `adr-format list` and `adr-format status` provide the verification interface.

### Test-Gap Findings

- **TG2.1** â€” No test for `list` encountering a corrupt `.toml` file. The acceptance criteria only test the happy path (valid files, formatted output). If one file in the ADR directory is malformed TOML, does `list` skip it with a warning, crash, or silently omit it? A single corrupt file should not block listing the rest.

- **TG2.2** â€” No test for file write atomicity. The `rename` and `status` subcommands deserialize, modify, and reserialize the entire file. The plan says "this is safer than awk-based in-place editing" but does not specify atomic writes. A crash between truncating the old file and writing the new content loses data. The `rename` subcommand is especially risky â€” it must write the new file AND remove the old filename.

- **TG2.3** â€” No test for `rename` with filesystem-invalid characters in the new title. The title is slugified into a filename. Characters like `/`, `\0`, and extremely long titles (exceeding filesystem path limits) need handling. The acceptance criteria test a clean rename but not edge cases.

- **TG2.4** â€” No test for lifecycle with missing `.adr/var/` directory or corrupt JSONL. The `lifecycle` subcommand reads cached work item data. If the cache directory doesn't exist, or the JSONL is malformed, the error path needs a clear message ("run `make sync` first" or equivalent).

- **TG2.5** â€” No test for `init` when the ADR directory already contains files. Should `init` refuse to overwrite an existing `0001-record-architecture-decisions.toml`? The acceptance criteria test creation but not the idempotency/overwrite case.

- **TG2.6** â€” Shell wrapper `dirname`-based path resolution (`$(dirname "$0")/../../crates/target/release/adr-format`) is fragile when the wrapper is invoked through a symlink. `dirname` resolves the symlink's location, not the target's location. The `readlink -f` or `realpath` pattern would be more robust.

- [ ] **S3.6** â€” Any new external input surface has validation at the boundary
  - `remote` and `id` arguments for locating the TOML file â€” same validation as Stage 2.

### UX Checks

- [ ] **U3.1** â€” Every error path produces a human-readable message on stderr
  - File not found, TOML parse error, missing required sections. The export reads the full Adr struct, so all Stage 1 validation errors apply.
- [ ] **U3.2** â€” Every user-facing command exits with code 0 on success, non-zero on failure
  - `export` should exit 0 on success, non-zero if the file doesn't exist or fails validation.
- [ ] **U3.3** â€” Invalid input is rejected with a helpful message, not a crash
  - Same argument validation as other subcommands (remote, id).
- [ ] **U3.4** â€” Resources (file handles, database connections) are cleaned up on error paths

- [ ] **U3.5** â€” If the stage writes data, there is a way to read it back or verify it
  - Output is stdout â€” user can pipe to file and read it. The snapshot test provides build-time verification.
- [ ] **U3.6** â€” If the stage creates state, there is a way to inspect the new state

### Test-Gap Findings

- **TG3.1** â€” No explicit idempotency test. ADR-0053 export rule 6 requires idempotent output. The acceptance criteria assert this property but no test exercises it directly (run export twice, diff the outputs, assert identical). A snapshot test alone does not prove idempotency â€” it proves the output matches a golden file once.

- **TG3.2** â€” No test for export of a minimal ADR. The snapshot test uses a "known TOML input" but the criteria don't specify testing with only required fields (no optional sections, no negative consequences, no links, empty work_item). ADR-0053 export rule 8 says empty sections are omitted â€” this rule needs explicit coverage with a minimal input.

- **TG3.3** â€” No test for `Last Updated` in Markdown output. The ADR-0053 example output includes `Last Updated: 2026-04-06` but the acceptance criteria list only "date, status, links, work-item" â€” `last_updated` is not mentioned. It is present in the TOML meta struct. The omission could mean it's exported and not verified, or not exported and nobody noticed.

- [ ] **S4.3** â€” No deserialization of untrusted input without validation
  - No new deserialization. N/A.
- [ ] **S4.4** â€” Dependencies pinned to specific versions (no wildcards)

- [ ] **S4.5** â€” File permissions on created artifacts are not overly permissive

- [ ] **S4.6** â€” Any new external input surface has validation at the boundary
  - Makefile variables (`REMOTE`, `ID`, `TITLE`, `ADR_AGENT_SKILL_FORMAT`) are passed through to the binary. Validation happens at the binary level (Stage 2).

### UX Checks

- [ ] **U4.1** â€” Every error path produces a human-readable message on stderr
  - `make build-tools` failure: Rust not installed, compilation error. These are cargo errors â€” readable. Missing binary at runtime: shell wrapper prints a clear message.
- [ ] **U4.2** â€” Every user-facing command exits with code 0 on success, non-zero on failure
  - `make build-tools` exits non-zero on cargo failure. Makefile targets propagate exit codes from the binary.
- [ ] **U4.3** â€” Invalid input is rejected with a helpful message, not a crash
  - Handled by the binary (Stage 2). Makefile passes through.
- [ ] **U4.4** â€” Resources (file handles, database connections) are cleaned up on error paths
  - No resources managed at this level. N/A.
- [ ] **U4.5** â€” If the stage writes data, there is a way to read it back or verify it
  - `make build-tools` produces a binary at `crates/target/release/adr-format`. Verify with `ls` or `adr-format --help`.
- [ ] **U4.6** â€” If the stage creates state, there is a way to inspect the new state
  - Binary existence. SKILL.md changes visible via `git diff`.
- [ ] **U4.7** â€” A user who did not write the code can verify the stage's output
  - `make build-tools && crates/target/release/adr-format --help` is the verification path.

### Test-Gap Findings

- **TG4.1** â€” No acceptance test for `adr-format --help`. The plan verifies `make build-tools` compiles and the binary exists, but no test verifies the binary runs and produces help output. A missing dynamic library or corrupted binary would pass the existence check but fail at runtime.

- **TG4.2** â€” No test for SKILL.md line count after edits. Task 4.2 acceptance criteria include "SKILL.md stays under 500 lines" but this is a manual check. Adding three table rows and a usage example could push a near-limit file over the threshold.

- **TG4.3** â€” No test for Makefile `export` target. Task 4.3 tests `new` and `list` via Makefile but not `export`. The export subcommand (Stage 3) is invoked directly via the binary, but if a Makefile target is added for it, it needs coverage.
