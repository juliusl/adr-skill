**Description:**

- Reference `qa-planning.md` for the full protocol

- Keep the addition concise — details live in the reference, not SKILL.md

Must keep SKILL.md under ~500 lines total.

**Implementation Notes:**

Insert after Step 4's closing paragraph, before Step 5:

```markdown
### Step 4b — QA Plan Generation (Mandatory)

After the plan-reviewer approves the plan (Step 4), spawn a **separate general-purpose agent** to generate the QA plan. The main executor must not write its own QA plan.

Read the [QA Planning Protocol](references/qa-planning.md) for the full QA planner prompt, procedural checklists, and finding eligibility gate.

**Workflow:**
1. Construct the QA planner prompt using the template in `qa-planning.md`.
2. Spawn a `general-purpose` agent with `mode="background"`.
3. The QA planner writes `docs/plans/<range>.<revision>.qa-plan.md`.
4. QA plan generation is mandatory — it runs for every plan, regardless of participation mode.
```

### Task 2.2: Add stage boundary QA hook    [medium]

**Description:**
Add documentation to SKILL.md describing the QA execution hook at stage boundaries. This should be added to the auto-commit section or as a new subsection near the execution workflow. Must describe:
- After stage tasks complete, before auto-commit, spawn a QA executor agent
- QA executor is a separate agent from the task executor
- Pass → commit; Fail → pause and report

Also update the planning workflow diagram to include the QA plan step.

**Implementation Notes:**

Add a `#### QA Validation at Stage Boundaries` subsection near the auto-commit documentation:

```markdown
#### QA Validation at Stage Boundaries

When a QA plan exists (`docs/plans/<range>.<revision>.qa-plan.md`), spawn a **separate general-purpose QA executor agent** after all tasks in a stage complete but before auto-commit. The agent that executed the tasks must not QA its own work.

Read the [QA Planning Protocol](references/qa-planning.md#qa-execution) for the executor prompt and pass/fail handling.

If no QA plan exists (plans generated before this feature), the stage boundary hook is a no-op — execution proceeds to auto-commit directly.
```

Update the planning workflow diagram:
```
Read ADR → Generate plan → Plan review → QA plan (4b) → Execute plan
```
