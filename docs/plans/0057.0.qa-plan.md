## Stage 1: Schema Struct Updates

### Security Checks

- [ ] **S1.1** — No user-supplied strings interpolated into SQL, shell commands, or file paths
- [ ] **S1.2** — No secrets, credentials, or API keys in committed files
- [ ] **S1.3** — `parse_adr()` validates TOML input through serde — no raw deserialization of untrusted input without validation
- [ ] **S1.4** — Dependencies pinned to specific versions in `Cargo.toml` (no wildcards)
- [ ] **S1.5** — `deny_unknown_fields` remains on top-level `Adr` struct — unknown TOML keys are rejected
- [ ] **S1.6** — `Decision` struct does NOT use `deny_unknown_fields` (required for backward compat) but is only reachable through the validated `Adr` struct

### UX Checks

- [ ] **U1.1** — `ValidationError::InvalidOptionIndex` produces a human-readable message on stderr including the out-of-bounds index and options array length
- [ ] **U1.2** — `parse_adr()` returns `Err` (not panic) when old-format TOML encounters new structs and vice versa
- [ ] **U1.3** — Invalid `chosen_option` values (negative, out of bounds) produce a validation error, not a crash
- [ ] **U1.4** — All `#[serde(default)]` annotations are present on new fields so missing TOML keys default gracefully
- [ ] **U1.5** — `serialize_adr()` round-trips new structs without data loss
- [ ] **U1.6** — After Stage 1, `cargo check` passes — the crate compiles
- [ ] **U1.7** — A developer can verify backward compat by parsing an old-format TOML string through `parse_adr()`

### Test-Gap Findings

- [ ] **TG1.1** — Plan does not require a test for partial old-format TOML (e.g., checkpoint with some named booleans AND some `items`). Verify serde handles the mixed case where both old boolean fields and new `items` array are present simultaneously.
- [ ] **TG1.2** — Plan does not test `chosen_option = 0` with an empty `options` array. This is an edge case for `validate()` — index 0 is out of bounds when options is empty.
- [ ] **TG1.3** — Plan does not verify that `Decision` with all three fields `None` (no `body`, no `chosen_option`, no `justification`) serializes and deserializes correctly. This is a valid state for a new/blank ADR.
- [ ] **TG1.4** — No test for `identifier` containing special characters (spaces, unicode). While `identifier` is typically `{remote}-{id}`, validation rules for the field format are not specified.

## Stage 2: Template and CLI Updates

### Security Checks

- [ ] **S2.1** — `generate_template()` does not interpolate user-supplied `remote` or `id` into shell commands or file paths beyond string formatting
- [ ] **S2.2** — `cmd_export()` does not write to files — output is stdout only, no injection vector
- [ ] **S2.3** — `cmd_init()` seed ADR uses hardcoded values for `chosen_option` — no user input reaches the index
- [ ] **S2.4** — No new external input surfaces introduced in template or CLI changes

### UX Checks

- [ ] **U2.1** — `cmd_export()` with `chosen_option` renders "Chose **{name}** (Option {n})" — the option name is resolved, not just the index
- [ ] **U2.2** — `cmd_export()` with legacy `body` renders the body text unchanged — no regression in existing export behavior
- [ ] **U2.3** — `cmd_export()` handles `chosen_option` pointing to a valid index without panic
- [ ] **U2.4** — `cmd_export()` gracefully handles `chosen_option` with `None` justification (no trailing empty line or crash)
- [ ] **U2.5** — `cmd_init()` produces a valid TOML file that passes `parse_adr()` and `validate()`
- [ ] **U2.6** — Template output does not include `[plan]` or `[deliverables]` sections
- [ ] **U2.7** — `meta.identifier` in template output matches `{remote}-{id}` pattern

### Test-Gap Findings

- [ ] **TG2.1** — Plan does not test `cmd_export()` when `chosen_option` is `Some(idx)` but the index is invalid (out of bounds). The export should handle this gracefully — either skip the decision section or print a warning. Current plan code silently skips when `adr.options.get(idx)` returns `None`.
- [ ] **TG2.2** — Plan does not test export when `chosen_option` is `None` and `body` is also `None` (both legacy and new fields empty). The decision section should be omitted entirely.
- [ ] **TG2.3** — Plan does not verify that `cmd_new()` also uses the new template — Task 2.1 updates `generate_template()` which `cmd_new()` calls, but no explicit test verifies the `new` subcommand output uses the new schema.

### UX Checks

- [ ] **U3.1** — All backward-compat tests use realistic old-format TOML (matching actual ADR-0051 schema output)
- [ ] **U3.2** — Test names clearly indicate what they verify (e.g., `backward_compat_old_checkpoint_format`)
- [ ] **U3.3** — Integration tests verify both the positive case (correct output) and negative case (excluded sections)
- [ ] **U3.4** — `cargo test` output shows zero failures across all crates

### Test-Gap Findings

- [ ] **TG3.1** — Plan does not include a backward-compat integration test for `cmd_export()` — only unit tests cover old-format parsing. An integration test should verify that a TOML file written in ADR-0051 format can be exported via the binary.
- [ ] **TG3.2** — Plan does not include a test for `cmd_init()` output validation — verifying the seed ADR round-trips and validates after the schema changes.
