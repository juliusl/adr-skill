#### Test-Gap Findings

**TG1.1: Resume protocol interaction with S-1.**
The plan's acceptance criteria validate that roadmap.md defines a milestone-level resume protocol. They do not validate what happens when S-1 is resumed *within* an S-2 milestone context. S-2 tracks progress via status markers on milestone headings. S-1 tracks progress via ADR states on branches. When S-2 delegates milestone N to S-1, and the session ends mid-milestone, the next invocation must know: (a) that S-2 is the outer scenario, (b) which milestone was in progress, and (c) where S-1 left off within that milestone. The acceptance criteria don't require `roadmap.md` to document this two-level resume interaction.

**TG1.2: Status marker robustness.**
The acceptance criteria check that "status marker format defined" but don't specify:
- Valid marker values (presumably: `in-progress`, `complete` — but what about `blocked`, `skipped`?)
- Default state when no marker is present (pending? unstarted?)
- Case sensitivity and whitespace tolerance (is `<!-- Status: Complete -->` valid?)
- What happens if a user manually edits a marker to an invalid value

An agent following an ambiguous spec could produce inconsistent state tracking.

**TG1.3: Roadmap format spec vs. adr-atelier.md structural alignment.**
The acceptance criterion says "Roadmap document format section matches `adr-atelier.md` structure." The actual `adr-atelier.md` has: description paragraph → `## Features` → `## Constraints` → `## Roadmap` with preamble → `### Milestone N` entries → supplementary design sections below `---`. The criterion doesn't specify which elements are mandatory vs. optional in the format spec, or how closely the format must mirror the model. A loose description could make roadmap parsing fragile.

**TG1.4: Branch naming collision between S-1 and S-2.**
S-1 uses `solve/<problem-slug>`. S-2 uses `solve/<project-slug>/milestone-<N>`. The nested path structure prevents most collisions, but if a problem slug happens to contain a `/` or if a project slug matches a prior problem slug, the branch namespace could conflict. `roadmap.md` should explicitly state that the path nesting prevents collision and how edge cases are handled.

#### Test-Gap Findings

**TG2.1: SKILL.md `description` field not updated (critical routing gap).**
The plan adds 5 routing entries to the routing *diagram* inside SKILL.md but does not mention updating the YAML frontmatter `description` field. Platform routing uses the `description` field to decide which skill to invoke. The current description contains "implement milestones X to Y" but does NOT contain "roadmap," "process roadmap," "continue roadmap," or "roadmap progress." If the description is not updated, the platform will not route these trigger phrases to solve-adr — the routing diagram would be a lie.

This is the highest-priority finding in this QA plan. The plan's acceptance criteria don't catch it because they focus on the procedure table and routing diagram, not the frontmatter.

**TG2.2: Routing trigger phrase overlap with implement-adr.**
The plan adds "Implement milestone N" as an S-2 trigger. `implement-adr`'s description triggers on "implement ADR 0013," "let's implement," "start implementing." The phrase "implement milestone 3" is structurally similar to "implement ADR 0013" — both are "implement [thing] [number]." If a user says "implement milestone 3," the platform may route to implement-adr instead of solve-adr, depending on description match scoring. The acceptance criteria don't validate disambiguation.

**TG2.3: S-2 section content completeness.**
The acceptance criterion says "S-2 section exists with lifecycle diagram and resume behavior." It doesn't specify whether the S-2 section should include:
- Branch naming convention (plan mentions it should)
- Composition note explaining S-1 delegation
- What happens when a milestone has no objectives (empty milestone)

#### Test-Gap Findings

**TG2.4: Insufficient should-trigger coverage.**
The plan introduces 5 trigger phrases but only requires "at least 3 should-trigger queries." This leaves 2 triggers untested. All 5 should be tested — partial coverage masks routing regressions.

**TG2.5: Insufficient should-not-trigger coverage.**
The plan lists only 1 should-not-trigger query: "create a roadmap document." This is insufficient to validate disambiguation. Missing negative cases:
- "implement ADR 0013" — must NOT trigger solve-adr (belongs to implement-adr)
- "create a project roadmap" — must NOT trigger solve-adr (general doc authoring)
- "review this roadmap" — must NOT trigger solve-adr (no review scenario)

Without adequate negative coverage, the eval_queries.json cannot detect false-positive routing.

**TG2.6: Existing query disruption.**
The current `eval_queries.json` has a should-trigger entry: "Drive this roadmap — implement the remaining proposed ADRs." Adding new roadmap triggers changes the semantic space. The acceptance criteria don't require re-validating that existing entries still behave correctly after the description field changes (see TG2.1).

#### Test-Gap Findings

**TG3.1: Status transition prerequisite.**
The plan says ADR-0049 will be in `Planned` status at execution time (transitioned from `Proposed` during I-5). The ADR currently says `Proposed`. If the pre-execution transition (I-5) is skipped or forgotten, the implementer will find `Proposed` instead of `Planned` and may be confused about which value to expect. The task should include a guard: "verify current status before changing."

**TG3.2: `Last Updated` date.**
The acceptance criteria say "No other ADR content is modified" — but changing status from Planned to Accepted without updating `Last Updated` leaves stale metadata. This criterion may conflict with good practice. Recommendation: explicitly include `Last Updated` update, or note that it's deferred.
