## Stage 1: Crate Scaffold and Build Guard

### Security Checks

- [ ] **S1.1** — `build.rs` does not interpolate user input into shell commands; `git rev-parse` is called with hardcoded args only
- [ ] **S1.2** — No secrets or credentials in `Cargo.toml` or `build.rs`
- [ ] **S1.3** — No deserialization of untrusted input — `build.rs` reads filesystem metadata only
- [ ] **S1.4** — Dependencies (`clap`, `rust-embed`, `dirs`, `tempfile`) are pinned to major versions; verify no wildcard versions
- [ ] **S1.5** — No new files with overly permissive access created at build time
- [ ] **S1.6** — No new external input surfaces in this stage (build-time only)

### UX Checks

- [ ] **U1.1** — `build.rs` panic messages are human-readable and state which directory is missing
- [ ] **U1.2** — Build failure exits non-zero (Cargo handles this via panic)
- [ ] **U1.3** — Invalid workspace layout (missing dirs) produces a clear compile error, not an opaque Rust backtrace
- [ ] **U1.4** — No runtime resources opened in this stage (build script only)
- [ ] **U1.5** — `cargo check` success confirms the scaffold works — the user can verify
- [ ] **U1.6** — Workspace members list in `Cargo.toml` is inspectable via `cargo metadata`
- [ ] **U1.7** — A contributor who clones the repo can run `cargo check -p adr-skills` to verify the scaffold

### Test-Gap Findings

- **TG1.1** — The plan tests that build fails when directories are "absent" (rename test), but does not test when directories exist but are **empty**. The `build.rs` code checks for emptiness, but the acceptance criteria only say "absent." If the empty-directory guard regresses, dev tests pass.
- **TG1.2** — The plan does not verify that `GIT_COMMIT_SHA` env var is actually consumable at compile time. `build.rs` emits it, but no test checks that `env!("GIT_COMMIT_SHA")` compiles successfully. A silent failure (e.g., git not installed in CI) would produce a build with no version info.
- **TG1.3** — The `build.rs` walkdir function does not handle symlinks. If `src/skills/` contains symlinks (unlikely but possible), `rerun-if-changed` tracking breaks silently — changes to symlink targets do not trigger rebuilds.

## Stage 2: CLI Structure and Embed Definitions

### Security Checks

- [ ] **S2.1** — `--prefix` path is used as-is for filesystem operations; verify it is not interpolated into shell commands
- [ ] **S2.2** — No secrets in the CLI struct definitions or help text
- [ ] **S2.3** — RustEmbed `#[folder]` paths are hardcoded compile-time constants — no runtime deserialization
- [ ] **S2.4** — RustEmbed `rust-embed = "8"` uses a major-version pin; verify the resolved version has no known CVEs
- [ ] **S2.5** — No files created in this stage
- [ ] **S2.6** — `--prefix` is a new external input surface; verify path validation exists before use in Stage 3

### UX Checks

- [ ] **U2.1** — `adr-skills` with no subcommand prints help text (not an error or crash)
- [ ] **U2.2** — `adr-skills --help`, `adr-skills install --help`, `adr-skills init --help` all exit 0 with useful output
- [ ] **U2.3** — Invalid subcommand (e.g., `adr-skills foo`) produces a clap error message listing valid subcommands
- [ ] **U2.4** — No runtime resources opened in this stage (struct definitions only)
- [ ] **U2.5** — `SkillAssets::iter()` and `AgentAssets::iter()` can enumerate embedded files — user can verify content is embedded
- [ ] **U2.6** — Embedded asset list is inspectable programmatically (iterator output)
- [ ] **U2.7** — A user running `adr-skills --version` sees version + git SHA, confirming which build they have

### Test-Gap Findings

- **TG2.1** — The plan verifies `SkillAssets::iter()` returns entries for "all 4 skill directories" and `AgentAssets::iter()` for "all 8 `.agent.md` files." These counts are hardcoded and will break silently when skills/agents are added or removed. The dev tests catch today's count but not tomorrow's drift.
- **TG2.2** — The plan does not test what happens when `--prefix` is an empty string, a relative path, or a path with spaces/special characters. Clap parses it as `PathBuf`, but downstream use (Stage 3) may fail in surprising ways.
- **TG2.3** — No test verifies that `--version` output actually contains the git SHA. If `build.rs` silently fails to emit `GIT_COMMIT_SHA` (see TG1.2), the version string format changes but no test catches it.

## Stage 3: Install and Init Logic

### Security Checks

- [ ] **S3.1** — `--prefix` is joined with hardcoded subdirectory names (`skills/`, `agents/`); verify no path traversal via crafted `--prefix` (e.g., `--prefix /etc/../../sensitive`)
- [ ] **S3.2** — No secrets written to installed files; embedded content comes from repo source
- [ ] **S3.3** — Embedded file content is not deserialized — it is written as raw bytes to disk
- [ ] **S3.4** — No new dependencies introduced beyond Stage 1
- [ ] **S3.5** — Installed files inherit default umask permissions; verify no explicit `chmod 777` or similar
- [ ] **S3.6** — `init_project` writes `.gitignore` and `preferences.toml` with hardcoded content — no user input in file content

### UX Checks

- [ ] **U3.1** — Permission denied, disk full, and other I/O errors produce human-readable messages listing what was written and what failed
- [ ] **U3.2** — `install skills`, `install agents`, `install all`, `init`, and `setup` all exit 0 on success, non-zero on failure
- [ ] **U3.3** — `--prefix` pointing to a non-writable directory produces a clear error, not a panic
- [ ] **U3.4** — `remove_dir_all` followed by write failure leaves the target in a partially-written state; the error message must list what was written so the user can recover
- [ ] **U3.5** — After install, the user can `ls ~/.copilot/skills/` to verify files were written
- [ ] **U3.6** — After init, the user can `cat .adr/preferences.toml` to inspect the created config
- [ ] **U3.7** — The parity check (Task 5.1: `diff -r` between make output and binary output) lets a third party verify correctness

### Test-Gap Findings

- **TG3.1** — The plan tests idempotent re-install (overwrite produces same result) but does not test the **destructive intermediate state**. `remove_dir_all` deletes existing files before writing new ones. If the process is interrupted between delete and write (crash, SIGKILL, disk full), the user loses their installed skills with no recovery path. Dev tests run to completion, so this race is invisible.
- **TG3.2** — The plan tests `install_skills` extracts "all 4 skill directories" — same hardcoded-count issue as TG2.1. The unit test should compare against the actual embedded asset count, not a magic number.
- **TG3.3** — `init_project` skips existing files, but the plan does not test what happens when `.adr/` exists as a **file** (not a directory). `create_dir_all` would fail, and the error handling path is untested.
- **TG3.4** — The plan does not test file content integrity after install. A unit test verifies file count and directory structure but not that file contents match the embedded originals. Silent truncation or encoding issues would pass dev tests.
- **TG3.5** — The `setup` command calls `install_all` then `init_project`. If `install_all` succeeds but `init_project` fails, the plan does not specify whether the exit code is 0 or non-zero, or what the user sees. Partial success reporting is unspecified.
- **TG3.6** — `preferences.toml` content must match current Makefile `init-project` output, but no test mechanically compares them. Content drift between the two install paths is possible.

- [ ] **S4.2** — No secrets in Makefile targets or README documentation
- [ ] **S4.3** — No deserialization in this stage
- [ ] **S4.4** — No new dependencies introduced
- [ ] **S4.5** — No new files created with permissions concerns
- [ ] **S4.6** — No new input surfaces

### UX Checks

- [ ] **U4.1** — `make build-adr-skills` failure produces cargo's standard error output (readable)
- [ ] **U4.2** — `make build-adr-skills` exits 0 on success, non-zero on failure
- [ ] **U4.3** — Not applicable — no new user input in this stage
- [ ] **U4.4** — Not applicable — no runtime resources
- [ ] **U4.5** — Build produces a binary at a known path; user can verify with `ls` or `file`
- [ ] **U4.6** — README documents all subcommands; user can read to understand available operations
- [ ] **U4.7** — `make build-adr-skills && ./target/release/adr-skills --help` lets any contributor verify the build

### Test-Gap Findings

- **TG4.1** — The plan does not verify that the README migration table matches the ADR's subcommand table. If the ADR subcommand table is updated in a future revision, the README could drift.
- **TG4.2** — The plan does not test that `make build-tools` actually includes `adr-skills` in its output. The acceptance criterion says "includes `adr-skills` in its build" but the verification is manual — a typo in the Makefile dependency would pass review.

## Stage 5: Validation

### Security Checks

- [ ] **S5.1** — Validation commands use `--prefix` with test directories — no writes to sensitive system paths
- [ ] **S5.2** — No secrets in test invocations
- [ ] **S5.3** — No deserialization in validation
- [ ] **S5.4** — No new dependencies
- [ ] **S5.5** — Test output written to temp directories, cleaned up after
- [ ] **S5.6** — No new input surfaces

### UX Checks

- [ ] **U5.1** — Test failures produce standard cargo/make error output
- [ ] **U5.2** — `make test`, `cargo test`, and `make check-refs` all exit non-zero on failure
- [ ] **U5.3** — Not applicable — validation uses fixed inputs
- [ ] **U5.4** — Temp directories used in parity check are cleaned up
- [ ] **U5.5** — `diff -r` output shows exactly what differs between make and binary installs
- [ ] **U5.6** — Test results are printed to stdout/stderr
- [ ] **U5.7** — Any contributor can run the validation commands independently

### Test-Gap Findings

- **TG5.1** — The parity check (`diff -r` between make output and binary output) is listed as a manual verification. If this is not automated as a test, it will be run once and never again. Future changes to embedded content or Makefile targets could break parity silently.
- **TG5.2** — The plan references `/tmp/test-install` and `/tmp/test-full` as test directories in multiple tasks. These are hardcoded paths that may conflict with other processes or users on shared systems. Using `tempfile::tempdir()` in tests is correct, but the manual verification steps use fixed paths.
