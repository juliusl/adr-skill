## Stage 1: Version-Aware Install (ADR-0071)

### Security Checks

- [ ] **S1.1** — No user-supplied strings interpolated into SQL, shell commands, or file paths (version marker writes a compile-time constant; `--prefix` path is validated by `resolve_base()`)
- [ ] **S1.2** — No secrets, credentials, or API keys in committed files (version string is semver + commit SHA only)
- [ ] **S1.3** — No deserialization of untrusted input without validation (`.version` file is read as a plain string and compared; no parsing)
- [ ] **S1.4** — Dependencies pinned to specific versions (no new dependencies introduced by this stage)
- [ ] **S1.5** — File permissions on `.version` marker are not overly permissive (inherits process umask, same as existing installed files)
- [ ] **S1.6** — `.version` file read path has validation: graceful handling when file is missing, empty, or unreadable

### UX Checks

- [ ] **U1.1** — Every error path produces a human-readable message on stderr (`write_version_marker` uses `eprintln!` on failure; version read failure handled gracefully)
- [ ] **U1.2** — `--dry-run` exits with code 0 on success; version-match "Already up-to-date" exits with code 0
- [ ] **U1.3** — Invalid flag combinations are rejected by clap (boolean flags have no invalid values; `--dry-run --force` is a valid combination with defined semantics)
- [ ] **U1.4** — No file handles or resources leaked on error paths (uses `fs::write`/`fs::read_to_string` which close handles on drop)
- [ ] **U1.5** — `--dry-run` output reports file count and target directory; `.version` file content is human-readable
- [ ] **U1.6** — `.version` file can be inspected with `cat` to verify installed version
- [ ] **U1.7** — A user can verify install state by: (a) reading `--dry-run` output, (b) reading `.version`, (c) checking install directory exists

### Test-Gap Findings

- **TG1.1** — Version file read error handling. Dev criteria test the happy paths (match, mismatch, missing). No test covers an unreadable `.version` file (e.g., permissions denied). If `fs::read_to_string` fails, the code should treat it as "no version" and proceed with install, not crash.
- **TG1.2** — Version string trailing whitespace. `fs::write` writes the version string; `fs::read_to_string` reads it back. If either adds a trailing newline, string comparison fails and every install reports "version mismatch" even when versions match. Dev criteria test "version match skips install" but don't verify the string round-trips exactly.
- **TG1.3** — Partial install recovery. The existing pattern is `remove_dir_all` then write. If install fails midway after removing old files, the `.version` file from the previous install is gone. The new version-check logic doesn't make this worse, but it doesn't improve it either. Dev tests don't cover mid-install failure.
- **TG1.4** — Dry-run combined with force. `--dry-run --force` is accepted by clap. Dev criteria define dry-run and force independently but don't specify the combined behavior. Logical expectation: dry-run takes precedence (preview only, never write).

## Stage 3: Configurable Setup Defaults (ADR-0073)

### Security Checks

- [ ] **S3.1** — User-supplied flag values (`--tpm`, `--review`, `--participation`, etc.) are interpolated into `preferences.toml` via `format!()`. Values containing TOML metacharacters (quotes, newlines, backslashes, brackets) could produce malformed or injected TOML content. Verify that string values are properly escaped or validated before interpolation.
- [ ] **S3.2** — No secrets, credentials, or API keys in committed files (flag default values are agent names, not credentials)
- [ ] **S3.3** — No deserialization of untrusted input without validation (TOML is generated, not parsed from external input)
- [ ] **S3.4** — Dependencies pinned to specific versions (verify no new wildcard dependencies added for TOML generation)
- [ ] **S3.5** — File permissions on `preferences.toml` are not overly permissive (same as existing `init_project` behavior)
- [ ] **S3.6** — New external input surface (10+ CLI flags) has validation at the boundary: `--participation` accepts only valid values; `--code-review` comma parsing handles edge cases

### UX Checks

- [ ] **U3.1** — Invalid `--participation` value (not "guided" or "autonomous") produces a clear error on stderr, not a silently wrong config file
- [ ] **U3.2** — Every user-facing command exits with code 0 on success, non-zero on failure (init with flags follows existing pattern)
- [ ] **U3.3** — Empty flag values (`--tpm ""`) are rejected with a helpful message or produce a sensible default, not a broken TOML file
- [ ] **U3.4** — Resources cleaned up on error paths (no change from existing init behavior)
- [ ] **U3.5** — Generated `preferences.toml` can be read back and is valid TOML (parseable by a TOML parser)
- [ ] **U3.6** — After `init --participation guided`, user can verify the setting by reading `preferences.toml`
- [ ] **U3.7** — `adr-skills init --help` lists all flags with descriptions and default values; a new user can determine the full set of options without reading source code

### Test-Gap Findings

- **TG3.1** — TOML injection via flag values. No dev test verifies that flag values containing TOML metacharacters produce valid output. Example: `--tpm 'evil\nkey = "injected"'` would write a newline into the TOML, breaking the file structure. The `format!()` approach has no escaping.
- **TG3.2** — Participation mode validation. The plan uses `String` type, not an enum. Any string is accepted: `--participation "typo"` produces `participation = "typo"` which is syntactically valid TOML but semantically wrong. The skill that reads this config would fail later with an unclear error.
- **TG3.3** — Comma-separated parsing edge cases. `--code-review "a,,b"` produces an empty element. `--code-review ","` produces two empty strings. `--code-review ""` produces a single empty string. No dev test covers these.
- **TG3.4** — Empty string flags. `--tpm ""` writes `tpm = ""` to TOML — syntactically valid but semantically broken. No validation prevents empty agent names.
- **TG3.5** — Setup-to-init flag pass-through. Dev criteria say "Setup passes its flags through to init" but don't test individual flag values surviving the pass-through. A wiring bug could silently drop a flag.
