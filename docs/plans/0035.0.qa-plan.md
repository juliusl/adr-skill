## Stage 1: Adapter Scripts

### Security Checks

| ID | Checklist Item | Finding |
|----|---------------|---------|
| S1.1 | No user-supplied strings interpolated into SQL, shell, or file paths (injection) | **FINDING.** The `gh_adapter` and `ado_adapter` consume JSON from MCP tool output. If adapters use `jq` to build normalized JSON, values are safe — `jq` handles escaping. But if any adapter constructs JSON via shell variable interpolation (e.g., `echo "{\"title\": \"$title\"}"`), then remote-controlled fields like `title` or `description` can inject arbitrary JSON keys or break structure. The `description` field is truncated to 500 chars — if truncation uses shell substring expansion (`${var:0:500}`) on content containing backslashes or double quotes, the resulting JSON is malformed. The `local_adapter` generates an ID from a timestamp hash — piping to `sha256sum` is safe, but using `eval` or unquoted `$()` on user-supplied title would be injectable. |
| S1.2 | No secrets, credentials, or API keys in committed files | N/A — no credentials involved. Adapters consume already-fetched MCP output; they do not call APIs or store tokens. |
| S1.3 | No deserialization of untrusted input without validation | **FINDING.** All three adapters deserialize JSON input. The `gh_adapter` and `ado_adapter` parse remote API responses (passed through MCP). If input is not valid JSON, `jq` will fail with a parse error — the script must handle this gracefully (non-zero exit + stderr message) rather than producing partial or empty output on stdout. The `validate_work_item` function itself deserializes input to check schema conformance — if it silently returns 0 on malformed JSON (e.g., `jq` returns null for missing fields and the check passes), invalid data propagates downstream. |
| S1.4 | Dependencies pinned to specific versions | **FINDING.** The script depends on `jq` for JSON processing. The plan does not specify a minimum `jq` version. Older `jq` versions (< 1.5) lack features like `//` (alternative operator) and `try-catch`. If the implementation uses `jq 1.6+` features (e.g., `$ENV`, `limit`, `builtins`), it will fail silently or crash on systems with `jq 1.5`. The script should document the minimum `jq` version or check at source-time. |
| S1.5 | File permissions on created artifacts | N/A — adapters write JSON to stdout. No files are created on disk by this script. |
| S1.6 | New external input surface has validation at boundary | **FINDING.** The `local_adapter` accepts a user-provided title as input. This is the only adapter where user-supplied freeform text enters the pipeline without first passing through an API response. The plan does not specify validation for the title argument — an empty title, a title containing only whitespace, or an extremely long title (>10,000 chars) should be rejected. The `gh_adapter` and `ado_adapter` receive structured JSON — but must validate that required fields (`number`/`System.Id`, `title`/`System.Title`) exist in the input before attempting transformation. |

### UX Checks

| ID | Checklist Item | Finding |
|----|---------------|---------|
| U1.1 | Every error path produces human-readable message on stderr | **FINDING.** The plan specifies four functions (`validate_work_item`, `gh_adapter`, `ado_adapter`, `local_adapter`) but does not describe error messages for any of them. What happens when: (a) `gh_adapter` receives JSON without a `number` field? (b) `ado_adapter` receives JSON without `System.WorkItemType`? (c) `local_adapter` is called with no arguments? (d) `validate_work_item` receives empty input? Each function should emit a descriptive error to stderr identifying which adapter failed and what field was missing. |
| U1.2 | Every command exits 0 on success, non-zero on failure | **FINDING.** The plan says "`validate_work_item` returns 0 for valid JSON, non-zero for invalid." Good — but the adapter functions have no specified exit behavior. If `gh_adapter` receives malformed input, does it exit non-zero or does `jq` exit non-zero inside a pipeline (which `set -e` may or may not catch depending on pipe position)? Each adapter must explicitly `return 1` (not just rely on `jq` failure propagation) on error, to ensure callers see the failure. |
| U1.3 | Invalid input rejected with helpful message, not crash | **FINDING.** The plan's acceptance criteria test happy paths: "produces valid normalized JSON from GitHub-style input." No criteria test what happens with: (a) input missing required fields, (b) input with unexpected types (e.g., `number` as string instead of integer), (c) input with null values for required fields, (d) completely empty input (`echo "" | gh_adapter`). A bare `jq` crash (e.g., `null (null) and string ("open") cannot be iterated`) is not a helpful message. |
| U1.4 | Resources cleaned up on error paths | N/A — adapters are pure functions (stdin → stdout). No temp files, no state to clean up. |
| U1.5 | Written data can be read back or verified | **FINDING.** Adapters write JSON to stdout. The plan includes `validate_work_item` which can verify output. But the plan does not specify whether adapters should pipe their output through `validate_work_item` before emitting it (validate-at-source) or whether callers are responsible for validation (validate-at-consumer). If validate-at-source, an adapter that produces invalid JSON will self-report. If validate-at-consumer, invalid output propagates silently to the cache layer. |
| U1.6 | Created state can be inspected | N/A — no persistent state created. Output is transient JSON on stdout. |
| U1.7 | Non-author can verify output through tool interface | OK — the JSON output is machine-readable and can be piped through `jq .` for pretty-printing or `validate_work_item` for schema check. No additional finding. |

### Test-Gap Findings

| ID | Finding |
|----|---------|
| TG1.1 | **No test for malformed JSON input to adapters.** The plan's tests provide known-good input. No test passes truncated JSON, empty string, or non-JSON text (e.g., plain HTML error page) to `gh_adapter` or `ado_adapter`. MCP tool output can fail — adapters must handle this. |
| TG1.2 | **No test for GitHub issue with multiple type-relevant labels.** The ADR says `bug` label maps to `type: "bug"`. What if an issue has both `bug` and `enhancement` labels? The plan only tests "with bug label" vs "without" — no test for label priority or conflict. |
| TG1.3 | **No test for ADO work item types outside the mapping table.** The ADR lists Bug, User Story, Task, Feature, Epic → normalized types. What about "Test Case", "Impediment", "Issue" (ADO's own Issue type)? These should map to `type: "other"` per the ADR. No test covers this. |
| TG1.4 | **No test for ADO state values outside the mapping table.** ADO projects can customize states (e.g., "Design", "Ready for Review"). The plan tests New/Active/Resolved/Closed but not custom states. These should either map to `other` or default to `open`. |
| TG1.5 | **No test for `description` truncation at exactly 500 characters.** The ADR specifies "first 500 chars." No test verifies that a 600-character body is truncated to 500, or that a 499-character body is preserved intact, or that truncation handles multi-byte UTF-8 correctly. |
| TG1.6 | **No test for `local_adapter` ID uniqueness.** The ADR says local IDs are timestamp-based hashes. Two rapid calls to `local_adapter` (within the same second) could produce the same ID. No test verifies uniqueness or documents the collision risk. |
| TG1.7 | **No test for `labels` field as empty array vs missing.** The normalized schema shows `"labels": ["string"]`. If a GitHub issue has no labels, does `gh_adapter` emit `"labels": []` (empty array) or omit the field? The `validate_work_item` function must enforce one convention. |
| TG1.8 | **No test for `url` field for `local` remote.** The ADR says URL is "optional, empty for local." No test verifies that `local_adapter` emits `"url": ""` (empty string) vs `"url": null` vs omitting the field. |

### Test-Gap Findings

| ID | Finding |
|----|---------|
| TG2.1 | **No negative tests for adapter functions.** Task 2.1 describes four test scripts covering success cases: GitHub adapter with valid input, ADO adapter with valid input, local adapter with valid input, validation with valid and invalid inputs. Only `wi-adapter-validate.sh` touches error behavior, and only for the validation function — not for adapter functions receiving bad input. Missing: (a) `gh_adapter` with missing `number` field, (b) `ado_adapter` with missing `System.Id`, (c) `local_adapter` with empty title. |
| TG2.2 | **Test harness runs under fixed `ADR_DATE=2026-01-15`.** The adapter tests produce timestamps via `created`/`updated` fields. If adapters use `date` internally (e.g., `local_adapter` generating `created` timestamp), the output will vary per run and diff tests will fail. Either the tests must mock the date or the `.expected` files must use a pattern matcher. The plan does not address this. |
| TG2.3 | **No integration test verifying adapter output is consumable by downstream tools.** The plan tests each adapter in isolation. No test pipes `gh_adapter` output into `validate_work_item` as part of the same test — the acceptance criteria only say each passes individually. A round-trip test (`gh_adapter | validate_work_item`) would verify contract alignment. |
| TG2.4 | **`make -C src/skills/author-adr/scripts check` acceptance criterion assumes test harness discovers new test files.** The Makefile uses `$(wildcard tests/*.sh)` — this auto-discovers tests. But if a new test file is named `wi-adapter-gh.sh` and the corresponding `.expected` is missing, `make check` will fail with a Make error, not a test failure. The plan must ensure `.expected` files are committed alongside test scripts. |

## Recommendations

| # | Stage | ID | Classification | Finding | Recommendation |
|---|-------|----|----------------|---------|----------------|
| 1 | 1 | S1.1 | Quality concern | Adapters may construct JSON via shell interpolation, allowing remote-controlled fields (`title`, `description`) to break JSON structure or inject keys. | Use `jq` for all JSON construction — never shell string interpolation. Build normalized JSON via `jq` filters (e.g., `jq '{remote: "gh", id: (.number \| tostring), title: .title}'`), which handles escaping automatically. |
| 2 | 1 | S1.3 | Quality concern | Adapters do not specify behavior for malformed JSON input. `jq` parse errors produce raw error messages; empty or partial output may propagate downstream. | Each adapter must check `jq` exit code explicitly. On parse failure, emit "ERROR: <adapter_name>: invalid JSON input" to stderr and return 1. Do not write partial JSON to stdout. |
| 3 | 1 | S1.4 | Quality concern | `jq` dependency is undocumented. Older versions lack features the implementation may use. | Add a comment at the top of `work-item-adapters.sh` documenting `jq >= 1.5` as a requirement. Optionally, add a version check at source-time: `jq --version 2>/dev/null \| grep -qE '1\.[5-9]'`. |
| 4 | 1 | S1.6 | Quality concern | `local_adapter` accepts freeform title without validation. Empty, whitespace-only, or extremely long titles are not rejected. | Validate title is non-empty and ≤ 1000 chars in `local_adapter` before generating normalized JSON. Emit error to stderr on validation failure. |
| 5 | 1 | U1.1 | Quality concern | No error messages specified for adapter functions. Missing fields in input produce raw `jq` errors instead of actionable diagnostics. | Each adapter should validate required input fields before transformation. Example: `gh_adapter` checks that `.number` and `.title` exist; emits "ERROR: gh_adapter: input missing required field 'number'" on failure. |
| 6 | 1 | U1.2 | Quality concern | Adapter exit behavior on failure is unspecified. `jq` failures inside pipelines may not propagate under `set -e` depending on pipe position. | Each adapter function must end its error path with an explicit `return 1`. Avoid relying on `set -e` propagation through `jq` pipelines — use `jq ... \|\| { echo "ERROR: ..." >&2; return 1; }`. |
| 7 | 1 | U1.3 | Quality concern | No acceptance criteria for invalid-input behavior. Plan only tests happy paths for adapters. | Add acceptance criteria: "gh_adapter returns non-zero and emits error on stderr when input lacks `number` field." Same for `ado_adapter` (missing `System.Id`) and `local_adapter` (empty title). |
| 8 | 1 | U1.5 | Quality concern | Plan does not specify whether adapters self-validate output or callers must validate. | Recommend validate-at-source: each adapter pipes its output through `validate_work_item` before writing to stdout. This catches schema drift at the point of origin. |
| 9 | 1 | TG1.1 | Quality concern | No test for malformed input. MCP tool output can return errors, HTML, or truncated JSON — adapters must handle this. | Add a test case per adapter with malformed input: `echo "not json" \| gh_adapter` should exit non-zero with an error message. Include in `wi-adapter-gh.sh` as an additional test scenario. |
| 10 | 1 | TG1.3 | Quality concern | Unmapped ADO work item types (Test Case, Impediment, etc.) have no test coverage. The ADR says they map to `other` but no test verifies this. | Add a test in `wi-adapter-ado.sh` with `System.WorkItemType: "Impediment"` and verify output has `"type": "other"`. |
| 11 | 1 | TG1.5 | Quality concern | Description truncation at 500 chars is specified in the ADR but untested. Multi-byte UTF-8 truncation could produce invalid characters. | Add a test with a 600-char description body. Verify output `description` is exactly 500 chars. If using `jq`, use `.[0:500]` which handles UTF-8 correctly. |
| 12 | 1 | TG1.7 | Quality concern | Schema ambiguity: `labels` as empty array vs missing field vs null. The validator and adapters must agree on the canonical representation. | Define convention in `validate_work_item`: `labels` must always be an array (empty `[]` if no labels). Adapters must emit `"labels": []`, never omit the field. Add a test for a GitHub issue with zero labels. |
| 13 | 2 | U2.2 | Quality concern | Tests verifying `validate_work_item` with invalid input will exit non-zero, causing `set -e` to abort the test script before subsequent assertions run. | In `wi-adapter-validate.sh`, wrap invalid-input tests in subshells or use `\|\| true`: `validate_work_item <<< '{}' && echo "FAIL: should have rejected" \|\| echo "PASS: rejected invalid"`. |
| 14 | 2 | TG2.2 | Quality concern | `local_adapter` may use `date` for `created`/`updated` timestamps, producing non-deterministic output that breaks diff-based tests. | Either: (a) accept a `WI_DATE` environment variable override in `local_adapter` (parallel to `ADR_DATE` in the format script), or (b) use `jq` to strip/normalize timestamps in test scripts before diffing. |
| 15 | 2 | TG2.3 | Quality concern | No round-trip integration test. Adapter output is tested against expected JSON, but never piped through `validate_work_item` in the same test. Schema could drift between adapter output and validator expectations. | Add a round-trip assertion in each adapter test: after checking expected output, pipe the output through `validate_work_item` and assert exit code 0. |
| 16 | 3 | TG3.1 | Quality concern | Plan says `Status: Planned` → `Status: Accepted`, but ADR currently reads `Status: Proposed`. Implementer will encounter a mismatch. | Correct the plan to reflect actual starting status (`Proposed`), or update the ADR status to `Planned` first if that is the intended pre-implementation status. |
| 17 | 1 | TG1.2 | Preference | Multiple type-relevant labels on a GitHub issue (e.g., `bug` + `enhancement`) produce ambiguous type mapping. Edge case — GitHub label conventions vary by project. | Defer — document that `bug` takes priority. If needed later, add a label-priority list to adapter configuration. |
| 18 | 1 | TG1.4 | Preference | ADO custom states (e.g., "Design", "Ready for Review") are untested. These are project-specific and uncommon in default ADO configurations. | Defer — recommend mapping unknown ADO states to `open` as a safe default. Document this in a code comment. |
| 19 | 1 | TG1.6 | Preference | `local_adapter` ID uniqueness on rapid successive calls (same-second collision) is a theoretical risk but unlikely in practice — ADRs are not created in batch. | Defer — document the collision risk in a code comment. If it becomes a problem, add a random suffix to the hash input. |
| 20 | 1 | TG1.8 | Preference | `url` field representation for `local` remote (empty string vs null vs omitted) is a schema detail that should be decided but is low-impact. | Defer — define `"url": ""` as the convention for local. Add to `validate_work_item` as a non-required string field. |
