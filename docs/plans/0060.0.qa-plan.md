### Test-Gap Findings

- **TG1.1** — **Config-to-C-1 consistency gap.** The dev acceptance criteria check individual sections in isolation (Task 1.1 checks config section; Task 1.2 checks C-1 substeps). No criterion verifies that the config documentation and C-1 substeps use consistent terminology for the normalized list. For example, if the config section says "normalized to a single-element list" but C-1a says "the list is empty," a reader must infer that "list" in C-1a is the same normalized list from config parsing. Verify both sections use the same term consistently.

- **TG1.2** — **Summary table staleness.** Task 1.2 criteria say "Summary table (C-1 through C-1f) descriptions updated for plural." The dev plan does not specify which exact descriptions change. A criterion could pass ("descriptions updated") while leaving one row singular. Verify every row in the C-1 summary table that references the reviewer uses plural or list-aware language.

- **TG1.3** — **Backward-compatible phrasing completeness.** Task 1.1 criteria check that the key description "documents string-or-array type," but do not check whether the existing prose about the single-string case is preserved. A rewrite could drop the explanation of what happens when `code_review = ""` (empty string). Verify the updated key description still documents the skip-when-empty behavior.

- **TG1.4** — **Line count regression.** Both tasks add prose. The dev criteria check `SKILL.md stays under 500 lines` only in Task 1.1. Task 1.2 also adds lines but has no line count criterion. Verify line count after both tasks complete.
