- [ ] **S1.3** — No deserialization of untrusted input — dispatch keys are read from a local `preferences.toml` under the user's control, not from external sources
- [ ] **S1.4** — N/A — no new dependencies introduced (documentation-only change). Logged: no dependency pinning check needed.
- [ ] **S1.5** — N/A — no new files created; existing SKILL.md permissions unchanged. Logged: no file permission check needed.
- [ ] **S1.6** — New dispatch key values are validated at the boundary: empty string means skip, non-empty is treated as agent reference. The instruction text must document this validation rule explicitly.

### UX Checks

- [ ] **U1.1** — Skip behavior for unconfigured hooks produces a clear log message (e.g., "Step 4a skipped — no UX/DX review agents configured"), not a silent no-op
- [ ] **U1.2** — N/A — no user-facing commands introduced; this is instruction text. Logged: no exit code check needed.
- [ ] **U1.3** — Invalid dispatch values (whitespace-only strings) are addressed — instructions must specify how whitespace-only values are handled (consistent with existing `tech_writer` behavior: "Values containing only whitespace are treated as empty")
- [ ] **U1.4** — N/A — no resources opened; documentation-only change. Logged: no resource cleanup check needed.
- [ ] **U1.5** — The updated SKILL.md can be read back and verified: hook table has 6 rows, TOML block has 6 keys, description mentions six hook points
- [ ] **U1.6** — The dispatch configuration state is inspectable: a user can read `preferences.toml` and the hook table to confirm which hooks are active
- [ ] **U1.7** — A developer unfamiliar with the change can verify the dispatch section by counting keys (6), rows (6), and confirming the description says "six hook points"

### Test-Gap Findings

- **TG1.1** — Dev criteria for Task 1.1 check that the TOML example contains all 6 keys and the hook table has 6 rows, but do not verify **consistency between the TOML block and the hook table**. If a key appears in the TOML but not the table (or vice versa), all 4 dev criteria could pass while the documentation is internally inconsistent.
- **TG1.2** — Dev criteria for Task 1.2 verify that A-0 step 1 lists all 6 keys and defaults are `""`, but do not verify that the **default values in A-0 match the defaults in the hook table**. A mismatch between A-0 defaults and the hook table defaults would leave the instructions contradictory.
- **TG1.3** — Neither task's criteria verify that the **"Contract: same instructions, configurable executor" paragraph** still applies to the new hooks. The new hooks have a different contract: they dispatch agents with their own review procedures, not shared reference instructions. If the contract paragraph is not updated, it overstates what the new hooks do.

## Stage 2: Evaluation Checkpoint Integration

### Security Checks

- [ ] **S2.1** — Step 4a dispatches agents via `task` tool with ADR content as payload. Verify that the instructions do not interpolate user-supplied option text into shell commands — the dispatch must pass content as structured data (file path + section reference), not as a command string.
- [ ] **S2.2** — No secrets or credentials in the Step 4a/4b instruction text
- [ ] **S2.3** — ADR content passed to dispatched agents is read from a local file, not deserialized from an external source. No untrusted deserialization vector.
- [ ] **S2.4** — N/A — no new dependencies. Logged: documentation-only change.
- [ ] **S2.5** — N/A — no new files created. Logged: documentation-only change.
- [ ] **S2.6** — Dispatched agents receive a scoped payload (file path, section, scope label). Verify the instructions define the payload boundary — agents should not receive unbounded access beyond the ADR file.

### UX Checks

- [ ] **U2.1** — Fallback behavior for unresolvable agents produces a warning message, not silent failure. Both Step 4a and Step 4b specify fallback behavior.
- [ ] **U2.2** — N/A — no user-facing commands; these are procedural instructions for an agent. Logged: no exit code check needed.
- [ ] **U2.3** — Invalid input scenarios are covered: empty dispatch value → skip; unresolvable agent → fallback with warning; all agents fail → skip the step entirely
- [ ] **U2.4** — N/A — no resources opened by the instructions themselves. Agent dispatches are managed by the `task` tool runtime. Logged: no resource cleanup check needed.
- [ ] **U2.5** — Step 4a/4b findings are incorporated into the Evaluation Checkpoint's Assessment field, which is written to the ADR file — verifiable output
- [ ] **U2.6** — The procedure table and flow diagram in create.md are updated to reflect new sub-steps — a reader can inspect the table to confirm Step 4a and Step 4b exist
- [ ] **U2.7** — A developer unfamiliar with the change can verify the stage by: (1) checking the procedure table for Step 4a/Step 4b rows, (2) checking the flow diagram for sub-step indentation under Step 4, (3) reading each sub-section's condition/skip/fallback behavior

### Test-Gap Findings

- **TG2.1** — Dev criteria for Task 2.1 verify the procedure table and flow diagram, but do not verify that the **"Conditional steps" paragraph** (line 114 of create.md) is updated to mention Step 4a and Step 4b. Currently it says "Step 4 and Step 6 are checkpoints" and "Step 3b is conditional on `tech_writer` being configured." The new sub-steps are also conditional and should be mentioned here.
- **TG2.2** — Dev criteria for Task 2.2 verify Step 4a content, including verdict mapping (Redesign → Pause, Revise → considerations, Accept → proceed). But they do not verify that the **verdict values used in the mapping match the actual output format** of the UX/DX reviewer agents. If the agents use different verdict labels (e.g., "Reject" instead of "Redesign"), the mapping instruction would be wrong. This is a documentation correctness gap — the instruction should either reference the agent's output format or define the expected verdict vocabulary.
- **TG2.3** — Dev criteria for Task 2.3 check that the TPM dispatch context includes "optional UX/DX findings" but do not specify **what happens when Step 4a runs but produces no findings** (all agents returned Accept with no findings). The instruction should distinguish "Step 4a ran, no findings" from "Step 4a was skipped" — both result in no findings but have different semantic meaning for the TPM.
- **TG2.4** — Neither Task 2.2 nor 2.3 criteria verify **backwards compatibility of the existing Step 4 content**. The plan adds sub-sections after Step 4 but before Step 5. Dev criteria confirm the new content exists but do not confirm that the existing Step 4 text (lines 200–216 of create.md) is preserved unchanged. A merge error could silently delete or modify existing checkpoint instructions.
- **TG2.5** — Dev criteria do not verify that `make check-refs` passes after Stage 2 changes. The plan defers this to Stage 3 (Task 3.1), but if Stage 2 introduces a broken cross-reference, it remains undetected until finalization.
