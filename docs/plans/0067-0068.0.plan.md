## Stage 2: S-3 Fast-Path Procedure Reference

Write the full S-3 procedure in a new `references/fast-path.md` file, then wire it into SKILL.md.

### Task 2.1: Create `references/fast-path.md` with the full S-3 procedure [medium]

**Description:** Create `src/skills/solve-adr/references/fast-path.md`. This file is the runtime reference for S-3 — follow the same structure and depth as `references/problem.md`.

The file must contain:

**Header and purpose:** Self-contained reference for S-3. Read when the user provides a list of findings from a retrospective, bug bash, or amendment.

**Trigger types (listed near top, before procedure steps):** The reference explicitly names the 3 trigger types that route to S-3: retrospective findings, bug bash findings, and amendments to existing decisions.

**Classification test (prominent, before procedure steps):**
> "Would seeing this in the ADR list give useful ambient context to someone orienting in this codebase?"
> - Yes → ADR-worthy → route to step 2
> - No → plan-only → route to step 3

**S-3 procedure steps (4 numbered steps matching ADR-0067 Decision §S-3 procedure, + Conclusion routing note):**

Intake is an entry condition, not a numbered step — the agent receives findings before the numbered procedure begins. The ADR's 4 numbered steps start at classification.

```
Entry: Receive finding list, identify source (retro / bug-bash / amendment)
   ↓
Step 1: Classify — apply ADR list test to each finding
   ↓
Step 2: Author Y-statement ADRs — for ADR-worthy findings only
   ↓
Step 3: Build plan list — collect plan-only findings with [Source: <origin>] notes
   ↓
Step 4: Delegate — invoke /implement-adr with all Ready ADRs + plan-only task list
   ↓
C. Conclusion — same conclusion sequence as S-1 (C-1 → C-2 → C-3)
```

**Step 2 detail (Y-statement ADR authoring):**
- Create the ADR file using the Makefile
- Populate standard metadata: Date, Status, Last Updated, Links
- Write provenance in the **Context section**: source type, date/session, finding text, why it earns an ADR
- Write the decision as a **Y-statement in the Decision section**: "In the context of X, facing Y, we chose Z over A to achieve B, accepting C."
- Fill the Quality Strategy checklist
- **Skip Options and Evaluation Checkpoint — absent by design.** These sections are intentionally omitted for pre-decided findings.
- **Invoke author-adr A-3 onward** (review, revise, re-review). Inform the reviewer: Options and Evaluation Checkpoint are absent by design — R-1 criterion 1 (≥2 alternatives) does not apply to S-3 ADRs.

**Step 3 detail (plan-only findings):**
- Format each item as a task with a `[Source: <finding-origin>]` note, e.g.:
  > Fix the timeout default in the retry handler [Source: Q1 2026 retrospective — "timeout default caused cascading failures in load test"]

**Step 4 detail (delegate):**
- Invoke `/implement-adr` with: all Ready ADRs from step 2 and the non-decision task list from step 3 in a single batch.

**Conclusion routing:** After step 4 completes, run C-1 → C-2 → C-3 as defined in SKILL.md. No separate conclusion for S-3.

**Test & Acceptance Criteria:**
- [ ] [implement, verify-manual] `src/skills/solve-adr/references/fast-path.md` exists
- [ ] [verify-manual] Trigger types (retro, bug bash, amendments) listed before the procedure steps
- [ ] [verify-manual] Classification test appears prominently before the procedure steps
- [ ] [verify-manual] Entry condition (receive findings, identify source) present as a pre-step before the 4 numbered steps
- [ ] [verify-manual] All 4 numbered S-3 procedure steps present with step IDs (Classify, Author, Build, Delegate), plus Conclusion routing note
- [ ] [verify-manual] Step 2 explicitly states: provenance written in Context section; Y-statement in Decision section; Options and Evaluation Checkpoint skipped with "absent by design" note; A-3 onward invocation with R-1 N/A note
- [ ] [verify-manual] Step 3 includes the `[Source: <finding-origin>]` format with a concrete example
- [ ] [verify-manual] Step 4 explicitly states `/implement-adr` is invoked with Ready ADRs + non-decision task list
- [ ] [verify-manual] Conclusion routing note directs to C-1 → C-2 → C-3
- [ ] [verify-manual] File follows the writing style: technical, simple, no double negatives, concise

**Description:** In `src/skills/solve-adr/SKILL.md`, update the Conclusion section:

   ```
   C-1 QA Triage → C-2 Code Review (optional) → C-3 Report → C-4 Retrospective (optional)
   ```

2. Add C-4 row to the conclusion step table:
   | C-4 | Retrospective (optional) | Run structured retro, classify findings, write preference updates to `.adr/preferences.toml` |

3. Add a `### C-4: Retrospective` section after `### C-3: Report`:

   ```markdown
   ### C-4: Retrospective

   After C-3, run a structured retrospective on the completed solve run. C-4 is optional — skip when `[solve.retro] enabled = false` in `.adr/preferences.toml`. Skip also when `skip_when_no_findings = true` and the retrospective produces no actionable findings.

   **Entry condition:** C-3 has completed. C-4 does not affect C-3 artifacts.

   **Retrospective questions:**
   - What slowed things down?
   - What quality steps added the most value?
   - Were any steps unnecessary for this project's context?
   - What would you change for the next solve?

   **Classify each finding:**
   - **Pipeline preference** — expressible as an existing key in the preference schema. Propose a config update. If no matching key exists, the finding is note-only — do not invent new keys; new keys require a future ADR.
   - **Note-only** — useful context but not expressible as a current preference key.

   **Apply updates:**
   - Autonomous mode: write proposed changes to `.adr/preferences.toml` and log what changed.
   - Guided mode: present proposals, apply on confirmation.

   **Write retro record:** Save findings to `.adr/var/retro-<slug>.md`. Include: run summary, findings, preference changes applied. Create `.adr/var/` if it doesn't exist.
   ```
