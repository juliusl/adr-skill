# QA Plan: Accept git remote names and auto-detect adapter type

**Generated:** 2026-04-08

## Stage 1: Detection Logic (Rust)

### Security Checks

- [ ] **S1.1** — `detect_adapter()` passes the `remote` argument to `std::process::Command::args()` (array form), not through a shell. Verify no path uses `Command::new("sh").arg("-c")` or string interpolation that would allow shell injection via crafted remote names (e.g., `; rm -rf /` or `$(malicious)`)
- [ ] **S1.2** — `extract_host()` handles adversarial URLs without panicking: empty string, missing `://`, URL with embedded newlines, URL with null bytes, extremely long strings (>4KB)
- [ ] **S1.3** — `detect_adapter_from_url()` pattern matching uses `contains()` on the extracted host only, not the full URL. A URL like `https://evil.com/github.com/exploit` must not match `github.com` — verify `extract_host` returns `evil.com`, not a string containing `github.com`
- [ ] **S1.4** — No new dependencies added (pure stdlib subprocess call). If any new crate is added, it must be pinned to a specific version
- [ ] **S1.5** — The `remote` argument is not used in file path construction before `detect_adapter()` resolves it to an adapter type. Verify that `cmd_new` uses the returned adapter type (not the raw user input) in the filename format string
- [ ] **S1.6** — `VALID_REMOTES` constant and `validate_remote()` function are fully removed — no dead code path that could be called to bypass detection

### UX Checks

- [ ] **U1.1** — When `git` binary is not found (not installed), the error message says "failed to run git" (not a raw `io::Error` or panic)
- [ ] **U1.2** — When the remote name doesn't exist, the error message includes the remote name and suggests `git remote -v`
- [ ] **U1.3** — When the URL is unrecognized, the error message includes the actual URL and lists supported hosts
- [ ] **U1.4** — All five subcommands (`new`, `rename`, `status`, `lifecycle`, `export`) exit with code 1 on detection failure — verify through `run_author`'s error handler
- [ ] **U1.5** — The clap `--help` text for each subcommand's `remote` field says "Git remote name" (not "Remote identifier (gh, ado, gitea, local)")
- [ ] **U1.6** — A user passing the old-style adapter label `gh` as a remote name gets a clear error ("git remote 'gh' not found") rather than a confusing/silent failure
- [ ] **U1.7** — `detect_adapter("local")` returns immediately without attempting to run `git` — verify no subprocess call on the `local` path

### Test-Gap Findings

- [ ] **TG1.1** — The plan tests `detect_adapter_from_url` and `extract_host` in isolation but never tests that all five call sites (`cmd_new`, `cmd_rename`, `cmd_status`, `cmd_lifecycle`, `cmd_export`) actually use the **returned adapter type** for filename construction instead of the raw `remote` argument. A bug where one call site still uses `remote` instead of the adapter type would pass all unit tests.
- [ ] **TG1.2** — No acceptance criterion tests the behavior when `git` is not installed (`Command::new("git")` returns `Err`). The plan tests "remote not found" but not "git not found" — these are different failure modes with different error messages.
- [ ] **TG1.3** — No acceptance criterion covers the `ssh://git@github.com/org/repo` URL format (non-shorthand SSH). The plan's implementation notes show `extract_host` handles `git@` prefix and `://` scheme, but the SSH URL `ssh://git@github.com:22/org/repo` (with port and `ssh://` scheme) would hit the `://` branch, not the `git@` branch. Verify both branches produce correct results for this URL format.
- [ ] **TG1.4** — No acceptance criterion tests `extract_host` with a URL containing a port number in HTTPS format (e.g., `https://github.com:443/org/repo`). The implementation strips ports via `host.split(':').next()` but this is untested.
- [ ] **TG1.5** — No acceptance criterion verifies that the `generate_template()` call in `cmd_new` receives the adapter type. The template encodes `work_item = "{remote}#{id}"` — if `remote` is the git remote name (`origin`) instead of the adapter type (`gh`), the work item reference would be `origin#42` instead of `gh#42`.

## Stage 2: Detection Logic (Shell)

### Security Checks

- [ ] **S2.1** — `detect_adapter()` passes the remote name to `git remote get-url` inside double quotes (`"$remote"`). Verify that a remote name containing spaces, semicolons, pipes, backticks, or `$()` does not cause command injection. Shell quoting is the defense — confirm no unquoted expansion exists in the call chain
- [ ] **S2.2** — `extract_host()` uses `echo "$url"` piped to `sed`. If the URL contains characters meaningful to `echo` (e.g., `-n`, `-e` on some shells) or `sed` (e.g., newlines, null bytes), verify no unexpected behavior. Consider whether `printf '%s\n' "$url"` would be safer than `echo "$url"`
- [ ] **S2.3** — `detect_adapter_from_url()` case pattern `*github.com*` matches against the extracted host, not the full URL. Verify `extract_host` strips path components so a URL like `https://evil.com/github.com/exploit` returns `evil.com`, not a string matching `*github.com*`
- [ ] **S2.4** — The `ADR_TEST_ADAPTER` override variable is only checked in `detect_adapter()`. Verify it cannot be set by untrusted input to bypass detection in production (it's an env var, so this is acceptable — but document that it's a test seam, not a user-facing feature)
- [ ] **S2.5** — All call sites capture the adapter type via `adapter=$(detect_adapter "$remote") || exit 1` and use `$adapter` for filename construction. No call site continues to use `$remote` in file paths or metadata after detection
- [ ] **S2.6** — Error messages in `detect_adapter()` and `detect_adapter_from_url()` include user-supplied strings (`$remote`, `$url`). Verify these go to stderr only and are not interpolated into downstream commands or file operations

### UX Checks

- [ ] **U2.1** — `detect_adapter()` outputs the error to stderr and returns 1 (not `exit 1`) when the remote doesn't exist, allowing the caller to handle the error. Verify the function uses `return 1` not `exit 1`
- [ ] **U2.2** — Error messages match the Rust implementation's phrasing for consistency: "git remote '...' not found" and "Could not detect adapter type for URL '...'"
- [ ] **U2.3** — When `git` is not installed, the `2>/dev/null` in `git remote get-url "$remote" 2>/dev/null` suppresses git's error but the function still produces its own "remote not found" message. Verify the user sees a helpful message, not silent failure
- [ ] **U2.4** — Existing subcommand usage messages in the help text (`case` dispatch) are updated to reflect that `<remote>` is now a git remote name, not an adapter label
- [ ] **U2.5** — The `validate_remote` function is fully removed from the script — no dead code that could confuse future maintainers
- [ ] **U2.6** — `detect_adapter_from_url()` echoes the adapter type to stdout on success — verify no debug output or extra whitespace contaminates the captured value
- [ ] **U2.7** — A user who reads the script's help text (`wi-nygard-agent-format.sh` without args) understands that `<remote>` means a git remote name

### Test-Gap Findings

- [ ] **TG2.1** — The plan adds `ADR_TEST_ADAPTER` as a test seam but does not test the actual `git remote get-url` code path. All shell tests bypass detection entirely. The real subprocess call is never exercised in automated tests — a typo in the `git remote get-url` invocation (e.g., wrong flag) would not be caught.
- [ ] **TG2.2** — The plan says "URL→adapter mapping results match between Rust and Shell for all tested patterns" (Task 2.2 criteria) but provides no mechanism to enforce this. The Rust and Shell test suites are independent — a change to one without the other would create a divergence with no test failure.
- [ ] **TG2.3** — The `extract_host` shell implementation uses `echo "$url" | sed ...` which may behave differently on macOS (`sed`) vs Linux (`sed`) for edge-case patterns. The plan does not specify which `sed` variant is targeted or test on multiple platforms.
- [ ] **TG2.4** — No test verifies that `detect_adapter` correctly handles a remote name that happens to look like an adapter label (e.g., a git remote literally named `gh`). If someone has `git remote add gh https://github.com/org/repo`, calling `detect_adapter "gh"` should resolve via `git remote get-url` and return `"gh"` — but this case is not tested.
- [ ] **TG2.5** — The plan's shell implementation note shows `return 1` in `detect_adapter_from_url` for unknown hosts, but the existing `validate_remote` uses `exit 1`. If any call site uses a subshell pattern that swallows `return 1`, the error will be silently ignored. No test specifically verifies the exit code propagation chain.

## Stage 3: Integration and Tooling

### Security Checks

- [ ] **S3.1** — Integration test creates a temp git repo. Verify the test uses a unique temp directory and cleans it up on both success and failure — no leftover state that could interfere with other tests or leak file paths
- [ ] **S3.2** — The integration test runs `adr-db author new origin 99 "Test ADR" <temp-dir>/docs/adr`. Verify the temp directory path cannot escape its sandbox (no symlink following, no `..` in constructed paths)
- [ ] **S3.3** — The `--help` text updates do not expose internal implementation details (function names, file paths) that could aid targeted attacks

### UX Checks

- [ ] **U3.1** — `adr-db author new --help` shows the updated remote description mentioning "Git remote name (e.g., origin) or 'local'" — not the old "Remote identifier (gh, ado, gitea, local)"
- [ ] **U3.2** — `adr-db author --help` lists all subcommands with descriptions that reflect the new behavior
- [ ] **U3.3** — The integration test for a nonexistent remote produces an error message that includes the remote name and suggests `git remote -v`
- [ ] **U3.4** — The integration test for `local` keyword works outside a git repo (no git required)

### Test-Gap Findings

- [ ] **TG3.1** — The integration test verifies the output filename (`gh-99-test-adr.toml`) and TOML content (`work_item = "gh#99"`) but does not verify that the TOML's `title` field uses the adapter type prefix (e.g., `gh-99. Test ADR`), not the remote name (`origin-99. Test ADR`). This is a data integrity gap — the filename could be correct while the internal title is wrong.
- [ ] **TG3.2** — No integration test verifies behavior when the command is run outside a git repository entirely (not just "no remote configured" but "not a git repo at all"). The error path is different: `git remote get-url` will fail with "not a git repository" vs "remote not found."
- [ ] **TG3.3** — The plan does not include an integration test for the shell script (`wi-nygard-agent-format.sh`) end-to-end with a real git repo. Only the Rust binary gets an integration test. The shell script's `detect_adapter` function is tested only with the `ADR_TEST_ADAPTER` bypass.

## Stage 4: Finalize

### Security Checks

- [ ] **S4.1** — The ADR status update modifies only the `Status:` line in the ADR file. No other file content is changed
- [ ] **S4.2** — No new files are created or deleted in this stage

### UX Checks

- [ ] **U4.1** — After the status update, `make test` passes — confirming no regressions from the full changeset
- [ ] **U4.2** — The ADR file's status reads exactly `Accepted` (not `accepted`, `ACCEPTED`, or any variant)

### Test-Gap Findings

- [ ] **TG4.1** — No mechanical verification that only the `Status:` line changed in ADR-0062. The acceptance criterion says "No other ADR content is modified" but relies on manual review. A `git diff` check scoped to the ADR file would provide automated verification.
