- [x] **S1.2** — `extract_host()` handles adversarial URLs without panicking: empty string returns `""`, missing `://` returns the input lowercased, embedded newlines are trimmed by `String::from_utf8_lossy().trim()` before reaching `extract_host`, Rust strings cannot contain null bytes (UTF-8 guarantee), long strings work with `contains`/`split`/`strip_prefix`. PASS.
- [x] **S1.3** — `detect_adapter_from_url()` pattern matching uses `contains()` on the extracted host only. Traced `extract_host("https://evil.com/github.com/exploit")`: `split("://").nth(1)` → `"evil.com/github.com/exploit"`, `split('/').next()` → `"evil.com"`. Then `"evil.com".contains("github.com")` → false. PASS.
- [x] **S1.4** — Cargo.toml checked: dependencies are `adr-db-lib` (path), `clap`, `diesel`, `serde_json`. No new crates added for detection — uses only `std::process::Command`. PASS.

## Stage 2: Detection Logic (Shell)

### Security Checks

- [x] **S2.1** — `detect_adapter()` at line 75 uses `git remote get-url "$remote"` with double-quoted `$remote`. A remote name containing `; rm -rf /` or `$(malicious)` is passed as a single argument to `git`, not interpreted by the shell. PASS.
- [x] **S2.2** — Implementation uses `printf '%s' "$url"` (lines 33, 34, 38), not `echo "$url"`. This avoids misinterpretation of URLs starting with `-n`, `-e`, etc. PASS.
- [x] **S2.3** — `extract_host` tested with `https://evil.com/github.com/exploit`: `sed 's|.*://||; s|.*@||; s|/.*||; s|:.*||'` strips path components, yielding `evil.com`. The `case *github.com*` pattern then fails to match. Manually verified. PASS.
- [x] **S2.4** — `ADR_TEST_ADAPTER` is checked only in `detect_adapter()` (line 64). It is an env var, documented in the function comment as "Testing seam." It cannot be set by untrusted input in production — only by the process environment. PASS.
- [x] **S2.5** — All four call sites (`cmd_new` line 202, `cmd_rename` line 396, `cmd_status` line 463, `cmd_lifecycle` line 547) use `adapter=$(detect_adapter "$remote") || exit 1` and then use `$adapter` for file paths and metadata. **Exception:** `log_lifecycle` on line 620 receives `$remote` (raw user input) instead of `$adapter` — see Finding F1 below. Not a security issue (jq-serialized to JSONL) but a data integrity concern. PASS with finding.
- [x] **S2.6** — Error messages on lines 76 and 52 include `$remote` and `$url` respectively, both sent to stderr (`>&2`). Neither is interpolated into downstream commands or file operations. PASS.

### UX Checks

- [x] **U2.1** — `detect_adapter()` uses `return 1` (line 77), not `exit 1`. Callers handle the error via `|| exit 1`. `detect_adapter_from_url()` also uses `return 1` (line 53). PASS.
- [x] **U2.2** — Shell error messages match Rust: `"git remote '...' not found"` (shell line 76, Rust line 131) and `"Could not detect adapter type for URL '...'"` (shell line 51, Rust line 105). PASS.
- [x] **U2.3** — When `git` is not installed, `git remote get-url` fails, `2>/dev/null` suppresses git's error, and the function produces `"git remote '<name>' not found"`. This is slightly misleading (the issue is git missing, not the remote) but provides an actionable message. Matches Rust behavior for the "remote not found" case. PASS (the "git not found" case produces a different, more specific error in Rust via `map_err`; shell conflates the two — accepted as a known limitation).
- [x] **U2.4** — The dispatch help text (lines 639-644) uses `<remote>` in subcommand descriptions. The text does not explicitly say "Git remote name" but no longer references adapter labels. The parameter name `<remote>` is consistent with git terminology. PASS (minor — could be improved with a parenthetical like `<remote>  Git remote name or 'local'`).
- [x] **U2.5** — `grep -n 'validate_remote' wi-nygard-agent-format.sh` returns zero matches. Function fully removed. PASS.
- [x] **U2.6** — `detect_adapter_from_url()` uses `printf 'gh'` / `printf 'ado'` (lines 48-49) — no trailing newline, no debug output. PASS.
- [x] **U2.7** — The help text says `<remote>` for subcommands that take it. Combined with the function comment "Detect adapter type from a git remote name or 'local' keyword" (line 57), users reading the script can understand the intent. PASS.

### Test-Gap Findings

- [x] **TG2.1** — Confirmed: all shell tests use `ADR_TEST_ADAPTER` to bypass real `git remote get-url`. The actual subprocess call is never exercised in CI. Acknowledged — adding a real git repo shell integration test is deferred (see Recommendation 7). LOW RISK: the Rust integration tests do exercise the real detection path end-to-end.
- [x] **TG2.2** — Confirmed: no shared test data file exists. The `wi-detect-adapter.sh` test covers 6 URL patterns; Rust tests cover the same 6 plus more. Manual review confirms parity. Acknowledged — see Recommendation 6.
- [x] **TG2.3** — Implementation uses `printf '%s'` + `sed` + `tr`. These are POSIX-standard. The `sed` patterns used (`s/^git@//`, `s|.*://||`, etc.) are basic and portable. Acknowledged as a platform gap — CI runs on Linux only. LOW RISK.

### QA Finding: F1 — Shell `log_lifecycle` passes raw remote instead of adapter type

**Stage:** 2
**Classification:** Bug (data integrity)
**Severity:** Low
**Location:** `wi-nygard-agent-format.sh` line 620

`log_lifecycle "$remote" "$id" ...` passes `$remote` (the raw user input, e.g., `"origin"`) instead of `$adapter` (the resolved adapter type, e.g., `"gh"`). The JSONL log will contain `"remote": "origin"` instead of `"remote": "gh"`, which is inconsistent with the normalized data model where `remote` means adapter type.

**Fix:** Change line 620 from `log_lifecycle "$remote"` to `log_lifecycle "$adapter"`.

**Not a security issue:** The value is serialized via `jq` into a JSONL file (no injection risk). This is a data quality/consistency concern only.
