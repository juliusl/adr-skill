### Test-Gap Findings

## Stage 3: User-Mode ADR Storage (ADR-0047)

### Security Checks

- [ ] **S3.1** — No user-supplied strings interpolated into SQL, shell commands, or file paths — **FAIL**: `ADR_USERNAME` is directly interpolated into file paths without sanitization. In `nygard-agent-format.sh:135-139`, `resolve_username()` returns raw `ADR_USERNAME` → used as path prefix in `$dir/${prefix}${number}-${slug}.md`. In `new.sh:95-99`, `local_username` is used in glob patterns and sed regex. **Confirmed exploitation:** `ADR_USERNAME='../etc'` creates file at `.adr/usr/docs/etc-0001-evil-title.md` — outside the intended `.adr/usr/docs/adr/` directory. See QF3.1.
- [x] **S3.2** — No secrets, credentials, or API keys in committed files — Pass: reviewed all changed files, no secrets
- [x] **S3.3** — No deserialization of untrusted input without validation — N/A: no deserialization
- [x] **S3.4** — Dependencies pinned to specific versions (no wildcards) — N/A: no dependencies added
- [x] **S3.5** — File permissions on created artifacts are not overly permissive — Pass: `.adr/.gitignore` is 644; created dirs are 755; ADR files are 644
- [ ] **S3.6** — Any new external input surface has validation at the boundary — **FAIL**: `ADR_USERNAME` is a new external input surface with zero validation. No character whitelist, no rejection of path separators, no length check. See QF3.1.

### UX Checks

- [x] **U3.1** — Every error path produces a human-readable message on stderr — Pass: `cmd_list` has "ERROR: ADR directory not found: $dir" on stderr (nygard-agent-format.sh:239); `new.sh` has clear error messages on stderr for missing format/title/remote/id
- [x] **U3.2** — Every user-facing command exits with code 0 on success, non-zero on failure — Pass: `cmd_list` exits 1 on missing directory; `new.sh` exits 1 on all error paths; success paths exit 0
- [ ] **U3.3** — Invalid input is rejected with a helpful message, not a crash — **FAIL**: `ADR_USERNAME` containing path traversal characters (`../`) is silently accepted. The script doesn't crash, but it writes files outside the intended directory without warning. Invalid usernames should be rejected with a message like "ERROR: ADR_USERNAME contains invalid characters". See QF3.1.
- [x] **U3.4** — Resources (file handles, database connections) are cleaned up on error paths — N/A: shell scripts, no persistent resources

- [x] **U3.7** — A user who did not write the code can verify the stage's output — Pass: SKILL.md User Mode section (lines 76-101) documents naming convention, directory, activation, promotion; three passing tests demonstrate the workflow

### QA Execution Findings

- **QF3.1** — **CRITICAL: ADR_USERNAME path traversal (TG3.1 confirmed).** Tested `ADR_USERNAME='../etc'` with `ADR_SCOPE=user`. The script created `.adr/usr/docs/etc-0001-evil-title.md` instead of rejecting the input. The path traversal escapes `.adr/usr/docs/adr/` and writes to `.adr/usr/docs/etc-*`. Root cause: neither `resolve_username()` in `nygard-agent-format.sh` nor `local_username` assignment in `new.sh` validates `ADR_USERNAME`. **Two injection vectors**: (1) path traversal in filenames, (2) regex injection via `sed -E "s/^${local_username}-..."` in `new.sh:99`. **Remediation:** Add validation to reject `ADR_USERNAME` values containing characters outside `[a-zA-Z0-9_-]`. Fix in both `resolve_username()` (nygard-agent-format.sh:28-30) and `local_username` assignment (new.sh:95). Add a test case with a malicious username.
- **QF3.2** — **TG3.2 first-run directory creation: Pass.** The `user-create-record.sh` test creates `.adr/` but NOT `.adr/usr/docs/adr/`. The `mkdir -p` in `cmd_new` (nygard-agent-format.sh:136) and `new.sh:94` fires correctly during the test. Verified by test passing.
- **QF3.3** — **TG3.3 empty directory starts at 0001: Pass.** The `user-create-record.expected` shows the first file is `testuser-0001-explore-caching.md`. The second file increments to `testuser-0002-evaluate-framework.md`. Empty-directory and increment cases both covered.
- **QF3.4** — **TG3.4 scope inspection: Partially addressed.** SKILL.md documents `[author].scope` and `ADR_SCOPE` (lines 82, 87-89, 95), so a user can check their config. However, there's no runtime "which scope am I in?" command output. Recommendation: have `list` indicate scope in output header, or document `echo $ADR_SCOPE` as the inspection command.
- **QF3.5** — **All three user-mode tests pass:** `user-create-record` ✅, `user-list-records` ✅, `user-scope-isolation` ✅. Note: these tests were not reached by `make test` due to a pre-existing `rename-record` test failure (Linux `sed -i ''` portability issue). Tests were validated individually via `make test-user-*`.

### Test-Gap Findings

- **TG3.1** — **ADR_USERNAME path injection.** `ADR_USERNAME` is a user-controlled environment variable interpolated into file paths: `$dir/${username}-${number}-${slug}.md`. If it contains `../`, `/`, spaces, or shell metacharacters, the script could write files outside `.adr/usr/docs/adr/` or crash. The dev criteria don't include a validation test for malicious or malformed usernames. The script must sanitize `ADR_USERNAME` at the boundary — reject values containing `/`, `..`, or characters outside `[a-zA-Z0-9_-]`.

- **TG3.2** — **No test for first-run directory creation.** Task 3.2 mentions `mkdir -p` for the user directory, and Task 3.3's tests cover create, list, and scope isolation. But no acceptance criterion verifies that the `mkdir -p` actually fires correctly when `.adr/usr/docs/adr/` doesn't exist yet. If the test setup creates the directory before the test runs, the `mkdir -p` path is untested.

- **TG3.3** — **NNNN edge cases in user-scoped resolution.** Task 3.2 modifies NNNN resolution to scan for `${username}-*.md` files. The dev criteria verify basic create and list, but don't test: (a) empty directory → starts at 0001, (b) gaps in numbering → still increments from max, (c) files from other users present in the same directory (shouldn't happen, but defensive check). At minimum, the empty-directory case should be covered.

- **TG3.4** — **Observability gap: user-mode state inspection.** After creating user-mode ADRs, a user needs to verify the files landed in `.adr/usr/docs/adr/`. The `list` command in user mode (Task 3.2) provides this. But there's no documented way for a user to check which mode they're in — if `ADR_SCOPE` is set persistently, a user might not realize they're creating user-mode ADRs. The SKILL.md should document how to inspect current scope.
