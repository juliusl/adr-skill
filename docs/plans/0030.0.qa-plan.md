## Stage 1: QA Reference Document

### Test-Gap Findings

1. **Regeneration-on-revision is documented but not enforced.** Task 1.1
   acceptance criterion says qa-planning.md must document the
   regeneration-on-revision requirement. But no task in the plan creates
   **tooling or workflow steps** that actually trigger regeneration when a plan
   revision is created. The document will say "regenerate the QA plan on
   revision" but nothing in SKILL.md's Step 7 (revision workflow) will prompt
   the agent to do it. A dev could revise a plan, auto-increment the revision
   number, and the old QA plan would silently apply to the new plan.

   **Classification:** Quality concern — the stale QA plan is a silent failure
   mode that undermines the QA system's integrity.

   **Recommendation:** Add acceptance criteria to Task 2.1 or Task 2.2
   requiring that SKILL.md's plan-revision workflow (Step 7, item 5) includes a
   note to regenerate the QA plan on revision. Alternatively, add a small task
   to Stage 2 that patches Step 7.

2. **QA plan template (Task 1.2) has no Recommendations section.** The ADR's
   QA finding eligibility section describes quality-concern vs. preference
   classification. The template should have a `### Recommendations` section at
   the end for test-gap findings that recommend new work. The acceptance
   criteria only check for header, per-stage sections, test-gap findings, and
   checkbox format — not for the recommendations/eligibility section.

   **Classification:** Quality concern — without it, the QA planner has no
   structured place to put actionable findings, and the main executor won't
   know what to schedule.

   **Recommendation:** Add criterion to Task 1.2: "Template has a
   Recommendations section with eligibility classification (quality concern vs.
   preference)."

3. **No acceptance criterion verifies cross-referencing between qa-planning.md
   prompt templates and the actual SKILL.md step.** Task 1.1 says "contains
   prompt templates for QA planner and executor agents." Task 2.1 says
   SKILL.md references qa-planning.md. But nothing verifies the prompt
   template in qa-planning.md and the Step 4b instructions in SKILL.md are
   **consistent** — e.g., that the file naming convention
   `<range>.<revision>.qa-plan.md` matches between the two documents.

   **Classification:** Quality concern — inconsistency between the reference
   and the skill instructions would cause agent confusion at runtime.

4. **Asset index (Task 1.3) acceptance criteria say "lists the qa-planning
   reference under References" but qa-planning.md is a reference file, not an
   asset.** The existing index.md already has a References section with the
   other reference files. This is fine structurally, but the criterion is
   ambiguous — it says "lists the qa-planning reference under References"
   without specifying the expected format (table row matching the existing
   pattern). A dev could add a bullet point instead of a table row and still
   pass the criterion.

   **Classification:** Preference — the criterion is technically satisfiable
   either way, and the index format is not user-facing.

## Stage 2: SKILL.md Integration

### Test-Gap Findings

1. **Line count criterion is ≤ 520, but AGENTS.md says "under 500 lines."**
   The plan's acceptance criterion for Task 2.1 says "SKILL.md total line
   count is ≤ 520 lines." The AGENTS.md contribution guide says "Verify
   SKILL.md stays under 500 lines (spec recommendation for progressive
   disclosure)." The agentskills.io spec says "recommended < 500 lines." The
   plan is allowing a 20-line overshoot of the documented guideline. All dev
   acceptance criteria could pass at 519 lines while violating the project's
   own documented standard.

   **Classification:** Quality concern — the acceptance criterion contradicts
   the project's documented constraint. The implementor may target 520 and
   produce a SKILL.md that fails the project's own `wc -l` check documented
   in AGENTS.md. However, the spec says "recommended," not "required," so
   this is a soft boundary. Recommend tightening the criterion to ≤ 510 as a
   compromise, or updating AGENTS.md to say "under ~520 lines" for
   consistency.

2. **Backwards compatibility criterion is weak.** Task 2.1 says "No existing
   step content is modified (backwards compatible)." This checks that
   existing *text* is preserved, but backwards compatibility for this feature
   means more than text preservation: it means that **a plan generated before
   ADR-0030 executes identically** with the new SKILL.md. The stage boundary
   QA hook (Task 2.2) addresses this with "no QA plan → no-op," but Task 2.1
   does not verify that Step 4b gracefully handles the case where it's
   invoked during execution of a *pre-existing* plan that was generated
   before QA planning existed.

   **Classification:** Quality concern — a pre-existing plan being executed
   after the SKILL.md update could encounter Step 4b instructions and
   attempt to generate a QA plan for a plan that already passed review and is
   mid-execution. The instructions should clarify that Step 4b applies only
   during plan generation, not during execution of existing plans.

3. **The planning workflow diagram update (Task 2.2) has no standalone
   acceptance criterion for the diagram format.** The criterion says "the
   planning workflow diagram includes the QA plan step" — but the existing
   diagram on line 57 uses a specific ASCII format:
   `Read ADR → Generate plan → Plan review (sub-agent) → [Iterate] → Execute plan`.
   The plan's implementation notes show a different format:
   `Read ADR → Generate plan → Plan review → QA plan (4b) → Execute plan`.
   These differ in whether `(sub-agent)` and `[Iterate]` are kept. A dev
   could update the diagram in a way that loses existing information
   (removing `[Iterate]` or `(sub-agent)` labels) and still satisfy the
   criterion.

   **Classification:** Quality concern — the diagram is user-facing
   documentation and losing the `[Iterate]` notation would obscure the
   plan-review iteration cycle.

4. **No criterion verifies that the Deep References section is updated.** The
   SKILL.md has a "Deep References" section (line 466) listing all reference
   files. Adding qa-planning.md as a reference without adding it to this list
   would leave a discoverability gap — an agent reading the Deep References
   section would not know qa-planning.md exists.

   **Classification:** Quality concern — agents use the Deep References
   section to discover available references. An unlisted reference is
   effectively invisible to agents that don't also check the asset index.

   **Recommendation:** Add acceptance criterion to Task 2.1 or Task 2.2:
   "The Deep References section in SKILL.md lists qa-planning.md."

5. **`make test` criterion (Task 2.2) validates scripts, not SKILL.md
   content.** The criterion "`make test` still passes" checks that the
   script test suites (adr-tools + madr-tools) are unbroken. These tests
   validate shell scripts, not the SKILL.md document. The relevant
   validation for SKILL.md changes is `make validate-implement` and
   `make check-refs`, neither of which appear in any acceptance criterion.

   **Classification:** Quality concern — `make test` passing gives false
   confidence for this stage. The actual validators (`validate-implement`,
   `check-refs`) should be run.

   **Recommendation:** Add criteria: "`make check-refs` passes" and
   "`make validate-implement` passes" (or note that validator installation
   is optional).

## Stage 3: Finalize

### Test-Gap Findings

1. **"No existing plan files are modified" is unverifiable as stated.** Task
   3.1 criterion says "No existing plan files are modified." The intent is to
   verify backwards compatibility — that the implementation didn't
   accidentally edit someone else's plan. But this is a negative assertion
   over an open set. The dev would need to `git diff` all files in
   `docs/plans/` to verify. More importantly, the *only* plan file in the
   repo at this point is `0030.0.plan.md` itself, which *was* modified (to
   check off acceptance criteria per the Task Execution Protocol). So the
   criterion is technically self-contradictory.

   **Classification:** Quality concern — the criterion as written is either
   unverifiable or contradicted by the execution protocol. Recommend
   rewording to: "No plan files other than `0030.0.plan.md` and its QA plan
   are modified."

2. **ADR status transition validation is incomplete.** Task 3.2 says "ADR-0030
   status reads `Accepted`" but the ADR's current status is `Planned`.
   ADR-0003 (referenced in the task) presumably defines valid status
   transitions. The plan doesn't verify that `Planned → Accepted` is a valid
   transition per ADR-0003. If the valid transition is
   `Planned → Implemented → Accepted`, the task would produce an invalid
   status transition.

   **Classification:** Preference — this is an edge case in the ADR
   lifecycle management, and the implementing agent will read ADR-0003 to
   determine the correct transition. No user-facing breakage.

### Meta-Note

This is the first QA plan generated by the process ADR-0030 describes. The
irony is operational: this plan is validating the specification for the very
system that would normally generate it. Two observations from this exercise:

1. **The most significant finding is #1 (regeneration-on-revision).** The
   ADR explicitly calls out staleness as a negative consequence and says "the
   QA plan must be regenerated on plan revision." But the dev plan only
   documents this requirement in qa-planning.md — it doesn't wire it into the
   SKILL.md revision workflow. This is exactly the kind of gap that test-gap
   analysis is designed to find: all dev criteria pass, but the system has a
   silent failure mode.

2. **Most security and UX checks are N/A for documentation-only stages.**
   This is expected — the checklists are designed for implementation stages
   that produce code. For ADR-0030's implementation (which produces only
   markdown), the checklists correctly produce mostly PASS results. The
   value of the checklists will be more apparent in code-producing plans.
