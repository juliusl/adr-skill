## Stage 1: Schema & Model Changes

### Test-Gap Findings

1. **`cargo build` does not compile tests — existing test breakage is deferred
   silently to Stage 3.** Task 1.2 adds `source_plan: String` to
   `NewTaskSummary`. The acceptance criterion is "`cargo build` succeeds." But
   `cargo build` only compiles the binary, not `#[cfg(test)]` code. Two
   existing test sites construct `NewTaskSummary` with all fields listed
   explicitly:
   - `models.rs:86` — `test_round_trip_insert_query`
   - `view.rs:145` — `insert_sample` helper (used by 7 view tests)

   After adding the `source_plan` field, these will fail to compile — but only
   under `cargo test`, not `cargo build`. The developer would complete Task 1.2
   believing everything is green, then hit compilation errors in Stage 3. This
   creates unnecessary rework.

   **Classification:** Quality concern — the acceptance criterion gives false
   confidence. Either replace "`cargo build` succeeds" with "`cargo test
   --no-run` succeeds" (compiles tests without running them), or add a note
   that existing test code in `models.rs` and `view.rs` must be updated as part
   of Task 1.2.

2. **Diesel struct field ordering must match the database column order.** The
   plan says to "update `schema.rs` to include the new column" but doesn't
   specify using `diesel migration run` to auto-generate it. `ALTER TABLE ADD
   COLUMN` in SQLite appends the column after `ingested_at`, so the actual DB
   column order will be: `id, task_id, status, cost, commit_sha, description,
   ingested_at, source_plan`. If `schema.rs` is edited manually and
   `source_plan` is placed before `ingested_at`, the `Queryable` derive will
   map columns to the wrong struct fields. The existing `as_select()` pattern
   (via `Selectable`) uses the schema definition to generate explicit column
   lists, so schema.rs correctness is critical.

   **Classification:** Quality concern — a misordered column in schema.rs would
   cause silent data corruption (fields mapped to wrong values). Recommend
   adding an acceptance criterion: "Column order in `schema.rs` matches the
   actual DB column order (`source_plan` appears after `ingested_at`)."

3. **`down.sql` content is unspecified.** The plan says "a no-op down migration
   is acceptable" but doesn't specify what the file should contain. An empty
   file is fine for Diesel, but a SQL comment (e.g.,
   `-- SQLite does not support DROP COLUMN; no-op`) is better for future
   readers. This is minor.

   **Classification:** Preference — any valid content works; readability is
   a polish concern.

## Stage 2: Producer & Consumer Updates

### Test-Gap Findings

1. **No test exercises awk basename extraction with a directory-prefixed path.**
   The `extract-basic.sh` test creates a local file `plan.md` and runs
   `awk -f extract-summary.awk plan.md`. The FILENAME variable will be
   `plan.md` — a bare name with no directory components. The plan's basename
   extraction code (`split(FILENAME, path_parts, "/")`) is untested for the
   real-world invocation pattern `awk -f extract-summary.awk
   docs/plans/0029.0.plan.md` where path stripping is actually needed.

   If the `split`/basename logic has a bug (e.g., off-by-one, wrong separator
   on the test platform), all dev criteria pass but production use produces
   wrong `source_plan` values like `docs/plans/0029.0.plan.md` or empty string.

   **Classification:** Quality concern — the existing test doesn't exercise the
   code path that justifies the basename extraction. Recommend adding a test
   case that invokes awk with a path-prefixed filename (e.g.,
   `awk -f extract-summary.awk subdir/plan.md`) and verifies the output
   contains `"source_plan":"plan.md"` (not `"source_plan":"subdir/plan.md"`).

2. **awk does not escape `source_plan` for JSON output.** The existing awk
   script escapes double quotes in `description` (`gsub(/"/, "\\\"",
   description)`) but applies no escaping to `task_id`, `status`, `cost`, or
   `commit`. The plan's implementation adds `source_plan` derived from
   FILENAME, which is a filesystem path — it could theoretically contain
   backslashes (Windows paths), double quotes, or other JSON-special characters.
   The awk `printf` emits it directly into a JSON string without escaping.

   For this project's plan filenames (`0029.0.plan.md`), this is a non-issue.
   But it is a pre-existing pattern weakness that `source_plan` inherits.

   **Classification:** Preference — the risk is theoretical for plan filenames.
   The existing fields have the same gap. Fixing this would be a broader awk
   hardening effort beyond this ADR's scope.

3. **Stdin/pipe invocation produces empty `source_plan`.** If awk reads from
   stdin (`echo "..." | awk -f extract-summary.awk`), FILENAME is empty or
   unset. The basename extraction would produce an empty string, which maps to
   the `DEFAULT ''` behavior in the database. This is acceptable degradation
   for an undocumented usage pattern, but the ADR doesn't mention it.

   **Classification:** Preference — the documented usage is with file
   arguments. Empty source_plan on stdin is consistent with the `DEFAULT ''`
   design.

4. **View TSV column reorder shifts all existing column positions.** Adding
   `source_plan` as the first TSV column moves `task_id` from column 1 to
   column 2, `status` from column 2 to column 3, etc. The view command is
   explicitly labeled "proto-porcelain" with no stability guarantees
   (`main.rs:33`), so this is by design. But the plan's Task 2.2 acceptance
   criteria don't mention updating the view unit tests that may assert on
   column positions — only "Unit tests updated to verify new column ordering"
   which is vague about what "updated" means.

   **Classification:** Quality concern — the criterion should specify that
   `insert_sample` and all view tests that construct `NewTaskSummary` must
   include `source_plan` values. See also Stage 1 finding #1.

## Stage 3: Testing

### Test-Gap Findings

1. **No test verifies `source_plan` round-trips through the database with a
   non-empty value.** Task 3.1 criteria say "JSONL with source_plan
   deserializes and stores correctly" and "view JSONL output includes
   source_plan field." But the view test fixtures (`insert_sample` in view.rs)
   may use `source_plan: "".to_string()` for all entries, meaning the
   round-trip test only exercises the empty-string case. A test that inserts a
   record with `source_plan: "0029.0.plan.md"` and verifies it appears
   correctly in both TSV and JSONL output would close this gap.

   **Classification:** Quality concern — the stated criteria could pass with
   empty-string-only test data, which doesn't verify the feature works for its
   intended purpose.

2. **Backward compat test verifies deserialization but not database
   round-trip.** Task 3.1 says "JSONL without source_plan deserializes with
   empty string default." This tests serde. But it doesn't test that the empty
   default actually survives insert → query → view. The `DEFAULT ''` in the
   schema and `#[serde(default)]` interact at different layers; testing only
   serde leaves the database layer unverified for the backward-compat case.
   The existing `test_round_trip_insert_query` could cover this if it's updated
   to verify `source_plan` is empty after insertion.

   **Classification:** Quality concern — recommend a criterion: "Round-trip
   test inserts a `NewTaskSummary` without explicit `source_plan` and verifies
   the retrieved `TaskSummary.source_plan` is empty."

   Wait — actually, after Task 1.2, `NewTaskSummary` requires `source_plan` as
   a field (it's `String`, not `Option<String>`). So you can't create a
   `NewTaskSummary` "without" source_plan — you'd pass `"".to_string()`. The
   backward compat boundary is at JSONL deserialization (`JsonlTaskRecord` with
   `#[serde(default)]`), not at the Insertable struct. So the existing
   `test_jsonl_deserialization` test (which uses a JSON string without
   `source_plan`) plus the `From` impl test covers the full backward-compat
   path. Revised classification below.

   **Revised classification:** Preference — the serde-level backward compat
   test combined with the `From` impl mapping is sufficient to verify the
   default-value path. A database round-trip test adds confidence but is not
   strictly necessary.
