### Test-Gap Findings

- **TG1.1** — **Cross-file anchor breakage not covered by dev acceptance criteria.** `docs/PREFERENCES.md:69` contains a link to `prototype-adr/SKILL.md#p-1-select-isolation`. Renaming `### P-1: Select Isolation` to `### E-1: Select Isolation` changes the anchor from `#p-1-select-isolation` to `#e-1-select-isolation`, breaking this link. The dev plan's Task 1.1 implementation notes claim "No cross-file references exist (confirmed by grep) — changes are contained to SKILL.md" — this is incorrect. The grep only searched within `src/skills/prototype-adr/`, missing links from `docs/PREFERENCES.md`. **Verification command:** `grep -rn '#p-[0-6]' docs/ src/ --include='*.md'` should return 0 hits after remediation.

- **TG1.2** — **`make check-refs` does not validate anchors.** The `scripts/check-refs` tool strips anchor fragments (`filepath="${target%%#*}"`) and only checks file existence. Even if the executor runs `make check-refs`, it will not detect the broken anchor in TG1.1. The broken anchor can only be caught by manual grep or an anchor-aware link checker. The dev plan's Stage 4 Task 4.1 criteria include `make test` and `wc -l` but not a repo-wide anchor validation. **Recommended verification:** `grep -rn 'prototype-adr/SKILL.md#p-' docs/ src/ --include='*.md'` after Stage 1 completes.

- **TG1.3** — **Identifier scheme consistency not validated.** The dev plan suggests `E-0`–`E-6` ("Experiment" prefix) but the acceptance criteria only verify absence of P-prefixed identifiers — they don't verify the replacement scheme is internally consistent (all same prefix, sequential numbering, table entries match headers). A partial rename (e.g., table says `E-3` but header says `EX-3`) would pass all three dev acceptance criteria.

### Test-Gap Findings

- **TG2.1** — **Semantic content preservation during elevation.** The dev plan's acceptance criterion "No policy content is lost — full text preserved in the Policies section" is a subjective check. Concrete risk: the inline dispatch compliance paragraph (line 118) includes specific context about "autonomous workflows triggered by other skills (e.g., implement-adr invoking author-adr for ADR creation)." The elevated P-1 text in the plan includes this sentence — but the acceptance criteria don't specify a mechanical verification method (e.g., a diff showing the original paragraph content exists in the new location). **Recommended verification:** After elevation, extract the original paragraph text and confirm every sentence appears in the P-1 section. - **TG2.2** — **Back-reference format consistency.** The dev plan specifies two different back-reference formats: line 118 becomes "See P-1 (Mandatory Dispatch Compliance)" while line 200 becomes "**Guardrail (P-2):** See P-2 (Cross-ADR Modification Guardrail)." The inconsistency is intentional (line 200 already has a `**Guardrail:**` prefix), but the dev acceptance criteria don't verify the back-references are formatted correctly or that they actually point to valid policy IDs. A typo like "See P-3" (nonexistent) would pass the "contains back-references" criterion.

- **TG2.3** — **Document structure integrity after insertion.** The plan inserts ~20 lines between line 10 (skill description) and line 11 (`## Procedure`). The dev acceptance criteria verify section ordering (Policies before Procedure) but don't verify that no existing content is disrupted — specifically that the `## Assets` table (lines 25–32) and the `## Procedure` table (lines 13–22) remain intact. **Recommended verification:** After insertion, confirm `## Procedure` table still has 7 rows (A-0 through A-6) and `## Assets` table entries resolve to existing files.

- **TG2.4** — **Line count after insertion.** Author-adr SKILL.md is currently 296 lines. This is under the 500-line limit, but should be verified explicitly since the dev plan's Task 4.1 criteria check line counts only in Stage 4, not at the stage boundary.
