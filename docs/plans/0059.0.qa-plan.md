## Stage 2: Lifecycle and Step 4d

### Security Checks

- [ ] **S2.1** — The `git merge-base HEAD <base-branch>` and `git diff` commands use a branch name retrieved from session state — verify the instructions do not interpolate user-supplied strings directly into shell commands without validation. The base branch name should come from `git rev-parse --abbrev-ref HEAD` output (machine-generated, not user-supplied)
- [ ] **S2.2** — No secrets in committed files — Step 4d instructions contain only procedural text
- [ ] **S2.3** — N/A — no deserialization introduced. Justification: Step 4d reads git output and dispatches an agent; no structured data parsing
- [ ] **S2.4** — N/A — no dependencies introduced. Justification: procedural text changes only
- [ ] **S2.5** — N/A — no files created. Justification: edits to existing problem.md and SKILL.md
- [ ] **S2.6** — The code review agent dispatch is a new input surface — findings from the dispatched agent are external input. Verify the instructions specify that findings are presented for triage (not auto-applied) in guided mode

### UX Checks

- [ ] **U2.1** — Step 4d failure modes produce clear messages: agent dispatch failure (warn and skip), base branch not found (error with recovery guidance), no diff to review (skip with log)
- [ ] **U2.2** — N/A — Step 4d is a procedural step, not a user-facing command. Justification: the agent executes git commands internally
- [ ] **U2.3** — Invalid states are handled: missing base branch in session state, agent dispatch failure, empty diff
- [ ] **U2.4** — N/A — no persistent resources opened. Justification: git commands are stateless
- [ ] **U2.5** — Step 4d results are observable — findings are presented in the triage output; the agent logs the dispatch result
- [ ] **U2.6** — The new Step 4d state is inspectable — the lifecycle diagram and procedure table show Step 4d's position; the resume protocol documents the new state
- [ ] **U2.7** — A user who did not write the code can verify Step 4d was executed correctly: the triage output shows findings and their disposition; the gate decision (proceed or block) is logged

### Test-Gap Findings

- **TG2.1** — **Sub-step identifier consistency (P-3 Rules 4, 11).** Task 2.2 says "Procedure table updated with Step 4d and its sub-steps." Dev criteria do not specify which alpha-numeric identifiers to use for sub-steps. An implementer could use inconsistent naming (e.g., "Step 4d.1" vs "Step 4d-a") and satisfy all criteria while violating P-3's identifier conventions. The existing pattern in problem.md uses "Step Na" format (e.g., Step 4a, Step 4b, Step 4c) — Step 4d sub-steps should follow "Step 4d-1" through "Step 4d-5" or similar to avoid collision with the step's own "4d" identifier.

- **TG2.2** — **Cross-reference consistency between SKILL.md and problem.md.** Task 2.1 updates the SKILL.md lifecycle diagram. Task 2.2 updates the problem.md procedure flow. No dev criterion verifies the two diagrams match. Both could individually pass their criteria while being inconsistent with each other (e.g., different step descriptions, different ordering).

- **TG2.3** — **Resume state detection mechanism.** Task 2.2 criterion 8 requires adding a resume state for "all ADRs Accepted, Step 4d not yet run." Dev criteria do not specify how the agent detects this state. Since Step 4d is optional and has no persistent artifact, the agent may have no reliable way to distinguish "Step 4d was skipped because unconfigured" from "Step 4d hasn't run yet." The detection mechanism should be documented — e.g., the agent checks both (a) `[solve.dispatch].code_review` is configured and (b) no code review triage output exists in the conversation.

- **TG2.4** — **Step 5 conditional prerequisite.** Task 2.2 criterion 9 says "Step 5 updated to reference Step 4d completion as prerequisite." But Step 4d is optional — if Step 5 unconditionally requires Step 4d completion, it breaks backwards compatibility when `code_review` is unconfigured. The prerequisite must be conditional: "Step 4d complete OR Step 4d skipped (unconfigured)."

- **TG2.5** — **Base branch recording timing in Step 1b.** Task 2.3 says "Record the current branch before `git checkout -b`." Dev criteria verify the instruction is clear about timing. But: what if the user is on a detached HEAD (no branch name)? `git rev-parse --abbrev-ref HEAD` returns `HEAD` for detached states. The instructions should specify fallback behavior — e.g., use the commit SHA as base reference, or warn the user.

- **TG2.6** — **Procedure table completeness (P-3 Rule 6).** Task 2.2 criterion 3 says "Procedure table updated with Step 4d and its sub-steps." The existing procedure table in problem.md has entries for Steps 1–5 and sub-steps (1b, 2a–2c, 3a–3b, 4a–4c). All Step 4d sub-steps (base branch detection, dispatch, findings, triage, gate) must appear in this table with their identifiers. Dev criteria do not enumerate the expected rows, so an implementer could add a single "Step 4d" row without sub-steps and satisfy the criterion literally while violating P-3 Rule 6.
