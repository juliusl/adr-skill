## Stage 1: Format Script

### Security Checks

| ID | Checklist Item | Finding |
|----|---------------|---------|
| S1.1 | No user-supplied strings interpolated into SQL, shell, or file paths (injection) | **FINDING.** The `new` subcommand will build filenames as `"$dir/${remote}-${id}-${slug}.md"`. The `remote` and `id` values are user-supplied and interpolated directly into a file path. A `remote` value of `../../etc` or an `id` containing `/` would write outside the ADR directory. The `rename` and `status` subcommands use `"$dir"/${remote}-${id}-*.md` as a glob — if `remote` or `id` contain glob metacharacters (`*`, `?`, `[`), the pattern expands unpredictably. Additionally, `sed -i "1s/.*/# ${remote}-${id}. ${new_title}/"` will break or inject sed commands if `remote`, `id`, or `title` contain `/`, `&`, or `\`. The existing `nygard-agent-format.sh` has the same `sed` issue on line 278, but the risk is lower there because `display_number` is always numeric. With `remote` and `id` as freeform strings, the attack surface is larger. |
| S1.2 | No secrets, credentials, or API keys in committed files | N/A — no credentials involved. |
| S1.3 | No deserialization of untrusted input without validation | N/A — no deserialization. Shell scripts read text metadata only. |
| S1.4 | Dependencies pinned to specific versions | N/A — no external dependencies. Shell script with coreutils only. |
| S1.5 | File permissions on created artifacts | N/A — files created by redirection inherit umask. Consistent with existing `nygard-agent-format.sh`. No change needed unless the plan makes scripts world-writable. |
| S1.6 | New external input surface has validation at boundary | **FINDING.** The `new` subcommand introduces two new positional parameters (`remote`, `id`) that flow into filenames, headings, sed expressions, and glob patterns. The plan's implementation notes show no input validation. At minimum: (a) `remote` must be validated against an allowlist (`gh`, `ado`, `local`), (b) `id` must be validated as alphanumeric (digits for `gh`/`ado`, hex for `local`), (c) both must be checked for path separators and shell metacharacters. |

### UX Checks

| ID | Checklist Item | Finding |
|----|---------------|---------|
| U1.1 | Every error path produces human-readable message on stderr | **FINDING.** The plan's acceptance criteria for Task 1.1 do not include error cases. What happens when `new` is called with: zero arguments, missing `id`, missing `title`, missing `dir`? The existing `nygard-agent-format.sh` dispatches `cmd_new "$@"` without arg-count validation — it relies on `set -euo pipefail` to crash on unset variables. The new script must validate arg count and emit "Usage:" on stderr. Task 1.3 mentions "Error message when ADR not found" but has no criterion for missing/empty `remote` or `id` arguments. |
| U1.2 | Every command exits 0 on success, non-zero on failure | N/A if inherited from existing pattern (existing scripts use `exit 1` on error). Verify the new script follows the same convention. |
| U1.3 | Invalid input rejected with helpful message, not crash | **FINDING.** If `remote` is not in the known set (`gh`, `ado`, `local`), or `id` is empty, `set -euo pipefail` will produce a raw bash error (e.g., `unbound variable`) rather than a human-readable message. The plan does not specify validation for unknown remote types. A user passing `remote=github` instead of `remote=gh` will get either a crash or a silently misspelled file. |
| U1.4 | Resources cleaned up on error paths | N/A — no resources to clean up beyond file writes. If `generate_template` fails mid-write, a partial file is left. Same behavior as existing script — no regression. |
| U1.5 | Written data can be read back or verified | Covered by `cat` in test scripts. No new finding. |
| U1.6 | Created state can be inspected | OK — files in the ADR directory are inspectable via `ls`, `cat`, and `list`. |
| U1.7 | Non-author can verify output through tool interface | **FINDING.** The `list` subcommand (Task 1.2) must show work-item-prefixed ADRs. The plan says "sorted by Date: metadata" but the acceptance criteria do not verify sort order with multiple work-item ADRs having different dates. A user cannot verify ordering is correct without a test that creates ADRs with different dates and checks output order. |

### Test-Gap Findings

| ID | Finding |
|----|---------|
| TG1.1 | **No test for titles containing sed metacharacters.** If a title is `Use foo/bar & baz`, the `sed` expression `1s/.*/# ${remote}-${id}. ${new_title}/` will fail. The existing test suite does not cover this for `nygard-agent-format.sh` either, but the new format script widens the exposure because `remote` and `id` also appear in the sed pattern. At minimum, one test should use a title containing `/`. |
| TG1.2 | **No test for `id` containing non-numeric characters with `gh` or `ado` remotes.** The ADR says `gh` maps to GitHub Issue numbers and `ado` maps to ADO work item IDs — both should be numeric. But nothing in the plan validates this. A test with `id=42a` would reveal whether the script silently creates a misspelled file. |
| TG1.3 | **No test for duplicate creation.** The ADR notes that same-work-item collisions are intentional ("surfaces a genuine decision conflict"). But the plan has no test verifying the behavior when `new gh 42 "Title A"` is called twice. Does the second call overwrite the first? Error out? Silently succeed with a different slug? This is a critical user-facing behavior left unspecified. |
| TG1.4 | **No test for empty `id` or empty `remote`.** `new "" "" "Title"` should produce an error, not a file named `---title.md`. |
| TG1.5 | **No test that `local` remote generates a valid 8-char hex ID.** The ADR says `local` uses "short hash (8 chars)." The plan's acceptance criteria test `new local <id> <title>` — the ID is user-supplied. But in real usage, who generates the local ID? The plan doesn't address auto-generation. If the user must supply it manually, there's no validation that it's 8 hex chars. |
| TG1.6 | **`list` date-based sort has no test with controlled dates.** Task 1.2 says work-item ADRs sort by `Date:` metadata. The test harness uses `ADR_DATE=2026-01-15` (a fixed date) — all ADRs in a test will have the same date. The test cannot verify sort order. A test needs at least two ADRs with different `ADR_DATE` values. |

## Stage 2: Orchestrator and Makefile Integration

### Security Checks

| ID | Checklist Item | Finding |
|----|---------------|---------|
| S2.1 | No user-supplied strings interpolated into SQL, shell, or file paths (injection) | **FINDING.** The plan shows `new.sh` doing: `remote="${1:-}"; shift \|\| true; id="${1:-}"; shift \|\| true; title="$*"; exec "$format_script" new "$remote" "$id" "$title" "$adr_dir"`. This passes unsanitized `remote` and `id` through to the format script. The orchestrator is the natural validation boundary — it should validate `remote` and `id` before exec. The Makefile target `new` uses `$(ORCHESTRATOR) $(ADR_AGENT_SKILL_FORMAT) $(REMOTE) $(ID) $(TITLE)` — Make does not quote these variables, so values with spaces or shell metacharacters will word-split. |
| S2.2 | No secrets | N/A. |
| S2.3 | No deserialization of untrusted input | N/A. |
| S2.4 | Dependencies pinned | N/A. |
| S2.5 | File permissions | N/A. |
| S2.6 | Validation at input boundary | **FINDING.** The orchestrator (`new.sh`) is the entry point for both human users and the Makefile agent interface. The plan's `case` dispatch checks format name but does not validate `remote` or `id`. This is the correct place to reject bad input before it reaches the format script. The plan does not mention validation here. |

### UX Checks

| ID | Checklist Item | Finding |
|----|---------------|---------|
| U2.1 | Error paths produce human-readable messages | **FINDING.** Task 2.1 acceptance criteria say "Error when remote or id missing for wi-nygard-agent format." Good — but the plan snippet shows `remote="${1:-}"; shift \|\| true` which silently sets remote to empty string and continues. The error check must come after the assignment, before exec. If the error message is missing, the format script will crash with a raw bash error instead. |
| U2.2 | Exit codes | Verify backward-compatible: `new.sh nygard-agent "Title"` must still work. Acceptance criteria cover this. |
| U2.3 | Invalid input rejected helpfully | **FINDING.** The plan mentions error for missing remote/id but not for invalid format names. Currently, `new.sh` checks `[ ! -x "$format_script" ]` — this handles unknown formats. The new case dispatch must preserve this fallback. If a user types `new.sh wi-nygard "Title"` (typo), the case falls through to the sequential path, silently creates a sequential ADR. This is a behavioral surprise. |
| U2.4 | Resources cleaned up | N/A. |
| U2.5 | Written data verifiable | OK — the created file path is echoed. |
| U2.6 | Created state inspectable | OK. |
| U2.7 | Non-author can verify output | **FINDING.** The Makefile targets in Task 2.2 use `$(NUM)`, `$(REMOTE)`, `$(ID)`, `$(STATUS)` Make variables. If a user calls `make status REMOTE=gh ID=42` with format `nygard-agent`, the format script receives `remote` where it expects a number. The plan does not specify how the Makefile detects which variables to use based on format. The acceptance criterion "Existing sequential targets still work when format is nygard-agent" is critical but the plan shows no conditional logic in the Makefile to dispatch differently. |

### Test-Gap Findings

| ID | Finding |
|----|---------|
| TG2.1 | **No test for `new.sh` with wrong argument order.** `new.sh wi-nygard-agent "Use PostgreSQL" gh 42` (title before remote/id) will silently create a file with remote=`Use` and id=`PostgreSQL`. There's no way for the script to detect this since all args are strings. |
| TG2.2 | **No test for Makefile variable quoting.** `make new REMOTE=gh ID=42 TITLE="Use PostgreSQL for persistence"` — Make splits `TITLE` on spaces when passing to shell. The test should verify that multi-word titles survive the Make→shell boundary. |
| TG2.3 | **Backward compatibility test coverage.** Task 2.1 says "Existing tests pass (`make test`)." This is necessary but not sufficient — the existing tests don't exercise the `case` dispatch path. A test that calls `new.sh nygard-agent "Title"` after the case dispatch is added would verify the fall-through path. |
| TG2.4 | **No test for `adr_dir` resolution with wi-nygard-agent.** The orchestrator reads `.adr/adr-dir` to resolve the directory. A test should verify that `new.sh wi-nygard-agent gh 42 "Title"` respects this config file. The existing `resolve-dir-priority` test only covers the sequential path. |
