## Stage 1: Create adr-db-lib Crate

### Test-Gap Findings

1. **`embed_migrations!()` path resolution depends on `diesel.toml` location
   relative to `CARGO_MANIFEST_DIR`.** The plan says to move `diesel.toml` and
   `migrations/` to `adr-db-lib/`, and `db.rs` will call `embed_migrations!()`.
   The Diesel `embed_migrations!()` macro resolves migration paths relative to
   `CARGO_MANIFEST_DIR` (the directory containing `Cargo.toml`), *not* relative
   to the source file calling the macro. If `diesel.toml` is present, it reads
   the `[migrations_directory]` key; if absent, it defaults to `./migrations/`.
   The plan's `diesel.toml` only specifies `[print_schema] file = "src/schema.rs"`
   and has no `[migrations_directory]` key, so the default `./migrations/`
   relative to `adr-db-lib/Cargo.toml` will be used. This should work — but the
   acceptance criteria do not verify that `embed_migrations!()` compiles
   correctly. The `cargo build -p adr-db-lib` criterion catches this implicitly,
   but a compilation failure at the `embed_migrations!()` macro call site
   produces a cryptic "could not find migrations directory" error that may
   confuse the developer.

   **Classification:** Preference — the `cargo build` criterion already catches
   this. Adding a note about the resolution path would help debugging but is not
   strictly necessary.

2. **No acceptance criterion verifies that `db.rs` compiles with the correct
   `embed_migrations!()` expansion.** The criterion says "`cargo build -p
   adr-db-lib` succeeds," which is sufficient. However, `embed_migrations!()`
   is a `const` evaluation that embeds SQL files at compile time. If the
   migration directory structure is incomplete (e.g., missing `up.sql` file),
   the error will be a build failure — but the error message points at the
   macro call site, not the missing file. This is a standard Diesel behavior
   and not a plan defect, but it is a known footgun during migration moves.

   **Classification:** Preference — standard Diesel behavior; the build
   criterion catches it.

3. **`cargo build -p adr-db-lib` does not compile `#[cfg(test)]` code.**
   Stage 1 creates `adr-db-lib` as a library crate. If any test code is added
   to the library crate (e.g., unit tests for `db.rs`), `cargo build` won't
   compile it. The plan does not add tests to `adr-db-lib` in Stage 1, so this
   is not a gap *now* — but it is worth noting for future stages. The
   workspace-level `cargo test` in Stage 2 would catch it.

   **Classification:** Preference — no test code exists in `adr-db-lib` at
   this stage.

4. **`models.rs` has `use crate::schema::task_summaries` — moving it verbatim
   to `adr-db-lib` changes the crate root.** The plan says `models.rs` is
   "unchanged." However, `models.rs:4` contains `use crate::schema::task_summaries;`.
   In `adr-db`, `crate` refers to the `adr-db` binary crate. In `adr-db-lib`,
   `crate` refers to the `adr-db-lib` library crate. Since `lib.rs` declares
   `pub mod schema`, the `use crate::schema::task_summaries` path resolves
   correctly in the new crate — the re-anchoring is transparent. So the file
   *is* unchanged and correct. No action needed.

   **Classification:** Not a gap — `crate::` re-anchors automatically to the
   new crate root. Documented here for traceability.

5. **No criterion verifies that the moved `migrations/` directory contains both
   migration subdirectories.** The criterion says "migrations/ directory moved
   with both migration subdirectories intact," which is good. However, it does
   not specify that each subdirectory must contain both `up.sql` and `down.sql`.
   A partial copy (e.g., missing `down.sql`) would still pass the "subdirectories
   intact" check but would cause `embed_migrations!()` to fail at build time.
   The `cargo build` criterion catches this transitively.

   **Classification:** Preference — the build criterion is the real gate.

## Stage 2: Rewire adr-db Binary Crate

### Test-Gap Findings

1. **`view.rs` tests use a local `embed_migrations!()` that will break after
   `migrations/` is removed from `adr-db`.** Currently, `view.rs:138` declares
   `const MIGRATIONS: EmbeddedMigrations = embed_migrations!();` inside the
   `#[cfg(test)]` module. The `embed_migrations!()` macro resolves migrations
   relative to `CARGO_MANIFEST_DIR` — which is `crates/adr-db/`. After
   Stage 2 removes `migrations/` from `adr-db`, this macro call will fail with
   "could not find migrations directory."

   The plan says (Task 2.1, point 4): "Update view.rs tests — the
   `embed_migrations!()` calls in test helpers need to reference
   `adr_db_lib::db::MIGRATIONS` instead of embedding migrations locally." This
   is correct, but the acceptance criteria don't explicitly call it out as a
   separate checkable item. The `cargo test --workspace` criterion catches it,
   but the developer might be confused by a compilation error in test code when
   the plan's focus is on import rewiring.

   **Classification:** Quality concern — add an acceptance criterion:
   "`view.rs` test module uses `adr_db_lib::db::MIGRATIONS` instead of a local
   `embed_migrations!()`." This makes the migration dependency explicit and
   prevents the developer from debugging a cryptic macro resolution failure.

2. **`models.rs` tests also use a local `embed_migrations!()` that will break.**
   The `test_round_trip_insert_query` test in `models.rs:88` declares its own
   `const MIGRATIONS: EmbeddedMigrations = embed_migrations!();`. After
   removing `migrations/` from `adr-db`, this will also fail. The plan's Task
   2.1 says to "update imports in `main.rs`, `ingest.rs`, and `view.rs`" but
   does not mention `models.rs` test code. The `models.rs` file itself is
   removed from `adr-db` (moved to `adr-db-lib`), so its tests are no longer
   in `adr-db`. However, the plan says `models.rs` is "unchanged" when moved
   — meaning the `embed_migrations!()` call in the test module should still
   work because migrations are now co-located in `adr-db-lib`.

   Wait — let me re-read the plan. Task 1.1 says `models.rs` is moved
   "unchanged." The test code in `models.rs:83-118` uses
   `use crate::schema::task_summaries;` (line 4) which re-anchors correctly,
   and `embed_migrations!()` (line 88) which will resolve relative to
   `adr-db-lib/Cargo.toml` — where `migrations/` now lives. So the test
   compiles correctly in the new location. No gap.

   **Classification:** Not a gap — `models.rs` moves to `adr-db-lib` where
   migrations are co-located. Documented for traceability.

3. **The plan says to remove `diesel` and `diesel_migrations` from `adr-db`'s
   direct dependencies. But `view.rs` uses `diesel::prelude::*`,
   `diesel::sql_types::Text`, and `diesel::sqlite::SqliteConnection` directly.**
   Removing `diesel` from `adr-db/Cargo.toml` means these imports must come
   transitively through `adr-db-lib`. For transitive re-exports to work,
   `adr-db-lib` must either re-export diesel types or `adr-db` must keep
   `diesel` as a direct dependency.

   The plan says `adr-db-lib/Cargo.toml` depends on `diesel`. In Rust,
   transitive dependencies are *not* automatically available to downstream
   crates unless they are re-exported. `view.rs` uses `diesel::prelude::*` and
   `diesel::sqlite::SqliteConnection` directly — these are `diesel` crate
   imports, not `adr-db-lib` re-exports. After removing `diesel` from
   `adr-db/Cargo.toml`, these imports will fail.

   **Resolution options:**
   - Keep `diesel` as a direct dependency in `adr-db/Cargo.toml` (contradicts
     the plan's "remove diesel" instruction).
   - Have `adr-db-lib` re-export `diesel` (e.g., `pub use diesel;` in
     `lib.rs`), then `view.rs` uses `adr_db_lib::diesel::prelude::*`.
   - Rewrite `view.rs` to avoid direct diesel imports by moving query logic
     to `adr-db-lib`.

   The plan says "remove diesel and diesel_migrations direct dependencies (they
   come transitively through adr-db-lib)." This is **incorrect for Rust** —
   transitive deps are not automatically usable unless re-exported. This is a
   plan defect that will cause compilation failure.

   **Classification:** Quality concern — the plan's instruction to remove
   `diesel` from `adr-db/Cargo.toml` will cause `view.rs` to fail compilation.
   Either: (a) keep `diesel` as a direct dep in `adr-db`, (b) add
   `pub use diesel;` to `adr-db-lib/lib.rs`, or (c) move query logic to the
   library. Recommend option (a) as the minimal fix — it's the standard Rust
   pattern for binary crates that use library types directly.

4. **`view.rs` tests import `crate::models::NewTaskSummary` — after removal,
   this must become `adr_db_lib::models::NewTaskSummary`.** The plan's
   acceptance criterion says "All imports in `ingest.rs` and `view.rs`
   reference `adr_db_lib::`" which covers this. But the test module's imports
   (line 135: `use crate::models::NewTaskSummary`) are easy to overlook because
   they are inside `#[cfg(test)]` which `cargo build` doesn't compile.

   **Classification:** Quality concern — `cargo build -p adr-db` will succeed
   even if test imports are wrong. Only `cargo test` catches test compilation
   errors. Recommend changing the Stage 2 build criterion from "`cargo build -p
   adr-db` succeeds" to "`cargo test -p adr-db --no-run` succeeds" to ensure
   test code compiles.

5. **`ingest.rs` uses `diesel::Connection`, `diesel::RunQueryDsl`, and
   `diesel::sqlite::SqliteConnection` directly.** Same issue as finding #3 —
   removing `diesel` from `adr-db/Cargo.toml` will break `ingest.rs` imports.
   The plan says to update imports to `adr_db_lib::` but `SqliteConnection`
   and `RunQueryDsl` are diesel types, not adr-db-lib types.

   **Classification:** Duplicate of finding #3. Same root cause — transitive
   dependency visibility in Rust.

6. **`view.rs` derives `clap::ValueEnum` for `OutputFormat` — clap is a direct
   `adr-db` dependency that stays.** The plan correctly says to keep `clap` in
   `adr-db`. No issue here. Documented for completeness.

   **Classification:** Not a gap.
