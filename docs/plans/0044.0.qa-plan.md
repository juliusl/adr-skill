## Stage 1: Create solve-adr Skill

### Security Checks

- [ ] **S1.1** — Makefile targets do not interpolate user-supplied strings into shell commands (verify hardcoded paths only)
- [ ] **S1.2** — No secrets, credentials, or API keys in SKILL.md, solve.md, or Makefile
- [ ] **S1.3** — N/A — no deserialization of untrusted input (stage creates static documentation files; preference TOML parsing is agent runtime behavior, not code in this stage)
- [ ] **S1.4** — N/A — no external dependencies introduced (markdown files and a Makefile only)
- [ ] **S1.5** — Created files (SKILL.md, solve.md, Makefile) have standard permissions (not world-writable)
- [ ] **S1.6** — S-0 preference loading instructions specify hardcoded paths (`~/.config/adr-skills/preferences.toml`, `.adr/preferences.toml`), not user-supplied input

### UX Checks

- [ ] **U1.1** — Makefile `help` target produces readable output listing available targets
- [ ] **U1.2** — Makefile targets exit 0 on success, non-zero on failure
- [ ] **U1.3** — Makefile handles missing prerequisites (e.g., missing skill validator) with a clear error message, not a crash
- [ ] **U1.4** — N/A — no resources to clean up (stage creates static files only)
- [ ] **U1.5** — Created files can be verified via `make validate` and `make check-refs`
- [ ] **U1.6** — SKILL.md and solve.md structure can be inspected by reading the files directly
- [ ] **U1.7** — Another developer can run `make validate` to verify SKILL.md spec conformance without reading the source

### Test-Gap Findings

- **TG1.1** — Acceptance criteria verify solve-adr description excludes author-adr triggers (Task 1.1: "Description excludes create/review/revise triggers"), but do not verify the *converse*: that author-adr description excludes solve-adr triggers. Trigger overlap is tested piecemeal across stages (Task 1.1 and Task 2.3) but never as a cross-skill assertion. Both descriptions could pass their individual criteria while still sharing ambiguous phrases like "document this decision."

- **TG1.2** — Task 1.2 migrates solve.md content from author-adr to solve-adr, but acceptance criteria have no migration completeness check. All criteria verify presence of named sections in the new file — none compare against the source to confirm nothing was dropped. If a discovery technique or multi-turn session management detail is omitted, all tests pass.

- **TG1.3** — S-0 preference loading describes reading `preferences.toml` from two scopes, but acceptance criteria do not verify that the preference keys (`participation`, `auto_delegate`, `default_scenario`) and their valid values are documented in SKILL.md or solve.md. An implementer could describe preference loading without documenting the schema.

- [ ] **S2.6** — No new external input surfaces introduced (removing content reduces attack surface)

### UX Checks

- [ ] **U2.1** — N/A — no new commands or error paths; stage modifies existing files
- [ ] **U2.2** — N/A — no new commands
- [ ] **U2.3** — N/A — no new input paths
- [ ] **U2.4** — N/A — no resources to manage
- [ ] **U2.5** — Solve content removal verified by `make check-refs` passing (no dangling references to deleted solve.md)
- [ ] **U2.6** — author-adr SKILL.md can be inspected post-edit to confirm solve content is gone
- [ ] **U2.7** — `make test` and `make check-refs` provide third-party verification that removal caused no regressions

### Test-Gap Findings

- **TG2.1** — Acceptance criteria check for removal of specific strings ("Solving a Problem", "Solve vs. Create", solve-related flow chart entries) but do not verify the remaining routing flow chart is structurally valid. If the removal leaves orphaned lines, misaligned indentation, or broken ASCII art, all string-absence tests pass but the flow chart is unreadable.

- **TG2.2** — The redirect note to `/solve-adr` is checked for existence (Task 2.1: "Redirect note to `/solve-adr` is present") but not placement. If the note appears in a non-obvious location (e.g., appended at the bottom of the file), users who previously said "help me solve" to author-adr may not discover it. The criteria should verify the note appears in or near the routing section where solve was removed.

### UX Checks

- [ ] **U3.1** — `make validate-solve` produces a human-readable error message if validation fails
- [ ] **U3.2** — `make validate-solve` exits 0 on success, non-zero on failure
- [ ] **U3.3** — `make validate-solve` fails gracefully with a helpful message if solve-adr directory does not exist
- [ ] **U3.4** — N/A — no resources to clean up
- [ ] **U3.5** — eval_queries.json is validated by `python3 -m json.tool` (already in acceptance criteria); Makefile targets verifiable via `make help`
- [ ] **U3.6** — Root Makefile `help` target updated to list `validate-solve` alongside existing validate targets
- [ ] **U3.7** — eval_queries.json structure is verifiable by running the evaluation workflow documented in AGENTS.md

### Test-Gap Findings

- **TG3.1** — eval_queries.json acceptance criteria verify structural correctness (valid JSON, queries present, no duplicates) but do not test actual trigger discrimination. Running `python3 -m json.tool` validates syntax, not semantics. Both skill descriptions could accept the same query if the eval isn't run against the actual descriptions. This is the primary observability gap for the trigger separation concern raised in ADR-0044's Quality Strategy.

- **TG3.2** — Task 3.4 updates `install-skills` to install solve-adr, but no acceptance criterion verifies the installation works end-to-end. The copy target could have a typo or missing path, and all existing tests (`make test`, `make validate-all`) would still pass.
