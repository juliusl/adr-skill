## Stage 1: Schema Foundation

### Task 1.1: Create adr-format crate with TOML schema structs    [medium]

**Description:** Create a new `adr-format` crate in the `crates/` workspace. Define Rust structs with `serde::Serialize + Deserialize` that implement the wi-full-agent-adr TOML schema from ADR-0051. The structs ARE the schema — they define what fields exist, which are required, and their types.

**Implementation Notes:**

Add `adr-format` to `crates/Cargo.toml` workspace members. Create `crates/adr-format/Cargo.toml` with dependencies: `serde` (with derive), `toml`, `clap` (with derive), `chrono`. Create `crates/adr-format/src/lib.rs` with the schema module.

Key structs from ADR-0051 schema:

```rust
#[derive(Serialize, Deserialize)]
pub struct Adr {
    pub meta: Meta,
    pub context: Section,
    pub options: Vec<AdrOption>,
    pub evaluation_checkpoint: Checkpoint,
    pub decision: Section,
    pub consequences: Vec<Consequence>,
    pub deliverables: Option<Deliverables>,
    pub quality_strategy: QualityStrategy,
    pub conclusion_checkpoint: Checkpoint,
    pub plan: Option<Plan>,
    pub comments: Option<Comments>,
}

#[derive(Serialize, Deserialize)]
pub struct Meta {
    pub title: String,
    pub date: String,
    pub status: String,
    pub last_updated: String,
    pub work_item: String,       // "{remote}#{id}" format
    pub links: Vec<String>,
}

#[derive(Serialize, Deserialize)]
pub struct Checkpoint {
    pub assessment: String,      // mandatory — empty is a validation error
    // ... checklist booleans
}
```

Status validation: `Prototype`, `Proposed`, `Accepted`, `Delivered`, `Deprecated`, `Superseded`. Reject unknown values at deserialization.

Mutual exclusivity: `major_semantic_changes` and `minor_semantic_changes` — both `true` is invalid (ADR-0051 design rule 10).

**Test & Acceptance Criteria:**
- [x] `adr-format` crate exists in `crates/` workspace and compiles
- [x] All schema sections from ADR-0051 are represented as structs: meta, context, options, evaluation_checkpoint, decision, consequences, deliverables, quality_strategy, conclusion_checkpoint, plan, comments
- [x] Round-trip test: serialize an Adr struct to TOML string, deserialize back, assert equality
- [x] Invalid status values are rejected at deserialization
- [x] Empty `assessment` field in checkpoints is rejected
- [x] `major_semantic_changes = true` + `minor_semantic_changes = true` is rejected by validation
- [x] Deserializing TOML without optional sections (deliverables, plan, comments) succeeds and produces `None` values
- [x] `plan.stages` uses the schema: `{ name, status, tasks: [{ description, done }] }`
- [x] `comments.revision_entries` uses the schema: `{ id, date, body }`

**Dependencies:** None

**ADR Reference:** ADR-0051, Decision §Key design rules 1–10

**Description:** Add a `main.rs` to `adr-format` with clap-derived argument parsing. Implement the `new`, `init`, and `list` subcommands matching the interface of `wi-nygard-agent-format.sh`.

**Implementation Notes:**

```rust
#[derive(Parser)]
#[command(name = "adr-format")]
enum Cli {
    New { remote: String, id: String, title: String, dir: String },
    Init { dir: Option<String> },
    List,
    Rename { remote: String, id: String, new_title: String },
    Status { remote: Option<String>, id: Option<String>, new_status: Option<String> },
    Lifecycle { remote: String, id: String, #[arg(long)] auto: bool, #[arg(long)] sync: bool },
    Export { remote: String, id: String },
}
```

ADR directory resolution: `.adr/adr-dir` > `.adr-dir` > `docs/adr` (same as existing scripts).

### Task 2.2: Implement rename/status/lifecycle subcommands    [medium]

**Description:** Complete the remaining subcommands: `rename`, `status`, and `lifecycle`. These mirror the behavior of `wi-nygard-agent-format.sh` but operate on `.toml` files using serde deserialization.

**Implementation Notes:**

- `rename`: find `.toml` file by remote+id glob, deserialize, update meta.title, update meta.last_updated, reserialize, rename file
- `status`: without args → list all; with remote+id → show status; with remote+id+new_status → update status field in TOML and reserialize
- `lifecycle`: read cached work item from `.adr/var/work-items.jsonl`, compare states, recommend/execute transition. Requires `serde_json` for JSONL parsing.

Status update must deserialize the full TOML, modify only the status field, then reserialize — preserving all other content. This is safer than the awk-based in-place editing used by the shell scripts.

**Test & Acceptance Criteria:**
- [x] `adr-format rename gh 42 "New Title"` updates the file title, heading, last_updated, and filename
- [x] `adr-format status gh 42` prints the current status
- [x] `adr-format status gh 42 Proposed` updates the status field
- [x] Status update preserves all other TOML content (round-trip fidelity)
- [x] `adr-format lifecycle gh 42` reads cached work item and reports sync status
- [x] `adr-format lifecycle gh 42 --auto` executes the transition and logs to `.adr/var/lifecycle.jsonl`

**Dependencies:** Task 2.1

**ADR Reference:** ADR-0052, Decision §Implementation specifics 3; ADR-0038 (lifecycle)

## Stage 3: Markdown Export

### Task 3.1: Implement export subcommand    [medium]

**Description:** Add the `export` subcommand to `adr-format`. It reads a `.toml` ADR, extracts the decision record sections (context, options, decision, consequences), and renders them as nygard-agent-style Markdown to stdout.

**Implementation Notes:**

Per ADR-0053 export rules:
- **Included:** meta (title, date, status, links, work-item), context, options, decision, consequences
- **Excluded:** evaluation_checkpoint, conclusion_checkpoint, quality_strategy, plan, comments, deliverables
- Consequences grouped under Positive/Negative/Neutral headers
- Work-Item line derived from `meta.work_item`; omitted if absent
- Empty optional sections omitted entirely
- Output is idempotent

```rust
fn export_to_markdown(adr: &Adr) -> String {
    let mut md = String::new();
    // title heading
    writeln!(md, "# {}", adr.meta.title);
    writeln!(md, "");
    writeln!(md, "Date: {}", adr.meta.date);
    // ... selective rendering
}
```

**Test & Acceptance Criteria:**
- [ ] `adr-format export gh 42` outputs valid Markdown to stdout
- [ ] Output includes: title, date, status, links, work-item, context, options, decision, consequences
- [ ] Output excludes: evaluation_checkpoint, conclusion_checkpoint, quality_strategy, plan, comments, deliverables
- [ ] Consequences are grouped under `**Positive:**`, `**Negative:**`, `**Neutral:**` headers
- [ ] Work-Item line is omitted when `meta.work_item` is empty
- [ ] Empty sections (e.g., no negative consequences) are omitted entirely
- [ ] Snapshot test: golden-file comparison for a known TOML input

**Dependencies:** Task 1.1

**ADR Reference:** ADR-0053, Decision §Export rules 1–8
