## Stage 1: CLI Structure & Table Listing

### Security Checks

- [ ] **S1.1 — No user-supplied strings in SQL for table listing.** The `sqlite_master` query is a static string. Verify no `format!()` or string interpolation is used to construct it. The plan's implementation notes are correct — confirm the final code matches.
- [ ] **S1.2 — `--output` accepts arbitrary strings.** The plan defines `output: String` in the clap struct. Any string is accepted at parse time (e.g., `--output "'; DROP TABLE"`) — this value reaches application code without clap validation. While the match-per-table pattern protects SQL, the output string must also be validated before use. If the `output` string is ever interpolated into a format string or error message, it could cause issues. **Recommendation:** Use a clap `ValueEnum` instead of `String` so invalid formats are rejected at the argument-parsing boundary, never reaching application code.
- [ ] **S1.3 — `--db-path` path traversal.** The `db_path` parameter accepts arbitrary file paths (e.g., `--db-path /etc/passwd`). This is consistent with existing `init` and `ingest` commands, so it is not a new attack surface. Verify `view` does not create files or directories (unlike `init`). Read-only access to an arbitrary path is the expected behavior.
- [ ] **S1.4 — `--limit` negative value injection.** `limit: Option<i64>` allows negative values. In SQLite, `LIMIT -1` means "no limit." While not a security vulnerability (it's equivalent to omitting `--limit`), it's an unvalidated input that may behave unexpectedly. Verify `--limit 0` and `--limit -1` do not cause panics.
- [ ] **S1.5 — Dependencies pinned to major versions.** Cargo.toml pins to `clap = "4"`, `diesel = "2"`, `serde = "1"`, `serde_json = "1"`. These are semver-compatible ranges, not wildcards. Acceptable for a Rust workspace — no change needed.
- [ ] **S1.6 — No new files created by `view`.** Verify the `view` subcommand is strictly read-only — it must not create the database file, directories, or any other artifacts. (Unlike `init`, which creates `.adr/var/`.)

### UX Checks

- [ ] **U1.1 — Database-not-found produces a clear error.** If `--db-path` points to a nonexistent file, the connection attempt will fail. Verify the error message says something like `error: could not open database: <path>` on stderr, not a Diesel/rusqlite panic or connection string error.
- [ ] **U1.2 — Exit code is 0 on success, non-zero on failure.** Verify `adr-db view` exits 0 when listing tables, and exits 1 when given an invalid table name or unreachable database.
- [ ] **U1.3 — Invalid `--output` value produces a helpful message.** If `output` remains a `String`, then `--output csv` must produce a clear error (e.g., `error: unsupported output format: csv`) and exit 1 — not a Rust panic or raw `match` fallthrough. If changed to a `ValueEnum`, clap handles this automatically.
- [ ] **U1.4 — `sqlite_sequence` table is filtered from listing.** The migration uses `AUTOINCREMENT`, which causes SQLite to create an internal `sqlite_sequence` table. The current filter (`name NOT LIKE '__diesel%'`) does not exclude it. Users running `adr-db view` will see `sqlite_sequence` alongside `task_summaries` — this is confusing because it's not a user-facing table. **Recommendation:** Add `AND name NOT LIKE 'sqlite_%'` to the query filter.
- [ ] **U1.5 — Empty database (no user tables) produces no output and exits 0.** Verify this against a database that has been initialized (migrations run) but has the only table being `sqlite_sequence` and `__diesel_schema_migrations`.
- [ ] **U1.6 — Table names are sorted or in deterministic order.** The `sqlite_master` query does not specify `ORDER BY`. Verify the output order is stable across runs. If not, add `ORDER BY name`.
- [ ] **U1.7 — A user can verify the stage's output.** After implementation, running `adr-db view` against an initialized database should list `task_summaries`. This is self-verifying.

### Test-Gap Findings

- **TG1.1 — `sqlite_sequence` table leaks into listing.** The `AUTOINCREMENT` on `task_summaries.id` creates a `sqlite_sequence` table. The plan's filter (`NOT LIKE '__diesel%'`) misses it. All acceptance criteria pass (they test for `task_summaries` being present), but the output also includes an internal table. A user seeing `sqlite_sequence` in the list will be confused. (**Quality concern: UX violation — confusing output**)
- **TG1.2 — No test for database-not-found error path.** The acceptance criteria test empty databases but not missing databases. A missing db file will hit `SqliteConnection::establish()` error handling, which is wired in `main()`. But no test confirms the error message is human-readable. (**Quality concern: crash-inducing gap — unverified error path**)
- **TG1.3 — `--output` as String bypasses clap validation.** An invalid format string like `--output xml` passes argument parsing and reaches match-dispatch code. If that code has a `_ => unreachable!()` or `_ => panic!()` arm instead of a proper error, the tool crashes. (**Quality concern: crash-inducing gap**)

## Stage 2: Data Output Formats

### Security Checks

- [ ] **S2.1 — User-supplied table name never enters SQL.** The match-per-table pattern (`match table_name.as_str() { "task_summaries" => ..., _ => error }`) ensures no user input reaches SQL. Verify the final code does not use `format!()` or string interpolation anywhere in query construction.
- [ ] **S2.2 — TSV output does not execute field values.** Field values from the database are printed to stdout. If a field contains shell metacharacters (e.g., `$(rm -rf /)`), they are inert in TSV output. No action needed, but confirm no `format!` with field values is used in a way that could be misinterpreted.
- [ ] **S2.3 — JSONL output uses serde_json serialization.** Verify `serde_json::to_string()` is used for JSON encoding, not manual string formatting. This ensures proper escaping of quotes, backslashes, and control characters in field values.
- [ ] **S2.4 — No secrets in test fixtures.** The plan's implementation notes use placeholder data (`abc1234`, `def5678`). Verify no real commit SHAs, file paths, or identifiable data appear in test fixtures.

### UX Checks

- [ ] **U2.1 — TSV description truncation handles multi-byte UTF-8 correctly.** The plan says truncate `description` to 60 characters. If truncation is implemented as byte-level slicing (e.g., `&s[..60]`), descriptions with multi-byte characters (emoji, CJK, accented characters) will panic on a non-character boundary. **Recommendation:** Use `.chars().take(60)` or `.char_indices()` for safe truncation.
- [ ] **U2.2 — Truncation ellipsis character is UTF-8 safe.** The plan specifies `…` (U+2026, 3-byte UTF-8 sequence). Verify this character renders correctly when output is piped through `awk`, `cut`, or `grep` in common locale settings. Consider whether ASCII `...` would be safer for composability.
- [ ] **U2.3 — Tab characters in field values break TSV.** If a `description` or other field contains a literal tab (`\t`) or newline (`\n`), TSV output becomes malformed — the field merges with adjacent columns. The plan does not address sanitization. **Recommendation:** Replace `\t` with a space and `\n` with `\n` or a space in TSV mode.
- [ ] **U2.4 — Unknown table name error goes to stderr, not stdout.** Verify `eprintln!` is used (not `println!`), so that stdout remains clean for piping.
- [ ] **U2.5 — Empty table in TSV mode shows header only.** Verify the header row is printed even when no data rows exist, and exit code is 0.
- [ ] **U2.6 — Empty table in JSONL mode produces zero lines.** Verify no header, no wrapper, no empty array — just zero lines of output. Exit code 0.
- [ ] **U2.7 — `id` and `ingested_at` columns are intentionally excluded.** The `TaskSummary` struct has 7 fields (`id`, `task_id`, `status`, `cost`, `commit_sha`, `description`, `ingested_at`) but the TSV header shows only 5 (`task_id`, `status`, `cost`, `commit_sha`, `description`). Verify this is intentional and that the output function selects only the 5 user-facing fields — not that someone forgot to update the header after adding columns.
- [ ] **U2.8 — Data goes to stdout, errors to stderr.** Verify that all data output (TSV rows, JSONL lines, table names) goes to `println!` (stdout) and all errors go to `eprintln!` (stderr). This is critical for `adr-db view task_summaries | wc -l`.

### Test-Gap Findings

- **TG2.1 — UTF-8 truncation boundary panic.** If `description` is truncated using byte slicing (`&s[..60]`), a 60-char boundary that falls inside a multi-byte character will cause a runtime panic. All plan acceptance criteria pass (test data uses ASCII), but production data with non-ASCII characters crashes the tool. (**Quality concern: crash-inducing gap**)
- **TG2.2 — Tab/newline in field values corrupts TSV.** No acceptance criterion tests for tabs or newlines embedded in field values. Test data uses clean ASCII strings. In production, a description like `"Create config\tfile"` produces a malformed TSV row with an extra column. (**Quality concern: UX violation — output that prevents intended use**)
- **TG2.3 — Column selection mismatch.** The struct has 7 fields, output shows 5. No acceptance criterion verifies that `id` and `ingested_at` are excluded. If the implementation accidentally includes them, tests using simple string matching may still pass if they only assert on substrings. (**Quality concern: observability gap — wrong output silently accepted**)
- **TG2.4 — JSONL field with special characters.** No acceptance criterion tests JSONL output where field values contain quotes, backslashes, or Unicode. If manual string formatting is used instead of `serde_json::to_string()`, these break. (**Quality concern: crash-inducing gap**)
- **TG2.5 — `std::process::exit(1)` in unknown-table arm bypasses destructors.** The plan calls `std::process::exit(1)` directly for unknown table names. This skips Rust's drop/cleanup. In the current implementation this is probably harmless (no unflushed buffers at that point), but if the function is refactored to hold a database connection with pending writes, the exit could leave the database in an inconsistent state. (**Quality concern: crash-inducing gap — resource leak on error path**)

## Stage 3: Testing & Documentation

### Security Checks

- [ ] **S3.1 — Test databases use in-memory SQLite.** Verify no test writes to the filesystem. Tests using file-based databases in shared directories (e.g., the project root) could conflict with concurrent runs or leave stale files.
- [ ] **S3.2 — No secrets in README examples.** Verify the documentation examples use placeholder data, not real commit SHAs, file paths, or database contents.

### UX Checks

- [ ] **U3.1 — `adr-db view --help` includes instability warning.** The help text must include "proto-porcelain" and "no stability guarantees" so users see the warning before depending on the output.
- [ ] **U3.2 — README examples are copy-pasteable.** Verify the README command examples use consistent formatting (backtick blocks, correct flag syntax) and match the actual CLI interface.
- [ ] **U3.3 — README documents all five invocations.** The five forms from the ADR: `view`, `view task_summaries`, `view task_summaries --output jsonl`, `--limit`, `--no-header`.

### Test-Gap Findings

- **TG3.1 — No CLI integration test (end-to-end binary execution).** All tests in Task 3.1 are unit tests calling internal functions. No test builds the `adr-db` binary and runs it as a subprocess with `Command::new()`. This means the clap argument parsing → dispatch → output pipeline is never tested as a whole. A wiring bug in `main()` (e.g., wrong field mapping between clap struct and `run_view()` arguments) would pass all unit tests but fail in practice. (**Quality concern: observability gap — the end-to-end path is untested**)
- **TG3.2 — No test for `--output` validation.** If `output` is a `String`, the code path that rejects invalid format strings (e.g., `--output csv`) is untested. (**Quality concern: crash-inducing gap — untested error path**)
- **TG3.3 — No test for `--limit 0` or `--limit -1`.** Edge cases for the limit flag are not in the acceptance criteria. `--limit 0` should return 0 data rows (but possibly a header). `--limit -1` in SQLite means "no limit" — this is surprising behavior that should either be documented or rejected. (**Quality concern: UX violation — surprising behavior at boundary values**)
