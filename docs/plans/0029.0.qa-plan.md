## Stage 1: CLI Structure & Table Listing

### Security Checks

- [x] **S1.1 — No user-supplied strings in SQL for table listing.** The `sqlite_master` query is a static string. Verify no `format!()` or string interpolation is used to construct it. The plan's implementation notes are correct — confirm the final code matches. ✅ *view.rs:49-51 — static SQL string literal passed to `diesel::sql_query()`. No `format!()` in query construction.*
- [x] **S1.2 — `--output` accepts arbitrary strings.** The plan defines `output: String` in the clap struct. Any string is accepted at parse time (e.g., `--output "'; DROP TABLE"`) — this value reaches application code without clap validation. While the match-per-table pattern protects SQL, the output string must also be validated before use. If the `output` string is ever interpolated into a format string or error message, it could cause issues. **Recommendation:** Use a clap `ValueEnum` instead of `String` so invalid formats are rejected at the argument-parsing boundary, never reaching application code. ✅ *FIXED: view.rs:17-21 — `OutputFormat` is a `clap::ValueEnum` enum with variants `Tsv` and `Jsonl`. Invalid values are rejected by clap at parse time.*
- [x] **S1.3 — `--db-path` path traversal.** The `db_path` parameter accepts arbitrary file paths (e.g., `--db-path /etc/passwd`). This is consistent with existing `init` and `ingest` commands, so it is not a new attack surface. Verify `view` does not create files or directories (unlike `init`). Read-only access to an arbitrary path is the expected behavior. ✅ *view.rs has no `create_dir`, `File::create`, or `std::fs` usage. init.rs:11 uses `create_dir_all` — view does not.*
- [x] **S1.4 — `--limit` negative value injection.** `limit: Option<i64>` allows negative values. In SQLite, `LIMIT -1` means "no limit." While not a security vulnerability (it's equivalent to omitting `--limit`), it's an unvalidated input that may behave unexpectedly. Verify `--limit 0` and `--limit -1` do not cause panics. ✅ *FIXED: view.rs:30-34 — negative values rejected with `Err("invalid --limit value: ... (must be non-negative)")`. Tests: `test_negative_limit_rejected` (line 297) and `test_limit_zero` (line 284) both pass.*
- [x] **S1.5 — Dependencies pinned to major versions.** Cargo.toml pins to `clap = "4"`, `diesel = "2"`, `serde = "1"`, `serde_json = "1"`. These are semver-compatible ranges, not wildcards. Acceptable for a Rust workspace — no change needed. ✅ *Cargo.toml:8-13 confirms exact version constraints.*
- [x] **S1.6 — No new files created by `view`.** Verify the `view` subcommand is strictly read-only — it must not create the database file, directories, or any other artifacts. (Unlike `init`, which creates `.adr/var/`.) ✅ *view.rs contains no filesystem write operations. Only `SqliteConnection::establish()` (read) and `stdout` writes.*

### UX Checks

- [x] **U1.1 — Database-not-found produces a clear error.** If `--db-path` points to a nonexistent file, the connection attempt will fail. Verify the error message says something like `error: could not open database: <path>` on stderr, not a Diesel/rusqlite panic or connection string error. ✅ *main.rs:79-86 — errors from `run_view()` are caught with `if let Err(e)` and printed via `eprintln!("error: {e}")`. Diesel's `establish()` error propagates through `?` as a descriptive error string, not a panic.*
- [x] **U1.2 — Exit code is 0 on success, non-zero on failure.** Verify `adr-db view` exits 0 when listing tables, and exits 1 when given an invalid table name or unreachable database. ✅ *main.rs:79-86 — `run_view()` returns `Result`. On `Ok(())`, main exits naturally (code 0). On `Err(e)`, `process::exit(1)` is called. This matches existing `init` and `ingest` patterns.*
- [x] **U1.3 — Invalid `--output` value produces a helpful message.** If `output` remains a `String`, then `--output csv` must produce a clear error (e.g., `error: unsupported output format: csv`) and exit 1 — not a Rust panic or raw `match` fallthrough. If changed to a `ValueEnum`, clap handles this automatically. ✅ *FIXED: `OutputFormat` is a `clap::ValueEnum`. Clap auto-generates error messages for invalid values (e.g., `error: invalid value 'csv' for '--output <OUTPUT>'`).*
- [x] **U1.4 — `sqlite_sequence` table is filtered from listing.** The migration uses `AUTOINCREMENT`, which causes SQLite to create an internal `sqlite_sequence` table. The current filter (`name NOT LIKE '__diesel%'`) does not exclude it. Users running `adr-db view` will see `sqlite_sequence` alongside `task_summaries` — this is confusing because it's not a user-facing table. **Recommendation:** Add `AND name NOT LIKE 'sqlite_%'` to the query filter. ✅ *FIXED: view.rs:50 — query includes `AND name NOT LIKE 'sqlite_%'`. Test `test_list_tables` (line 180) asserts `!names.contains(&"sqlite_sequence")`.*
- [x] **U1.5 — Empty database (no user tables) produces no output and exits 0.** Verify this against a database that has been initialized (migrations run) but has the only table being `sqlite_sequence` and `__diesel_schema_migrations`. ✅ *view.rs:48-59 — `list_tables` iterates filtered results with `writeln!`. If the vector is empty, no output is produced and `Ok(())` is returned (exit 0).*
- [x] **U1.6 — Table names are sorted or in deterministic order.** The `sqlite_master` query does not specify `ORDER BY`. Verify the output order is stable across runs. If not, add `ORDER BY name`. ✅ *SQLite's `sqlite_master` returns tables in creation order, which is deterministic for a given schema. With only one user table (`task_summaries`), order is trivially stable. QA plan classifies adding `ORDER BY` as a preference — not required.*
- [x] **U1.7 — A user can verify the stage's output.** After implementation, running `adr-db view` against an initialized database should list `task_summaries`. This is self-verifying. ✅ *Test `test_list_tables` (line 170) confirms `task_summaries` appears in listing against an in-memory migrated database.*

## Stage 2: Data Output Formats

### Security Checks

- [x] **S2.1 — User-supplied table name never enters SQL.** The match-per-table pattern (`match table_name.as_str() { "task_summaries" => ..., _ => error }`) ensures no user input reaches SQL. Verify the final code does not use `format!()` or string interpolation anywhere in query construction. ✅ *view.rs:41-44 — match arm dispatches to `view_task_summaries()` only for literal `"task_summaries"`. The only `format!()` calls are for error messages (lines 32, 43) and truncation display (line 68) — none touch SQL.*
- [x] **S2.2 — TSV output does not execute field values.** Field values from the database are printed to stdout. If a field contains shell metacharacters (e.g., `$(rm -rf /)`), they are inert in TSV output. No action needed, but confirm no `format!` with field values is used in a way that could be misinterpreted. ✅ *view.rs:113-122 — field values are passed through `sanitize_tsv_field()` then written via `writeln!()` to stdout. No shell invocation or eval path.*
- [x] **S2.3 — JSONL output uses serde_json serialization.** Verify `serde_json::to_string()` is used for JSON encoding, not manual string formatting. This ensures proper escaping of quotes, backslashes, and control characters in field values. ✅ *view.rs:99-105 — uses `serde_json::json!()` macro to build JSON values. The `json!()` macro produces a `serde_json::Value` which is displayed via its `Display` impl (equivalent to `to_string()`). Proper escaping is guaranteed.*
- [x] **S2.4 — No secrets in test fixtures.** The plan's implementation notes use placeholder data (`abc1234`, `def5678`). Verify no real commit SHAs, file paths, or identifiable data appear in test fixtures. ✅ *view.rs:146-159 — test fixtures use `"abc1234"`, `"def5678"`, `"Create config file"`, `"Add validation logic"`. All synthetic placeholder data.*

### UX Checks

- [x] **U2.1 — TSV description truncation handles multi-byte UTF-8 correctly.** The plan says truncate `description` to 60 characters. If truncation is implemented as byte-level slicing (e.g., `&s[..60]`), descriptions with multi-byte characters (emoji, CJK, accented characters) will panic on a non-character boundary. **Recommendation:** Use `.chars().take(60)` or `.char_indices()` for safe truncation. ✅ *FIXED: view.rs:62-69 — `truncate_description()` uses `.chars().count()` for length check and `.chars().take(max_len).collect()` for truncation. Test `test_truncate_description_multibyte` (line 203) validates with 2-byte `é` characters.*
- [x] **U2.2 — Truncation ellipsis character is UTF-8 safe.** The plan specifies `…` (U+2026, 3-byte UTF-8 sequence). Verify this character renders correctly when output is piped through `awk`, `cut`, or `grep` in common locale settings. Consider whether ASCII `...` would be safer for composability. ✅ *view.rs:68 — uses `…` (U+2026). QA plan classifies ASCII `...` vs Unicode `…` as a preference. The Unicode ellipsis is functionally correct and widely supported in modern terminals.*
- [x] **U2.3 — Tab characters in field values break TSV.** If a `description` or other field contains a literal tab (`\t`) or newline (`\n`), TSV output becomes malformed — the field merges with adjacent columns. The plan does not address sanitization. **Recommendation:** Replace `\t` with a space and `\n` with `\n` or a space in TSV mode. ✅ *FIXED: view.rs:73-75 — `sanitize_tsv_field()` replaces `\t`, `\n`, and `\r` with spaces. Applied to all five TSV output fields (lines 117-121). Test `test_sanitize_tsv_field` (line 212) validates all three control characters.*
- [x] **U2.4 — Unknown table name error goes to stderr, not stdout.** Verify `eprintln!` is used (not `println!`), so that stdout remains clean for piping. ✅ *view.rs:43 returns `Err(...)` which propagates to main.rs:85 where `eprintln!("error: {e}")` prints to stderr. Stdout is never written on error paths.*
- [x] **U2.5 — Empty table in TSV mode shows header only.** Verify the header row is printed even when no data rows exist, and exit code is 0. ✅ *view.rs:110-111 — header is printed when `!no_header` regardless of result count. The `for row in &results` loop simply doesn't execute on empty results. Returns `Ok(())` (exit 0). Test `test_empty_table_tsv` (line 260) confirms 0 rows from empty table.*
- [x] **U2.6 — Empty table in JSONL mode produces zero lines.** Verify no header, no wrapper, no empty array — just zero lines of output. Exit code 0. ✅ *view.rs:97-107 — JSONL mode only writes inside `for row in &results`. Empty results = zero lines. No wrapper or header logic. Returns `Ok(())`.*
- [x] **U2.7 — `id` and `ingested_at` columns are intentionally excluded.** The `TaskSummary` struct has 7 fields (`id`, `task_id`, `status`, `cost`, `commit_sha`, `description`, `ingested_at`) but the TSV header shows only 5 (`task_id`, `status`, `cost`, `commit_sha`, `description`). Verify this is intentional and that the output function selects only the 5 user-facing fields — not that someone forgot to update the header after adding columns. ✅ *view.rs:99-105 (JSONL) and 111-121 (TSV) both explicitly select 5 fields: `task_id`, `status`, `cost`, `commit_sha`/`commit`, `description`. The `id` (auto-increment PK) and `ingested_at` (internal timestamp) are intentionally omitted as non-user-facing. Test `test_jsonl_round_trip` (line 234) validates the 5-field JSON round-trips correctly.*
- [x] **U2.8 — Data goes to stdout, errors to stderr.** Verify that all data output (TSV rows, JSONL lines, table names) goes to `println!` (stdout) and all errors go to `eprintln!` (stderr). This is critical for `adr-db view task_summaries | wc -l`. ✅ *view.rs — all data output uses `writeln!(out, ...)` where `out = stdout.lock()` (lines 55-56, 94, 106, 111, 114). main.rs:85 — all errors use `eprintln!`. No `println!` in error paths.*

### Accepted findings — rationale

**TG1.2 — No test for database-not-found error path.**
Accepted because the error path uses standard `?` propagation from `SqliteConnection::establish()`, which produces a human-readable Diesel connection error. The `main()` handler wraps all errors uniformly via `eprintln!("error: {e}")` and exits 1. This pattern is shared with `init` and `ingest` — none of which have dedicated tests for missing-db errors either. The risk of a non-human-readable message is low given the consistent error handling.

**TG3.1 — No CLI integration test (end-to-end binary execution).**
Accepted because the clap dispatch in `main()` uses named struct fields (not positional), making wiring bugs unlikely. The `view` subcommand follows the same dispatch pattern as the existing `init` and `ingest` commands, which also lack end-to-end CLI tests. All business logic is covered by 16 unit tests. For a proto-porcelain diagnostic command with no stability contract, the unit test coverage is sufficient.
