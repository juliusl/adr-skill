## Stage 1: Directory Relocation

### Security Checks

- [x] **S1.1** — No user-supplied strings are interpolated into the `git mv` command or Makefile path expressions. The Makefile uses `$(CURDIR)` (Make-internal), not user input, to construct the manifest path.
- [x] **S1.2** — No secrets, credentials, or `.env` files are committed. Confirm `.gitignore` still covers `src/crates/adr-db/.env` after the path update (Task 1.3).
- [x] **S1.3** — Not applicable. No deserialization occurs in this stage.
- [x] **S1.4** — Verify `Cargo.lock` is preserved intact after `git mv`. A missing or regenerated lockfile could alter pinned dependency versions.
- [x] **S1.5** — After `git mv`, confirm moved files retain their original permissions. `git mv` preserves mode bits in the index, but verify no executable bits are lost on shell scripts or gained on data files.
- [x] **S1.6** — Not applicable. No new external input surfaces are introduced in this stage.

### UX Checks

- [x] **U1.1** — If `cargo build --release --manifest-path src/crates/Cargo.toml` fails after the move, the error message must clearly indicate the manifest path. Verify `make build-tools` produces a readable error if `src/crates/Cargo.toml` is missing.
- [x] **U1.2** — `make build-tools` exits 0 on success and non-zero on failure after the Makefile path change (Task 1.2).
- [x] **U1.3** — If a developer runs the old `cargo build --manifest-path crates/Cargo.toml` after the move, they get a clear "file not found" error, not a cryptic failure. (Not a code change — verification only.)
- [x] **U1.4** — If `git mv` fails mid-operation (e.g., `src/crates/` already exists), the working tree is not left in a half-moved state. Verify rollback behavior.
- [x] **U1.5** — After `make build-tools` succeeds, the compiled binary exists at `src/crates/target/release/adr-format` and is executable.
- [x] **U1.6** — After Task 1.3, `.gitignore` correctly ignores `src/crates/target/` — run `git status` and confirm `target/` artifacts are not tracked.
- [x] **U1.7** — A developer unfamiliar with the move can verify Stage 1 by running `make build-tools && ls src/crates/target/release/adr-format`.

### Test-Gap Findings

- **TG1.1** — The plan's acceptance criteria for Task 1.1 checks that `cargo build` succeeds, but does not verify `cargo test` at this stage. An internal path issue (e.g., a `build.rs` or `include!()` using relative paths) could surface only during tests.
- **TG1.2** — Task 1.3 criteria check that `.gitignore` entries are correct, but do not verify the negative case — that `.gitignore` does not *also* still contain the old `crates/target/` pattern (which would be harmless but misleading). The criteria do say "no stale patterns," but there is no automated check for this.
- **TG1.3** — The plan does not verify that `Cargo.lock` is byte-identical after `git mv`. A corrupted lockfile would pass `cargo build` but silently resolve different dependency versions.

## Stage 2: Reference Updates

### Security Checks

- [x] **S2.1** — The `wi-full-agent-adr-format.sh` path change (Task 2.2) constructs `CRATES_BIN` from `$SCRIPT_DIR` (resolved via `readlink`/`cd && pwd`), not from user input. Confirm no user-controlled path components are introduced. Verify the path does not traverse outside the repo root via the new `../../../crates/` traversal.
- [x] **S2.2** — No secrets are introduced in AGENTS.md or documentation updates.
- [x] **S2.3** — Not applicable. No deserialization in this stage.
- [x] **S2.4** — Not applicable. No dependency changes in this stage.
- [x] **S2.5** — Not applicable. No new files created in this stage.
- [x] **S2.6** — Verify the script's fallback path resolution (`$SCRIPT_DIR/../../../crates/target/release/adr-format`) correctly resolves when invoked from the installed skill location (`~/.copilot/skills/author-adr/scripts/`) vs. the repo source. The installed copy does not have a sibling `crates/` directory — confirm the fallback fails gracefully (line 22 error message) rather than resolving to an unintended binary.

### UX Checks

- [x] **U2.1** — When `adr-format` is not found via either `command -v` or `CRATES_BIN`, the script emits "ERROR: adr-format binary not found. Run 'make build-tools' first." on stderr. Verify this message is still accurate after the path change.
- [x] **U2.2** — `wi-full-agent-adr-format.sh` exits 0 when `adr-format` succeeds and non-zero when it fails. Verify `exec` behavior is preserved (the script replaces itself with the binary).
- [x] **U2.3** — If `wi-full-agent-adr-format.sh` is called with no arguments or invalid arguments, the binary's own error handling produces a helpful message. Verify the path change does not mask the binary's stderr output.
- [x] **U2.4** — Not applicable. No resources created that need cleanup.
- [x] **U2.5** — After Task 2.3, `make test` passes all 31 tests and `make check-refs` reports zero broken references. This is the primary readback mechanism.
- [x] **U2.6** — After AGENTS.md updates (Task 2.1), a developer reading P-13 can find `src/crates/` and follow the instructions. All code examples in P-13 use the updated paths.
- [x] **U2.7** — A developer who did not perform the migration can verify Stage 2 by running `make test && make check-refs`.

### Test-Gap Findings

- **TG2.1** — Task 2.2 acceptance criteria verify the path string on line 15, but do not test end-to-end that the script actually locates and runs the binary. An integration test (invoke `wi-full-agent-adr-format.sh` with a simple subcommand and check exit code) would catch path resolution bugs that string matching misses.
- **TG2.2** — Task 2.1 criteria say "no stale `crates/` path references remain in AGENTS.md (excluding historical mentions of ADR-0028)." The plan does not specify how to detect stale references — a `grep` for `crates/` that does not immediately follow `src/` would be the mechanical check. Without this, a missed reference is discovered only by manual reading.
- **TG2.3** — The plan's Task 2.1 includes an "opportunistic scan" of `docs/` files, but provides no acceptance criteria for it. README.md contains three actionable `crates/` references (directory tree, prose description, `cd crates/adr-db` command) that would mislead contributors if not updated. The plan does not enumerate these or test for them.
- **TG2.4** — `make check-refs` only scans SKILL.md and `references/*.md` within each skill directory. It does not scan AGENTS.md, README.md, or the Makefile. A broken path in AGENTS.md would not be caught by this check.
