### Test-Gap Findings

**S1-GAP-1 — Routing tree is inside a fenced code block; check-refs skips code blocks**
`check-refs` explicitly tracks `in_code` state and skips link extraction inside fenced blocks. If the routing tree entries reference `fast-path.md` or `S-3` as a textual target inside the code block, a typo will not be caught. Only markdown links outside code fences are validated.

**S1-GAP-2 — No enforcement that SKILL.md stays under 500 lines after Task 1.1 and 1.2**

Cumulative additions across stages could push past 500 with no automated gate until Stage 4. A developer who applies stages out of order would have no warning.

**S1-GAP-3 — fast_path_sources TOML example is in a fenced code block; value is never validated**
The config example shows `fast_path_sources = ["retro", "bug-bash", "amendment"]`. This is a documentation-only example. There is no schema file, no parser, and no test that validates the key name or the allowed values. A typo in the example (e.g., `"bug_bash"` vs `"bug-bash"`) silently teaches users the wrong value — the agent will read whatever is in the real preferences file, which may not match.

### Test-Gap Findings

**S2-GAP-1 — fast-path.md content can drift from the SKILL.md S-3 section with no consistency check**
Task 2.1 creates fast-path.md; Task 2.2 adds the S-3 section to SKILL.md. No acceptance criterion requires verifying that the two agree. If fast-path.md says "Step 3: skip Options and Evaluation Checkpoint" but the SKILL.md S-3 summary section omits this, all per-task ACs pass while the agent has contradictory instructions depending on which source it reads first.

**S2-GAP-2 — Classification test is a judgment call with no tie-breaking rule**
Step 1 of S-3 classifies findings via: "Would seeing this in the ADR list be useful to someone orienting in this codebase?" This is inherently subjective. No tie-breaking rule is specified for borderline cases. An agent may route a finding as an ADR when it should be a non-decision task, or vice versa. The acceptance criteria do not include a worked example or decision tree that disambiguates edge cases.

**S2-GAP-3 — Zero-ADR edge case: Step 5 behavior when all findings are non-decision**
Step 5 invokes `/implement-adr` with "Ready ADRs + non-decision task list." ADR-0067's S-3 procedure doesn't define what happens when classification produces zero ADRs — only non-decision tasks. Does Step 5 still invoke `/implement-adr`? Is a bare task list a valid input to that skill? If no ADR is Ready, the invocation may fail or produce unexpected behavior. The acceptance criterion for Task 2.1 does not require this case to be documented in fast-path.md.

**S2-GAP-4 — Conclusion routing in fast-path.md will be stale when Stage 3 adds C-4**
Task 2.1 creates fast-path.md with "Conclusion routing → C-1 → C-2 → C-3." Stage 3 (Task 3.1) adds C-4 after C-3. If fast-path.md is not updated in Stage 3 to reflect the new sequence (C-1 → C-2 → C-3 → C-4), the procedure reference will describe a stale conclusion flow. No task in Stage 3 requires updating fast-path.md. This is a cross-stage dependency the plan does not track.

**S2-GAP-5 — R-1 criterion 1 N/A note has no verification path**
Task 2.1 requires "A-3 onward with reviewer note about R-1 criterion 1 N/A." R-1 is defined in the author-adr skill. The AC does not verify that the note accurately describes what R-1 criterion 1 requires, or that the note is placed in the right location within the fast-path.md procedure. A reviewer unfamiliar with R-1 cannot validate this without cross-referencing a bundled asset that is not listed in the plan.

## Stage 3: C-4 Retrospective Step

Tasks 3.1 and 3.2 modify SKILL.md. The C-4 section introduces runtime behavior (writing files and modifying config). Security and UX checks are evaluated against this runtime surface.

### Security Checks

- [ ] **S3-SEC-1** — Verify no user-supplied retro answers are interpolated into the retro record file path; the slug must be derived from a safe, controlled source (e.g., timestamp or sanitized session ID), not from free-text user input
- [ ] **S3-SEC-2** — No secrets or credentials in SKILL.md edits
- [ ] **S3-SEC-3** — Verify the preferences write path does not deserialize TOML from untrusted user input without validation; if the agent reads and rewrites the file, it must not blindly interpolate raw answer text into TOML values
- [ ] **S3-SEC-4** — N/A: No new package dependencies
- [ ] **S3-SEC-5** — Verify `.adr/var/` and `retro-<slug>.md` are created with restrictive permissions (not world-readable if the retro record could contain sensitive design rationale)
- [ ] **S3-SEC-6** — The 4 retrospective questions constitute a new user input surface; verify that all four question answers are treated as user text (not executable), and that the retro record write path cannot be influenced by answer content to escape `.adr/var/`

### UX Checks

- [ ] **S3-UX-1** — If `.adr/preferences.toml` does not exist and the agent attempts to write existing keys, the failure must produce a human-readable message explaining the file is missing — not a silent no-op or raw exception
- [ ] **S3-UX-2** — N/A: No shell commands introduced
- [ ] **S3-UX-3** — Verify C-4 documents what happens when `enabled` is absent from `[solve.retro]` — is the default true or false? An agent that reads a preferences file without the key must not crash or behave inconsistently
- [ ] **S3-UX-4** — If `.adr/var/` directory creation fails (permissions, read-only FS), the error must be surfaced — not swallowed
- [ ] **S3-UX-5** — Retro record is verifiable: after C-4 runs, the agent can confirm the record exists at `.adr/var/retro-<slug>.md` and display its path
- [ ] **S3-UX-6** — Preferences write is inspectable: after the autonomous write path, the agent must log which key(s) were written and what values were set; guided path must show the proposed change before applying
- [ ] **S3-UX-7** — A developer who did not write this can read `.adr/var/retro-<slug>.md` and verify its content matches the retro responses; config change verifiable via `cat .adr/preferences.toml`

### Test-Gap Findings

**S3-GAP-1 — .adr/preferences.toml write: no merge semantics defined; full overwrite would corrupt the file**
The plan says the agent writes "existing keys." TOML has no standard patch/merge operation. If the agent reads the file, modifies specific keys in memory, and rewrites the whole file, that is a merge — but the procedure does not specify how it handles TOML sections the agent didn't intend to touch, comment preservation, or ordering. A naive overwrite with only the `[solve]` block would destroy all other settings. The AC ("classification rule (no new keys)") prevents key invention but does not define the write mechanism.

**S3-GAP-2 — preferences.toml nonexistent: "write existing keys" rule is undefined when file is absent**
The rule says "agent writes only to existing keys; never invents new keys." If `.adr/preferences.toml` does not exist, there are no existing keys. The agent could: (a) fail silently, (b) create the file (which creates keys that didn't exist, violating the rule), or (c) surface an error. None of these cases is handled in the acceptance criteria. The plan adds a note about creating `.adr/var/` if absent but says nothing about the preferences file itself being absent.

**S3-GAP-3 — Retro slug collision: second retro for the same session/slug silently overwrites the first**
ADR-0068 specifies `.adr/var/retro-<slug>.md` but does not define how the slug is generated or how collisions are handled. If a user runs C-4 twice (e.g., after a resumption), the second write overwrites the first without warning. The AC does not require collision detection or append behavior.

**S3-GAP-4 — skip_when_no_findings = false behavior is ambiguous when there are no findings**
The key name implies the retro runs even when no findings exist. But the 4 retrospective questions reference "findings" from the solve session. If there are no findings, questions referencing them produce nonsensical answers. The acceptance criteria do not require C-4 to document graceful handling of the zero-findings case when `skip_when_no_findings = false`.

**S3-GAP-5 — No cross-check that C-4 opt-out behavior in SKILL.md matches the config key description in Task 3.2**
Task 3.1 adds opt-out documentation in the C-4 section. Task 3.2 adds `enabled` and `skip_when_no_findings` to the config table. The acceptance criteria for 3.1 and 3.2 are checked independently. There is no AC that requires verifying the opt-out prose in the C-4 section is consistent with the key description in the config table — they could describe different defaults or trigger conditions.

### Test-Gap Findings

**S4-GAP-1 — check-refs does not validate content inside fenced code blocks**
The routing tree entries added in Stage 1 live inside a code block. check-refs skips code fences. If an S-3 routing entry contains a misspelled reference to `fast-path.md` (as a textual label rather than a markdown link), check-refs will not catch it. The validation gate passes while the routing table is internally inconsistent.

**S4-GAP-2 — make test validates adr-tools and madr-tools scripts; it does not validate SKILL.md content**
The 31-test suite covers shell script behavior (diff-based). All tests pass regardless of whether SKILL.md has the correct S-3 section, C-4 section, or config block. A stage that modifies only SKILL.md has no automated correctness test — only a line-count check and a link-resolution check.

**S4-GAP-3 — 500-line check is manual; there is no Makefile target or CI gate enforcing it**
The plan requires `wc -l` to confirm SKILL.md is under 500 lines, but this is a manual step recorded by convention. Nothing in `make validate` or `make check-refs` enforces the limit. A developer who skips the `wc -l` step has no automated failure — the plan's only enforcement is the developer's own compliance.

**S4-GAP-4 — check-refs validates that fast-path.md exists but not that it contains the required sections**
`make check-refs` confirms the file is reachable (non-broken link). It does not verify fast-path.md has all 5 required steps, the Conclusion routing note, the R-1 N/A reviewer note, or the [Source: <origin>] format example. A file consisting of a single sentence passes check-refs.
