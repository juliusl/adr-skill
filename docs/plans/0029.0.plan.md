## Stage 1: CLI Structure & Table Listing

### Task 1.1: Add `view` subcommand to clap CLI    [small]

**Description:**
Add a `View` variant to the `Commands` enum in `main.rs` with the following clap arguments:
- `table_name: Option<String>` — positional, optional. When omitted, list tables.
- `--output <format>` — optional, accepts `tsv` (default) or `jsonl`.
- `--limit <N>` — optional, limits output rows.
- `--no-header` — optional flag, suppresses header row in TSV mode.
- `--db-path <path>` — reuses the existing `--db-path` convention with default `.adr/var/adr.db`.

Create a new `view.rs` module file with a placeholder `run_view()` function. Wire the subcommand dispatch in `main()`.

**Implementation Notes:**
```rust
// In Commands enum:
/// Inspect database contents (proto-porcelain, no stability guarantees)
View {
    /// Table name to inspect. Omit to list all tables.
    table_name: Option<String>,
    /// Output format: tsv (default) or jsonl
    #[arg(long, default_value = "tsv")]
    output: String,
    /// Limit number of rows
    #[arg(long)]
    limit: Option<i64>,
    /// Suppress header row in TSV mode
    #[arg(long)]
    no_header: bool,
    /// Path to the SQLite database file
    #[arg(long, default_value = ".adr/var/adr.db")]
    db_path: PathBuf,
},
```

**Test & Acceptance Criteria:**
- [x] `Commands` enum has a `View` variant with all five arguments
- [x] `view.rs` module exists with a `run_view()` function signature
- [x] `main()` dispatches `Commands::View` to `view::run_view()`
- [x] `cargo build` succeeds with no errors

**Dependencies:** None

**ADR Reference:** ADR-0029, Decision §Subcommand design

### Task 1.2: Implement table listing (`adr-db view` with no args)    [small]

**Description:**
When `table_name` is `None`, query `sqlite_master` for user tables and print each table name on its own line to stdout. Filter out Diesel internal tables (`__diesel%`).

Use `diesel::sql_query()` with a static string — no user input enters the SQL. Define a small `TableName` struct with `#[derive(QueryableByName)]` for the result.

**Implementation Notes:**
```rust
#[derive(QueryableByName)]
struct TableName {
    #[diesel(sql_type = diesel::sql_types::Text)]
    name: String,
}

let tables: Vec<TableName> = diesel::sql_query(
    "SELECT name FROM sqlite_master WHERE type='table' AND name NOT LIKE '__diesel%'"
).load(&mut conn)?;

for t in &tables {
    println!("{}", t.name);
}
```

**Test & Acceptance Criteria:**
- [x] `adr-db view` (no args) lists table names, one per line
- [x] Diesel internal tables are excluded from listing
- [x] Empty database (no user tables) produces no output and exits 0
- [x] Unit test: table listing against in-memory database returns `task_summaries`

**Dependencies:** Task 1.1

**ADR Reference:** ADR-0029, Decision §Table listing

## Stage 2: Data Output Formats

### Task 2.1: Implement TSV output for `task_summaries`    [medium]

**Description:**
When `table_name` is `"task_summaries"`, query all rows using a typed Diesel query (`TaskSummary::as_select()`). Output in tab-separated format with a header row. The `description` field is truncated to 60 characters in TSV mode (append `…` if truncated). Use DB column names in the header (`task_id`, `status`, `cost`, `commit_sha`, `description`).

Implement the `match` dispatch pattern: match the table name string against known tables. Unknown table names produce an error message to stderr and exit 1.

**Implementation Notes:**
```rust
match table_name.as_str() {
    "task_summaries" => view_task_summaries(&mut conn, output, limit, no_header),
    _ => {
        eprintln!("error: unknown table: {table_name}");
        std::process::exit(1);
    }
}
```

TSV output:
```
task_id	status	cost	commit_sha	description
1.1	done	small	abc1234	Create config file
```

**Test & Acceptance Criteria:**
- [x] TSV output includes header row with DB column names
- [x] Rows are tab-separated
- [x] `description` field truncated to 60 chars with `…` when exceeded
- [x] Unknown table names produce error on stderr and exit 1
- [x] Empty table produces header only (no rows) and exits 0
- [x] Unit test: TSV output format for sample data matches expected format

**Dependencies:** Task 1.2

**ADR Reference:** ADR-0029, Decision §Output formats (TSV)

### Task 2.2: Implement JSONL output for `task_summaries`    [small]

**Description:**
When `--output jsonl` is specified, emit each row as a JSON object on its own line. Use ingest-compatible field names (`commit` not `commit_sha`) so output can round-trip through `adr-db ingest`. Full fidelity — no truncation of `description`.

**Implementation Notes:**
```rust
// Use serde for serialization with renamed fields
#[derive(Serialize)]
struct TaskSummaryJsonl {
    task_id: String,
    status: String,
    cost: String,
    commit: String,       // Renamed from commit_sha for ingest compatibility
    description: String,
}
```

**Test & Acceptance Criteria:**
- [x] JSONL output uses ingest-compatible field names (`commit`, not `commit_sha`)
- [x] Each row is a single JSON line
- [x] `description` is NOT truncated in JSONL mode
- [x] Unit test: JSONL output round-trips — output can be deserialized as `JsonlTaskRecord`

**Dependencies:** Task 2.1

**ADR Reference:** ADR-0029, Decision §Output formats (JSONL), §Field-name alignment

### Task 2.3: Implement `--limit` and `--no-header` flags    [small]

**Description:**
- `--limit N`: Apply `.limit(N)` to the Diesel query to cap output rows. When not specified, return all rows.
- `--no-header`: Suppress the header row in TSV mode. Has no effect in JSONL mode.

**Test & Acceptance Criteria:**
- [ ] `--limit 2` returns at most 2 data rows
- [ ] `--no-header` suppresses the header row in TSV mode
- [ ] `--no-header` has no effect on JSONL output
- [ ] Unit test: limit and no-header behavior verified

**Dependencies:** Task 2.1

**ADR Reference:** ADR-0029, Decision §Subcommand design
