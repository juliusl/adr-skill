## Stage 1: Defer Verb (ADR-0033)

### Test-Gap Findings

1. **Step 5b Q&A building instructions still say "address or reject."** Task 1.1
   (already completed) correctly added the three choices to Step 3 and the
   Defer Mechanics subsection. However, the Step 5b prose at lines 104–113 of
   revise.md still describes Q&A entry transformation as converting "address or
   reject" responses — it does not mention Defer. A dev executing Task 1.2 could
   add the `**Deferred**` prefix format as specified but forget to update the
   surrounding prose that frames the transformation. The acceptance criteria
   check for the **Deferred** prefix presence and a complete example, but do NOT
   check that the Step 5b *prose instructions* say "address, defer, or reject"
   instead of "address or reject."

   **Classification:** Quality concern — if the prose instructions mention only
   two verbs while the format shows three prefixes, an agent reading the prose
   would not know when to produce a Deferred Q&A entry. The format example
   alone is insufficient; the building-instructions prose must also enumerate
   all three verbs.

   **Recommendation:** Add acceptance criterion to Task 1.2: "Step 5b building
   instructions prose references all three verbs (address, defer, reject), not
   just address/reject."

2. **SKILL.md revise summary and revise.md Step 5 summary line may diverge.**
   Task 1.3 updates the SKILL.md summary to mention three choices in Step 3,
   but SKILL.md's revise summary also paraphrases Step 5: "Produce revision
   summary — document what was addressed vs. rejected" (line 141). If Task 1.3
   only updates Step 3's description and leaves Step 5's paraphrase at two
   verbs, the SKILL.md summary will be internally inconsistent (Step 3 says
   three choices, Step 5 says two outcomes). No acceptance criterion covers the
   Step 5 paraphrase in SKILL.md.

   **Classification:** Quality concern — SKILL.md is the first document an
   agent reads. Internal inconsistency between its Step 3 and Step 5
   descriptions will confuse the agent about whether Defer produces a summary
   entry or is silently dropped.

   **Recommendation:** Add acceptance criterion to Task 1.3: "SKILL.md
   revision summary step description includes deferred alongside addressed
   and rejected."

3. **No cross-file consistency check after Stage 1.** Tasks 1.1, 1.2, and 1.3
   modify revise.md and SKILL.md independently. No acceptance criterion
   verifies that the revise.md three-way vocabulary (Address/Reject/Defer) and
   the SKILL.md summary use *exactly matching* terminology. An implementor
   could write "defer" in SKILL.md but "Deferred" in revise.md Step 5,
   creating a subtle mismatch.

   **Classification:** Preference — terminology normalization is a stylistic
   concern. Both forms are understandable. Defer to iteration.

## Stage 2: Dispatch Hooks (ADR-0031)

### Test-Gap Findings

1. **No cycle limit documented for multi-round editor→review→revise loop.**
   Task 2.3 acceptance criterion says: "when editor is delegated and re-review
   triggers, the editor→review→revise loop runs per the dispatch config until
   verdict is Accept or cycle limit is reached." But the criterion does not
   specify what the cycle limit IS, nor does it require the implementation to
   document one. If an implementor writes "runs until Accept or cycle limit is
   reached" in revise.md without specifying a default limit (e.g., 3 rounds),
   the instruction is incomplete — a delegated editor agent would have no
   concrete stopping condition and could loop indefinitely.

   **Classification:** Quality concern — an unbounded loop in a delegated
   agent workflow is a crash-inducing gap. The agent would consume context
   window or API budget without converging.

   **Recommendation:** Add acceptance criterion to Task 2.3: "A default cycle
   limit is specified (e.g., 3 rounds) with behavior when limit is reached
   (e.g., surface to user, accept as-is, or fail with message)."

2. **Steps 2 and 4 in revise.md still say "the user" without disambiguation.**
   Task 2.3 criterion says "Step 3 acknowledges that 'the user' may be the
   editor agent." But revise.md Steps 2 and 4 also reference "the user":
   Step 2 says "present" findings to "the user" (line 44–49); Step 4 says
   "the user addressed" (line 77). If Step 3 says "the user (or configured
   editor agent)" but Steps 2 and 4 still say "the user" without
   qualification, an editor agent reading the instructions sequentially would
   encounter inconsistent framing — Step 3 acknowledges its role, but Step 2
   ("present to the user") implies a human audience that an agent cannot
   "present" to in the same way.

   **Classification:** Quality concern — inconsistent "the user" references
   across steps could cause a delegated editor agent to misinterpret its role
   in Steps 2 and 4 while correctly understanding Step 3.

   **Recommendation:** Add acceptance criterion: "When editor dispatch section
   is added, clarify that 'the user' throughout the revision process refers
   to the configured editor (human or agent) or add a preamble note."

3. **Graceful fallback behavior is documented but not testable.** Task 2.1
   criterion says "Graceful fallback behavior documented (unresolved agent →
   default + warning)." The documentation will say "fall back to default and
   warn," but since this is a documentation-only plan, there is no code that
   implements the fallback. The risk is that the instruction says "warn" but
   provides no guidance on WHERE to warn (stderr? conversation output? a log?)
   or what the warning message should look like. An implementing agent
   would have to invent the warning format.

   **Classification:** Preference — the fallback behavior will be implemented
   by agent runtime code outside this repo. The documentation's role is to
   describe intent, not implement it. Defer.

4. **review.md intro update could create ambiguity when no dispatch config
   exists.** Task 2.2 changes the review.md intro from "Use this reference as
   a prompt for a general-purpose agent" to reference the configured reviewer
   agent. The criterion says "Default behavior unchanged when no dispatch
   config exists" — but the *text* will change from a concrete instruction
   ("use a general-purpose agent") to a conditional one ("use the configured
   agent"). An agent reading review.md without dispatch config would encounter
   instructions that reference a "configured reviewer agent" that doesn't
   exist in its context, causing confusion.

   **Classification:** Quality concern — the intro text must work correctly
   as a standalone prompt for BOTH the default case (no dispatch config) and
   the configured case. The acceptance criteria don't verify that the default
   case reads naturally.

   **Recommendation:** Add acceptance criterion to Task 2.2: "review.md intro
   is intelligible when read without dispatch context (e.g., 'the configured
   reviewer agent (default: general-purpose)')."

## Stage 3: Draft Worksheet (ADR-0032)

### Test-Gap Findings

1. **Worksheet dimension names are not verified against ADR-0032 across all
   consuming files.** Task 3.1 requires the SKILL.md worksheet structure to
   match ADR-0032's five dimensions (Framing, Tolerance, Uncertainty, Options,
   Candidates). Tasks 3.2, 3.3, and 3.4 then reference specific field names
   from the worksheet in create.md, solve.md, and revise.md respectively. No
   acceptance criterion performs a cross-file consistency check — if Task 3.1
   uses "Framing" but Task 3.2 references "Problem Framing," the downstream
   consumer will look for a field that doesn't exist.

   **Classification:** Quality concern — inconsistent field names across files
   would cause agent failures when consuming the worksheet. The worksheet is a
   structured data exchange format between draft mode and downstream workflows;
   field names must match exactly.

   **Recommendation:** Add a cross-file acceptance criterion to Task 4.1 (or a
   new criterion in Task 3.2/3.3/3.4): "All worksheet field names referenced
   in create.md, solve.md, and revise.md match the canonical names defined in
   SKILL.md's Drafting section exactly."

2. **Solve workflow Improvisation tolerance conflicts with existing option
   discovery guidance.** Task 3.3 says solve.md Step 2 should "Apply
   Improvisation tolerance to control divergence from the framing." But the
   existing solve.md Step 2 (Option Discovery) explicitly encourages broad
   exploration: "Aim for ≥3 genuine options" (line 80) and includes Discovery
   Techniques like "Constraint relaxation — what if we removed constraint X?"
   (line 87). A low Improvisation tolerance that constrains divergence would
   contradict the existing "constraint relaxation" technique and potentially
   suppress legitimate options. No acceptance criterion addresses this tension
   or specifies how to reconcile the worksheet calibration with existing
   discovery guidance.

   **Classification:** Quality concern — contradictory instructions in the
   same file (solve.md) will cause unpredictable agent behavior. An agent
   might suppress valid options because the tolerance says "stay close" while
   the techniques say "explore freely."

   **Recommendation:** Add acceptance criterion to Task 3.3: "Instructions
   clarify that Improvisation tolerance calibrates the FRAMING SCOPE of option
   discovery (how far from the stated problem to search), not the number or
   quality of options within that scope. Existing discovery techniques remain
   valid."

3. **Comments section ordering when all three features are present.** Tasks 1.2,
   2.3, and 3.4 all affect the Comments section behavior. After all three
   stages, Comments can contain: (a) a Draft Worksheet, (b) Q&A addendum
   entries with three prefixes, and (c) editor intent-grounding reads. Task
   3.4 criterion says "Q&A append logic verified to work with preceding Draft
   Worksheet" — but this is tested in isolation from the multi-round editor
   loop (Task 2.3). No acceptance criterion verifies the combined scenario:
   Draft Worksheet present + editor dispatched + multi-round revision + Q&A
   entries accumulating after the worksheet.

   **Classification:** Quality concern — the Comments section is now a
   structured multi-zone area (worksheet zone → Q&A zone) with ordering
   constraints. The interaction between worksheet detection, Q&A marker
   detection (`<!-- Generated by the revise task -->`), and multi-round
   appending could fail if any of the three features assumes it "owns" the
   Comments section exclusively.

   **Recommendation:** Add a cross-stage integration criterion (to Task 4.1
   or Task 3.4): "The Comments section layout is explicitly documented:
   Draft Worksheet (if present) → revise task marker → Q&A entries (rounds).
   Append logic uses the marker position, not the start of Comments, as the
   Q&A insertion point."

4. **No acceptance criterion verifies worksheet is truly optional.** Task 3.1
   says "Draft mode is optional — users can skip straight to create/solve."
   Tasks 3.2 and 3.3 say "no worksheet = current behavior unchanged." But
   these criteria are tested by the *absence* of worksheet instructions, not
   by the *presence* of conditional language in the instruction text. An
   implementor could write create.md instructions that say "Read the Draft
   Worksheet from Comments" unconditionally, with no "if present" guard,
   and still satisfy the criterion "Framing field seeds Context section" —
   because the criterion doesn't say "only when present." A following agent
   that encounters no worksheet would fail trying to read it.

   **Classification:** Quality concern — missing conditional guards in
   instruction text would cause agent errors in the common case where no
   worksheet exists (since draft mode is a new, optional feature).

   **Recommendation:** Strengthen Task 3.2 and 3.3 criteria: "Worksheet
   reading is gated by an explicit conditional (e.g., 'If a Draft Worksheet
   exists in Comments, read it… If no worksheet exists, proceed as before')."

5. **SKILL.md line count after Stage 3.** The plan requires SKILL.md to stay
   under 500 lines (Task 4.1). SKILL.md is currently 223 lines. Task 2.1 adds
   the dispatch config section (~20–30 lines). Task 3.1 adds the entire
   "Drafting an ADR" section (~30–50 lines). Task 1.3 makes a minor edit.
   Total estimate: ~280–300 lines. This is well under 500. No concern.

   **Classification:** Not a finding — included for completeness.

## Cross-Stage Findings

### Finding CS-1: Comments section becomes a multi-zone structured document

After all three stages, the ADR's `## Comments` section can contain:

1. **Draft Worksheet** (Stage 3) — five dimensions of intent capture
2. **Revise task marker** (`<!-- Generated by the revise task -->`) — signals
   the start of Q&A entries
3. **Q&A addendum entries** (Stage 1) — with three prefixes (Addressed,
   Deferred, Rejected)
4. **Multi-round markers** (`<!-- Round N -->`) — for iterated revisions
5. **Editor intent-grounding reads** (Stage 3, Task 3.4) — the editor agent
   reads the worksheet for triage calibration

No single task's acceptance criteria verify the full layout. Each task tests its
own zone in isolation. The risk is that an implementor structures the zones
correctly individually but produces an incoherent combined layout — e.g., the
Q&A marker appearing before the worksheet, or the editor reading Q&A entries as
worksheet content.

**Classification:** Quality concern — the Comments section is now a structured
interchange format with multiple producers and consumers. A layout specification
is needed.

### Finding CS-2: No end-to-end narrative test across all three features

The plan tests each feature independently (Defer in isolation, dispatch in
isolation, worksheet in isolation). No acceptance criterion describes a
scenario where all three interact: a user drafts an ADR with a worksheet,
dispatches review to a custom agent, the editor agent triages findings using
Defer with intent grounding from the worksheet, and Q&A entries are appended
after the worksheet. This combined path is the *intended* use case per the
ADRs (0031, 0032, 0033 are co-designed), but no criterion validates it.

**Classification:** Preference — integration testing of documentation
instructions is inherently manual (run the workflow and observe agent
behavior). The individual task criteria are sufficient for a documentation-only
plan. Defer to a follow-up iteration where the combined workflow is exercised.
