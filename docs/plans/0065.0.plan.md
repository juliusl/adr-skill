### Task 1.1: Add viability pre-screen guidance to Step 4a dispatch    [small]

**Description:** Add two-pass guidance to the Step 4a dispatch instructions in `src/skills/author-adr/references/create.md`. Before the current "Dispatch in parallel" step, insert guidance instructing each reviewer to perform a quick viability screen (~30 seconds per option) before deep analysis. Non-viable options should be flagged and excluded from detailed evaluation.

**Implementation Notes:**
Add a new sub-step between the current steps 1 (Build dispatch context) and 2 (Dispatch in parallel) in Step 4a, or modify step 2 to include pre-screen instructions in the dispatch prompt. The guidance should instruct agents to:
- First pass: assess each option's viability in ~30 seconds — flag obviously non-viable options
- Second pass: deep analysis only on viable options
- Report non-viable options with a brief reason, but do not spend time on element-by-element analysis

**Test & Acceptance Criteria:**
- [x] Step 4a in create.md contains viability pre-screen guidance
- [x] The guidance specifies a two-pass approach (quick screen, then deep analysis)
- [x] Non-viable options are explicitly excluded from deep analysis
- [x] The pre-screen does not block or modify the parallel dispatch mechanism

**Description:** Modify the "Incorporate findings" logic in Step 4a of `src/skills/author-adr/references/create.md` to add a "Revise — Major" verdict tier between "Revise" and "Redesign". This tier handles heavy-but-addressable revisions to the Options section without escalating to prototype-adr.

**Implementation Notes:**
Current verdict mapping in Step 4a step 3:
- Redesign → Pause for validation

New mapping:

- Revise → checkpoint considerations (minor adjustments)
- Accept → proceed

**Test & Acceptance Criteria:**

- [x] "Revise — Major" routes to inline rework of the Options section, not prototype-adr
- [x] "Redesign" retains its existing "Pause for validation" routing
- [x] "Revise" retains its existing "checkpoint considerations" routing

**Dependencies:** None

### Task 1.3: Add "Revise — Major" verdict mapping to Step 4b    [small]

**Description:** Update the TPM verdict mapping in Step 4b of `src/skills/author-adr/references/create.md` to include a "Revise — Major" equivalent. The TPM uses its own verdict vocabulary (not-ready / ready with findings / ready), so map the existing tiers to include a middle ground.

**Implementation Notes:**
Current TPM verdict mapping in Step 4b step 3:
- not-ready → Pause for validation
- ready with findings → checkpoint considerations
- ready → proceed

Add a mapping for findings that indicate heavy revision is needed but the approach is sound:
- not-ready (fundamental gaps) → Pause for validation
- not-ready (addressable gaps, no anti-patterns) → Revise — Major: incorporate findings as revision requirements
- ready with findings → checkpoint considerations
- ready → proceed

The distinction: "not-ready" with missing START criteria or detected anti-patterns routes to Pause. "not-ready" with addressable structural gaps (e.g., missing decision drivers, incomplete option analysis) routes to Revise — Major.

**Test & Acceptance Criteria:**
- [x] Step 4b in create.md distinguishes between fundamental and addressable "not-ready" verdicts
- [x] Addressable "not-ready" maps to Revise — Major (inline rework), not Pause for validation
- [x] Fundamental "not-ready" retains its Pause for validation routing

**Dependencies:** Task 1.2

**ADR Reference:** ADR-0065, Decision — Finding 2 (verdict tier gap)

## Stage 2: Polish.md Refinements and Consistency

### Task 2.1: Add complexity threshold to R-6a    [small]

**Description:** Modify R-6a (Accept-with-Suggestions Polish Pass) in `src/skills/author-adr/references/polish.md` to add a dispatch threshold. When an editor agent is configured, only dispatch for ≥3 suggestions or any Medium+ severity item. For ≤2 Low-severity suggestions, apply them inline without dispatch.

**Implementation Notes:**
Modify the R-6a condition section. Current condition: "Accept verdict with minor suggestions." Add a dispatch threshold decision:

```
When editor is configured:
  If suggestions count ≥ 3 OR any suggestion is Medium+ severity:
    → dispatch editor agent (current behavior)
  Else (≤2 Low-severity suggestions):
    → apply suggestions inline, log what was changed
    → do not dispatch editor agent
```

**Test & Acceptance Criteria:**
- [ ] R-6a in polish.md contains a dispatch threshold
- [ ] Threshold: ≥3 suggestions OR any Medium+ severity → dispatch
- [ ] ≤2 Low-severity suggestions → apply inline without dispatch
- [ ] Inline application still logs what was changed
- [ ] The threshold does not affect behavior when no editor is configured

**Dependencies:** None

**ADR Reference:** ADR-0065, Decision — Finding 3 (editor polish threshold)
