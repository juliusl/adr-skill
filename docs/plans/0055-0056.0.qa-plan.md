- **TG1.1** — **Lifecycle diagram graph structure not validated by acceptance criteria.** The plan specifies the diagram text but the acceptance criteria only check "Ready appears." The executor must verify the arrow sequence is exactly: `Prototype ──► Proposed ──► Ready ──► Planned ──► Accepted ──► Deprecated` with the Superseded fork from Accepted. **Verification:** visually confirm the diagram line matches the expected topology.

### Test-Gap Findings

- **TG2.1** — **Content preservation is the highest-risk item and the acceptance criteria rely on manual counting.** The plan lists 13 content-preservation criteria for Task 2.1 (7 fallacies, 11 anti-patterns, 6 implementability criteria, 7-point checklist, 3 verdict types, 7 guardrails, Q&A addendum, review cycle markers, editor dispatch). Each is a "preserved" checkbox — but none specify how to mechanically verify preservation. A fallacy could be paraphrased, truncated, or misnumbered and still get checked off. **Verification:** For the highest-risk items (fallacies, anti-patterns, guardrails), extract the specific names/titles from review.md and revise.md before deletion, then grep for each name in polish.md to confirm presence.

  Concrete verification commands:
  - Fallacies (7): `grep -c 'Wishful Thinking\|Sunk Cost\|Bandwagon\|Analysis Paralysis\|Anchoring\|Cargo Cult\|Golden Hammer' src/skills/author-adr/references/polish.md` should return 7+
  - Anti-patterns (11): grep for each anti-pattern name — Fairy Tale, Sales Pitch, etc.
  - Guardrails (7): grep for each guardrail identifier or key phrase from revise.md

- **TG2.2** — **Step ID collision between polish.md and SKILL.md.** Polish.md uses R-prefix (R-1 through R-6a) and V-prefix (V-1 through V-6). Author-adr SKILL.md already uses A-prefix steps. The plan assumes no collision, but if polish.md accidentally uses an A-prefix identifier, an agent following the combined context could confuse SKILL.md procedure steps with polish.md procedure steps. **Verification:** `grep -n '^### A-' src/skills/author-adr/references/polish.md` should return 0 matches.

- **TG2.3** — **Section marker format must be exact for agent dispatch.** The plan says "section markers: `## Review Phase` and `## Revision Phase`" and SKILL.md A-3/A-4 will direct agents to these headings. If the heading in polish.md uses any variation (e.g., `## Review phase`, `## Review`, `## Review Phase:`, or a markdown heading level other than `##`), the dispatch instruction won't match. **Verification:** `grep -xn '## Review Phase' src/skills/author-adr/references/polish.md` and `grep -xn '## Revision Phase' src/skills/author-adr/references/polish.md` each return exactly 1 match.

- **TG2.4** — **Q&A addendum semantic boundary preservation.** Revise.md line 199 uses a `---` horizontal rule as an immutable/mutable boundary in the Q&A addendum. If polish.md contains multiple `---` markers (e.g., section separators), an agent could mistake a structural separator for the semantic boundary. **Verification:** confirm the `---` boundary semantics are clearly documented in context within polish.md, and that the semantic boundary is distinguishable from any structural separators.

- **TG2.5** — **Accept verdict → Ready transition is the sole ADR-0055 integration point in this stage.** The plan specifies review.md line 183 ("caps at Proposed") becomes "caps at Ready" in polish.md. If this single line is missed during the merge, the entire ADR-0055 integration for the review workflow fails — author-adr would still cap at Proposed in practice. The acceptance criteria check for this, but it's buried among 15 other criteria. **Verification:** `grep -n 'caps at Ready\|Proposed → Ready\|Ready status' src/skills/author-adr/references/polish.md` should return matches confirming the status cap and transition.

- **TG2.6** — **Intent grounding (Draft Worksheet) from revise.md must survive consolidation.** Revise.md lines 247-259 describe how the editor agent reads the Draft Worksheet for triage. This is critical for editor autonomy — without it, the editor agent has no grounding for its accept/reject decisions. The acceptance criterion says "Editor dispatch mechanics preserved (including intent grounding)" but doesn't specify a verification method. **Verification:** `grep -n 'Draft Worksheet\|intent grounding\|tolerance calibration' src/skills/author-adr/references/polish.md` should return matches.

- **TG2.7** — **SKILL.md dispatch table update could leave stale section references.** Task 2.2 updates the dispatch table so both hooks receive polish.md with section focus. If the dispatch table currently specifies `review.md` by filename and the update changes the filename but omits the section focus (or vice versa), agents will receive polish.md but won't know which section to read. **Verification:** the dispatch table must specify both the file reference and the section focus for each hook (Review Phase for review hook, Revision Phase for editor hook).

- **TG2.8** — **Deletion order in Task 2.3 matters for git state.** If Task 2.3 runs `git rm` before Task 2.2 completes (violating the dependency), SKILL.md references to review.md/revise.md would point to deleted files. - **TG2.9** — **procedure-template.md compliance not mechanically verified.** Task 2.1 says polish.md "follows procedure-template.md structure" and lists expected sections, but no acceptance criterion verifies structural compliance (policies at top, summary table before step details, flow diagram present). **Verification:** confirm the first `##` heading after the title is the policies section; confirm a summary table with all step IDs exists before the first step detail section.

- **TG2.10** — **Deep References in SKILL.md: residual entries.** Task 2.2 removes review.md and revise.md from Deep References and adds polish.md. If there are other references to review.md or revise.md elsewhere in SKILL.md (e.g., inline mentions in step descriptions, comments, or the Assets table), they would survive as broken references. ### Test-Gap Findings

- **TG3.1** — **`make check-refs` does not validate anchor fragments.** The `scripts/check-refs` script strips anchor fragments (`filepath="${target%%#*}"`) and only checks file existence. If SKILL.md A-3 references `references/polish.md#review-phase` but the heading in polish.md is `## Review Phase` (anchor: `#review-phase`), check-refs will validate the file exists but not the anchor. Since the plan directs agents to specific sections by heading name rather than anchor link, this is low risk — but if future edits add anchor links, they won't be validated. **Verification:** confirm SKILL.md references use prose section names (e.g., "the Review Phase section") rather than markdown anchor links.

- **TG3.2** — **`make test` does not cover reference content.** The 31 tests (22 adr-tools + 9 madr-tools) test shell scripts for ADR file manipulation (create, list, status, rename). They do not test reference file content, procedure structure, or step identifier consistency. A Stage 2 error that corrupts polish.md content would pass `make test` cleanly. This is by design (the test suite covers tooling, not documentation), but it means Stage 3 validation provides no safety net for the highest-risk task (Task 2.1). **Verification:** the executor must perform manual content verification for Task 2.1 in addition to running `make test`.

- **TG3.3** — **check-refs scope: cross-skill references.** The plan modifies files in author-adr, implement-adr, and solve-adr. `make check-refs` scans each skill directory independently. If a reference in solve-adr points to a file in author-adr (e.g., `../author-adr/references/review.md`), check-refs may or may not catch the broken reference depending on how it resolves relative paths across skill boundaries. **Verification:** `grep -rn '\.\./author-adr/references/review\|../author-adr/references/revise' src/skills/solve-adr/ src/skills/implement-adr/ --include='*.md'` should return 0 matches.

- **TG3.4** — **Line count target range is aspirational.** Task 3.2 specifies polish.md should be "between 350-500 lines." The acceptance criterion treats this as a hard range. If polish.md is 345 or 505 lines, the criterion fails — but the task says "target: 350-450" in Task 2.1 while acceptance says "350-500" in Task 3.2. This inconsistency means the executor could produce a 475-line file that passes Task 3.2 but exceeds the Task 2.1 target. Low risk but worth noting. **Verification:** confirm the line count expectation is consistent between Task 2.1 and Task 3.2; use 500 as the hard limit and 350-450 as the target.
