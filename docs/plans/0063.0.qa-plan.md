## Stage 1: Skill Instruction Updates

### Security Checks
- [ ] **S1.1** — Verify `tech_writer` TOML value is consumed only as an agent name string. No shell expansion, interpolation, or command construction from user-supplied config values.
- [ ] **S1.2** — Confirm dispatch contract does not forward ambient secrets or credentials to the tech-writer agent beyond what the `task` tool normally provides.
- [ ] **S1.3** — Verify `tech_writer` input validation: empty string `""` and absent key both trigger the "inline" fallback. Whitespace-only values (e.g., `" "`) must not bypass the empty check.
- [ ] **S1.4** — Confirm the tech-writer agent runs with the same permission scope as the inline agent — no privilege escalation through dispatch.
- [ ] **S1.5** — N/A — no new binary dependencies introduced.
- [ ] **S1.6** — N/A — TOML parsing uses existing infrastructure; no new deserialization paths.

### UX Checks
- [ ] **U1.1** — Fallback warning: SKILL.md A-2 step 4 specifies "warn the user" when the configured agent cannot be resolved. Verify the warning is visible and actionable (not buried in agent logs).
- [ ] **U1.2** — N/A — documentation-only changes; no exit codes affected.
- [ ] **U1.3** — Verify that invalid `tech_writer` values (empty, whitespace, unresolvable agent name) all fall back to inline writing without crashing or blocking the workflow.
- [ ] **U1.4** — Verify that if tech-writer dispatch fails mid-execution (partial content written), the ADR file is left in a state the inline agent can recover from (continue writing or retry).
- [ ] **U1.5** — Verify the user can determine which `tech_writer` value is in effect. A-0 reads and stores the config; confirm the stored value is inspectable (e.g., logged at A-0 or visible in A-2).
- [ ] **U1.6** — Verify there is an observable difference between "tech-writer dispatched" and "inline agent wrote content" — the user should be able to tell which path executed.
- [ ] **U1.7** — Verify the TOML example, hook table, and contract description in SKILL.md are mutually consistent: same default value (`""`), same semantics, same fallback behavior described in all three locations.

### Test-Gap Findings
- **TG1.1** — The plan's acceptance criteria for Task 1.3 do not verify that SKILL.md step numbering remains consistent after inserting dispatch logic into step 4. If the insertion shifts subsequent step numbers, cross-references (e.g., "A-2 Steps 4–6" in ADR-0063) break.
- **TG1.2** — Task 1.3 acceptance criteria check that fallback behavior is "documented" but do not verify the fallback is consistent with the ADR-0063 specification (three fallback scenarios: absent, empty, unresolvable). All three must appear.
- **TG1.3** — No acceptance criterion verifies that P-1 (Mandatory Dispatch Compliance) applies to `tech_writer` without modification. The existing P-1 text says "When `[author.dispatch]` keys are configured" — this already covers `tech_writer`, but the plan does not ask the implementer to verify this.
