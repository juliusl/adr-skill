### Test-Gap Findings

- **TG1.1** — `template.rs` line 1 uses `use crate::schema::*` to import TOML types (`Adr`, `Meta`, etc.). After migration, `template.rs` lives at `adr-db-lib/src/format/template.rs`. In this new location, `crate::schema` resolves to the Diesel-generated `schema.rs` (table definitions for `task_summaries`), NOT the format's `schema.rs`. This will cause a compilation error. The import must change to `use super::schema::*` or `use crate::format::schema::*`. The plan does not flag this import adjustment.
- **TG1.2** — `template.rs` test block (line 89) uses `use crate::{parse_adr, serialize_adr}`. After migration, these functions are in `crate::format::schema` but re-exported at the crate root via `pub use format::*`. This path resolves correctly via re-export — but only if `lib.rs` has `pub use format::*`, AND the `format/mod.rs` has `pub use schema::*`. If either re-export is missing, tests break with a confusing "not found" error.
- **TG1.3** — Task 1.3 verifies format types are accessible at the crate root but does not verify that the existing `adr_db_lib::schema::task_summaries` Diesel module is unaffected. A `pub use format::*` glob could theoretically shadow the `schema` module. In Rust, explicit `pub mod schema` wins over glob-imported `schema`, so this is safe — but no test confirms it.
- **TG1.4** — The plan does not verify that `models.rs` (which uses `use crate::schema::task_summaries`) still compiles after adding `pub use format::*`. This import should be unaffected by Rust's shadowing rules, but no acceptance criterion confirms it.

### Test-Gap Findings

- **TG2.1** — Task 2.1 acceptance criteria specify incorrect CLI signatures for 5 of 7 subcommands. The criteria use placeholder signatures (`<file>`, `<old> <new>`, `<dir>`) that do not match the actual clap argument definitions. Correct signatures from source code:

  | Criterion (plan) | Actual signature (source) |
  |---|---|
  | `adr-db format list <dir>` | `adr-db format list` (no args — uses `resolve_dir()`) |
  | `adr-db format rename <old> <new>` | `adr-db format rename <remote> <id> <new_title>` |
  | `adr-db format status <file> [--set <status>]` | `adr-db format status [remote] [id] [new_status]` |
  | `adr-db format lifecycle <file>` | `adr-db format lifecycle <remote> <id> [--auto] [--sync]` |
  | `adr-db format export <file>` | `adr-db format export <remote> <id>` |

  An implementer following the plan literally could implement wrong signatures. The correct signatures should match the existing `adr-format` CLI exactly.

- **TG2.2** — The plan does not explicitly mention that `adr-format/src/main.rs` contains qualified type references to the `adr_format` crate: `adr_format::AdrOption` (line 217), `adr_format::CheckpointItem` (lines 228-230), `adr_format::Decision` (line 232), `adr_format::Consequence` (line 237). After migration to `adr-db/src/main.rs`, ALL of these must change to `adr_db_lib::AdrOption`, etc. The plan says "move the handler logic" but does not flag the import rewrite.
- **TG2.3** — The export tests use `args(["export", "gh", "42"])` (export_test.rs lines 113, 169, 199, 229). After migration, these must become `args(["format", "export", "gh", "42"])` — the `"format"` subcommand must be prepended. The plan's Task 2.3 description mentions this, but the change is across 4 test functions and easy to miss one.
- **TG2.4** — `cmd_list()` and `cmd_status(None, None)` use `resolve_dir()`, which reads `.adr/adr-dir` or `.adr-dir` from the current working directory. The export tests set `current_dir(dir.path())` on the Command to control this. After migration, the binary name in `env!("CARGO_BIN_EXE_adr-db")` resolves to a different binary path. Verify the test setup still works with the new binary.
- **TG2.5** — The shell script error message (line 22) currently says `"ERROR: adr-format binary not found. Run 'make build-tools' first."`. Task 2.2 says to update this to reference `adr-db`, but the acceptance criteria only check that the error message "references `adr-db` not `adr-format`" — they don't verify the message is actionable (i.e., `make build-tools` actually produces the `adr-db` binary at the expected path).
- **TG2.6** — No acceptance criterion tests what happens when a user has a stale pre-migration `adr-db` binary in PATH (one without the `format` subcommand). The shell script would `exec adr-db format "$@"`, and the old binary would fail with "unrecognized subcommand 'format'". This is transient but could confuse users during the transition.
