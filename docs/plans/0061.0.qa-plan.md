## Stage 1: Library Migration

### Security Checks

- [ ] **S1.1** — No user-supplied strings in SQL or shell commands. The format module (`schema.rs`, `template.rs`) does not interact with SQLite or execute shell commands. The `today()` function in `main.rs` shells out to `date +%Y-%m-%d`, but this is not user-controlled and stays in the CLI (Stage 2), not the library. No injection risk in Stage 1.
- [ ] **S1.2** — No secrets or credentials introduced. The moved files contain only type definitions, serialization logic, and validation. Verify no test fixtures contain real credentials.
- [ ] **S1.3** — Deserialization of TOML input uses `toml::from_str` with `#[serde(deny_unknown_fields)]` on the `Adr` struct (schema.rs line 19). This strictness must be preserved after migration — unknown fields are rejected at parse time. Verify `deny_unknown_fields` attribute is not lost during the move.
- [ ] **S1.4** — Task 1.2 adds `toml = "0.8"` to adr-db-lib. Verify this matches the exact version specifier in the original `adr-format/Cargo.toml` (`toml = "0.8"`). After migration, verify `Cargo.lock` preserves the same resolved toml version — a version bump could change parsing behavior.
- [ ] **S1.5** — No new files are created with permissive permissions. The format module files are source code committed to git.
- [ ] **S1.6** — No new external input surfaces. The format module only exposes library functions; the input surface (CLI) is in Stage 2.

### UX Checks

- [ ] **U1.1** — If `cargo build -p adr-db-lib` fails after adding the format module, the compiler error must clearly identify the issue. The most likely failure mode is the `use crate::schema::*` import in `template.rs` (see TG1.1). Verify the error points at the correct file and line.
- [ ] **U1.2** — `cargo build -p adr-db-lib` exits 0 on success, non-zero on failure. Standard Cargo behavior — no custom exit codes to verify.
- [ ] **U1.3** — Not directly applicable. Stage 1 introduces no user-facing input validation. The library functions (`parse_adr`, `validate`) produce structured error types (`ValidationError`), not panics. Verify these error paths are preserved.
- [ ] **U1.4** — If the migration is partially applied (e.g., files moved but `lib.rs` not updated), `cargo build` fails cleanly. No partial state can corrupt the database or filesystem.
- [ ] **U1.5** — After Stage 1, the format types are accessible at `adr_db_lib::format::Adr` and (via re-export) at `adr_db_lib::Adr`. Verify both paths resolve correctly.
- [ ] **U1.6** — After Stage 1, the Diesel schema module is still accessible at `adr_db_lib::schema::task_summaries`. Verify the `pub use format::*` glob does not shadow or break the existing Diesel module.
- [ ] **U1.7** — A developer can verify Stage 1 by running `cargo build -p adr-db-lib && cargo test -p adr-db-lib`.

### Test-Gap Findings

- **TG1.1** — `template.rs` line 1 uses `use crate::schema::*` to import TOML types (`Adr`, `Meta`, etc.). After migration, `template.rs` lives at `adr-db-lib/src/format/template.rs`. In this new location, `crate::schema` resolves to the Diesel-generated `schema.rs` (table definitions for `task_summaries`), NOT the format's `schema.rs`. This will cause a compilation error. The import must change to `use super::schema::*` or `use crate::format::schema::*`. The plan does not flag this import adjustment.
- **TG1.2** — `template.rs` test block (line 89) uses `use crate::{parse_adr, serialize_adr}`. After migration, these functions are in `crate::format::schema` but re-exported at the crate root via `pub use format::*`. This path resolves correctly via re-export — but only if `lib.rs` has `pub use format::*`, AND the `format/mod.rs` has `pub use schema::*`. If either re-export is missing, tests break with a confusing "not found" error.
- **TG1.3** — Task 1.3 verifies format types are accessible at the crate root but does not verify that the existing `adr_db_lib::schema::task_summaries` Diesel module is unaffected. A `pub use format::*` glob could theoretically shadow the `schema` module. In Rust, explicit `pub mod schema` wins over glob-imported `schema`, so this is safe — but no test confirms it.
- **TG1.4** — The plan does not verify that `models.rs` (which uses `use crate::schema::task_summaries`) still compiles after adding `pub use format::*`. This import should be unaffected by Rust's shadowing rules, but no acceptance criterion confirms it.

## Stage 2: CLI Migration

### Security Checks

- [ ] **S2.1** — The migrated CLI commands include `validate_dir()` which rejects paths containing `..` (path traversal prevention, main.rs line 100-105), `validate_remote()` which whitelists remotes (line 78-87), and `validate_id()` which enforces alphanumeric IDs (line 90-98). All three validators must be preserved in the migrated code — they guard against injection via file paths and work-item identifiers.
- [ ] **S2.2** — No secrets in the migrated CLI code. The `cmd_lifecycle` function reads `.adr/var/work-items.jsonl` (a local cache file). This path is hardcoded, not user-supplied. No credential exposure.
- [ ] **S2.3** — `cmd_lifecycle` deserializes JSONL from a local cache file using `serde_json::from_str::<serde_json::Value>`. The input comes from a local file previously written by `adr-db fetch`. This is trusted input, but malformed JSONL is handled gracefully (the `if let Ok(val)` pattern on line 386 skips bad lines silently). Verify this behavior is preserved.
- [ ] **S2.4** — Task 2.3 adds `tempfile = "3"` to dev-dependencies. This is a test-only dependency. Verify the version matches the original `adr-format/Cargo.toml` specification exactly (`tempfile = "3"`).
- [ ] **S2.5** — The `cmd_new` and `cmd_init` functions create files using `fs::write`. The `cmd_rename` and `cmd_status` functions use atomic writes (write `.tmp` file, then `fs::rename`). Default permissions are inherited from the umask. No overly permissive modes. Verify the atomic write pattern is preserved in the migrated code.
- [ ] **S2.6** — The shell script change (Task 2.2) replaces `command -v adr-format` with `command -v adr-db`. The `CRATES_BIN` path changes from `adr-format` to `adr-db` binary. Both paths are derived from `$SCRIPT_DIR` (resolved via symlink traversal), not from user input. The relative path `../../../crates/target/release/adr-db` is fixed. No new input surface.

### UX Checks

- [ ] **U2.1** — Every `cmd_*` function in the migrated CLI returns `Result<(), String>`, with errors printed as `"ERROR: {}"` on stderr and `exit(1)`. The existing adr-db pattern uses `"error: {e}"` (lowercase, no "ERROR:" prefix). Decide whether to unify the error format or preserve the original. Either way, all error paths must produce human-readable stderr output.
- [ ] **U2.2** — Existing adr-db commands (`init`, `ingest`, `view`, `fetch`) must continue to exit 0 on success. New `format` subcommands must also exit 0 on success and non-zero on failure. Verify `clap` derives produce non-zero exit on parse errors (standard clap behavior).
- [ ] **U2.3** — Invalid input to format subcommands produces helpful messages. Examples: `adr-db format new badremote 42 title dir` → "unknown remote 'badremote'. Allowed: gh, ado, gitea, local". Verify validation functions are called before any filesystem operations.
- [ ] **U2.4** — Atomic write pattern in `cmd_rename` and `cmd_status`: if `fs::rename` fails after writing the `.tmp` file, the `.tmp` file is not cleaned up. This is a pre-existing issue, not introduced by the migration, but worth noting. After migration, behavior is preserved.
- [ ] **U2.5** — After running `adr-db format new`, the created `.toml` file can be read back with `adr-db format list` or `adr-db format export`. After `adr-db format status <remote> <id> <status>`, the updated status is visible via `adr-db format status <remote> <id>`. Verify end-to-end round-trip.
- [ ] **U2.6** — After Task 2.2, `wi-full-agent-adr-format.sh --help` displays the format subcommand help. A user running the shell script with the same arguments as before gets equivalent behavior.
- [ ] **U2.7** — A developer can verify Stage 2 by running: `cargo build -p adr-db && adr-db format --help` (shows 7 subcommands) and `cargo test -p adr-db --test export_test` (all 4 tests pass).

### Test-Gap Findings

- **TG2.1** — Task 2.1 acceptance criteria specify incorrect CLI signatures for 5 of 7 subcommands. The criteria use placeholder signatures (`<file>`, `<old> <new>`, `<dir>`) that do not match the actual clap argument definitions. Correct signatures from source code:

  | Criterion (plan) | Actual signature (source) |
  |---|---|
  | `adr-db format list <dir>` | `adr-db format list` (no args — uses `resolve_dir()`) |
  | `adr-db format rename <old> <new>` | `adr-db format rename <remote> <id> <new_title>` |
  | `adr-db format status <file> [--set <status>]` | `adr-db format status [remote] [id] [new_status]` |
  | `adr-db format lifecycle <file>` | `adr-db format lifecycle <remote> <id> [--auto] [--sync]` |
  | `adr-db format export <file>` | `adr-db format export <remote> <id>` |

  An implementer following the plan literally could implement wrong signatures. The correct signatures should match the existing `adr-format` CLI exactly.

- **TG2.2** — The plan does not explicitly mention that `adr-format/src/main.rs` contains qualified type references to the `adr_format` crate: `adr_format::AdrOption` (line 217), `adr_format::CheckpointItem` (lines 228-230), `adr_format::Decision` (line 232), `adr_format::Consequence` (line 237). After migration to `adr-db/src/main.rs`, ALL of these must change to `adr_db_lib::AdrOption`, etc. The plan says "move the handler logic" but does not flag the import rewrite.
- **TG2.3** — The export tests use `args(["export", "gh", "42"])` (export_test.rs lines 113, 169, 199, 229). After migration, these must become `args(["format", "export", "gh", "42"])` — the `"format"` subcommand must be prepended. The plan's Task 2.3 description mentions this, but the change is across 4 test functions and easy to miss one.
- **TG2.4** — `cmd_list()` and `cmd_status(None, None)` use `resolve_dir()`, which reads `.adr/adr-dir` or `.adr-dir` from the current working directory. The export tests set `current_dir(dir.path())` on the Command to control this. After migration, the binary name in `env!("CARGO_BIN_EXE_adr-db")` resolves to a different binary path. Verify the test setup still works with the new binary.
- **TG2.5** — The shell script error message (line 22) currently says `"ERROR: adr-format binary not found. Run 'make build-tools' first."`. Task 2.2 says to update this to reference `adr-db`, but the acceptance criteria only check that the error message "references `adr-db` not `adr-format`" — they don't verify the message is actionable (i.e., `make build-tools` actually produces the `adr-db` binary at the expected path).
- **TG2.6** — No acceptance criterion tests what happens when a user has a stale pre-migration `adr-db` binary in PATH (one without the `format` subcommand). The shell script would `exec adr-db format "$@"`, and the old binary would fail with "unrecognized subcommand 'format'". This is transient but could confuse users during the transition.

## Stage 3: Cleanup

### Security Checks

- [ ] **S3.1** — Deleting `src/crates/adr-format/` removes no security-relevant code. The validation logic has already been migrated to `adr-db-lib` (Stage 1) and `adr-db` CLI (Stage 2). Verify no validation functions were left behind in the deleted crate.
- [ ] **S3.2** — No secrets in the files being deleted or in the Cargo.toml description updates.
- [ ] **S3.3** — Not applicable. No deserialization changes in this stage.
- [ ] **S3.4** — After removing `adr-format` from the workspace members list, `cargo build --workspace` regenerates `Cargo.lock`. Verify the lockfile does not introduce new or upgraded dependencies. The `toml` crate remains (now depended on by `adr-db-lib` instead of `adr-format`). The `tempfile` dev-dependency moves from `adr-format` to `adr-db`.
- [ ] **S3.5** — No new artifacts created. The `adr-format` binary is no longer produced by `make build-tools`. Verify no stale `adr-format` binary remains in `target/release/` that could be accidentally executed.
- [ ] **S3.6** — Not applicable. No new input surfaces.

### UX Checks

- [ ] **U3.1** — If `cargo build --workspace` fails after removing the crate, the error must clearly identify dangling references (e.g., a workspace member that still lists `adr-format` as a dependency). Verify no other crate in the workspace depends on `adr-format`.
- [ ] **U3.2** — `make build-tools` exits 0 and produces the `adr-db` binary. Verify it no longer produces an `adr-format` binary.
- [ ] **U3.3** — Not directly applicable. No user input processed in this stage.
- [ ] **U3.4** — After deleting `adr-format/`, verify no orphaned build artifacts remain in `target/` that reference the deleted crate. A clean build (`cargo clean && cargo build --workspace`) should succeed.
- [ ] **U3.5** — After Task 3.2, Cargo.toml descriptions accurately reflect the expanded scope. `adr-db-lib`'s description says "ADR library" (not just "database library"). `adr-db`'s description says "format operations" (not just "JSONL ingestion").
- [ ] **U3.6** — After Task 3.3, AGENTS.md P-13 accurately describes the two-crate workspace. A new contributor reading P-13 can follow the instructions without encountering stale references.
- [ ] **U3.7** — A developer can verify Stage 3 by running `make build-tools && make test && make check-refs`.

### Test-Gap Findings

- **TG3.1** — Task 3.3 covers AGENTS.md but does NOT cover `src/skills/author-adr/SKILL.md`, which contains 7 references to `adr-format` as a binary name (lines 273, 281, 299, 301, 302, 303, 304). After migration, these command examples are stale: `adr-format export gh 42` should become `adr-db format export gh 42`, etc. `make check-refs` only validates file link resolution — it does not check command names in code examples. A user or agent following SKILL.md instructions would invoke a nonexistent binary.
- **TG3.2** — `make check-refs` scans only SKILL.md and `references/*.md` within each skill directory. It does not scan AGENTS.md, README.md, or the Makefile. A stale reference in AGENTS.md would pass `check-refs` but mislead contributors.
- **TG3.3** — After removing `adr-format` from the workspace, a stale `target/release/adr-format` binary may persist from prior builds. The shell script (after Task 2.2) no longer resolves to it, but a user who runs `adr-format` directly from PATH would execute an orphaned binary. No cleanup mechanism exists for this.
- **TG3.4** — The plan does not verify that historical ADRs and plans (e.g., `docs/adr/0052-*.md`, `docs/plans/0051-0053.0.plan.md`) are NOT modified. These are historical records that correctly reference `adr-format` as it existed at the time. An over-eager search-and-replace could corrupt them.

### Test-Gap Findings

- **TG4.1** — The plan says ADR-0061 transitions from `Planned` to `Accepted`, noting it was moved from `Ready` to `Planned` before execution began (Step I-5). If the implementer forgets the intermediate `Ready → Planned` transition, the ADR would go directly from `Ready` to `Accepted`, skipping `Planned`. This is a process gap, not a code bug, but it means the ADR status history would not reflect that implementation was in progress.
- **TG4.2** — The plan does not include updating ADR-0052 (`Implement wi-full-agent-adr format tooling in Rust`) to note that the standalone `adr-format` crate it created has been absorbed. ADR-0052 remains accurate as a historical record, but a "Superseded by ADR-0061" note would improve traceability.
