# QA Plan Template

The template for QA plan documents generated by the QA planner agent
(per ADR-0030). Use alongside the implementation plan template.

## Template

```markdown
# QA Plan: [Title]

**Source Plan:** docs/plans/<range>.<revision>.plan.md
**Generated:** YYYY-MM-DD
**Location:** `docs/plans/<range>.<revision>.qa-plan.md`

---

### Security Checks

- [ ] No user-supplied strings interpolated into SQL, shell commands, or file paths
- [ ] No secrets, credentials, or API keys in committed files
- [ ] No deserialization of untrusted input without validation
- [ ] Dependencies pinned to specific versions (no wildcards)
- [ ] File permissions on created artifacts are not overly permissive
- [ ] Any new external input surface has validation at the boundary

### UX Checks

- [ ] Every error path produces a human-readable message on stderr
- [ ] Every user-facing command exits with code 0 on success, non-zero on failure
- [ ] Invalid input is rejected with a helpful message, not a crash
- [ ] Resources (file handles, database connections) are cleaned up on error paths
- [ ] If the stage writes data, there is a way to read it back or verify it
- [ ] If the stage creates state, there is a way to inspect the new state
- [ ] A user who did not write the code can verify the stage's output
```

## Section Guide

| Section | Purpose |
|---------|---------|
| **Header** | Links to source plan, generation date |
| **Test-Gap Findings** | Per-stage blind spots in dev acceptance criteria |
| **Security Checks** | 6-item procedural checklist from qa-planning.md |
| **UX Checks** | 7-item procedural checklist (items 5-7 = observability check) |
| **Recommendations** | Findings classified by eligibility — quality concerns are schedulable, preferences are deferred |
| **Accepted findings** | Rationale for quality concerns accepted without remediation — why the risk is acceptable and what mitigates it |
